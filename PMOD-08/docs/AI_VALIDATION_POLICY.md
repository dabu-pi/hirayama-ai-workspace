# AI_VALIDATION_POLICY — AI検証ループの方針

**プロジェクトID:** PMOD-08  
**最終更新:** 2026-04-10  
**位置づけ:** このプロジェクトで AI を使って知識・記事の品質を継続維持するルール

---

## 目的

物療の知識は更新される。
書いた時点では正確でも、数年後には古くなる可能性がある。

このプロジェクトでは、**AIを定期的に使って知識の誤り・陳腐化を検証し、
「現場で使えるが、言い過ぎていない」品質を維持する**。

---

## AI検証を行う4つのタイミング

| タイミング | 頻度 | 対象 | 使用プロンプト |
|---|---|---|---|
| note 公開前レビュー | 記事作成後・毎回 | 公開予定の全記事 | `prompts/ai_review_prompt.md` |
| 月次ドキュメントレビュー | 月1回 | 主要docs配下のファイル | `prompts/ai_review_prompt.md` |
| エビデンス更新確認 | 月1〜2回 | `data/evidence/` 内の文献表 | `prompts/evidence_update_prompt.md` |
| 症例分析レビュー | 症例蓄積後 | `data/cases/` のプロトコル | `prompts/case_analysis_prompt.md` |

## AI検証の観点（チェックリスト）

### 1. 事実の正確性

- [ ] 周波数・パルス幅・モードの説明は物理的に正確か
- [ ] 生体反応の説明は現在の科学的理解と整合しているか
- [ ] 禁忌・注意事項に漏れや誤りはないか
- [ ] 機器名・仕様・メーカー情報に古さはないか

### 2. エビデンスの適切さ

- [ ] 「確立」「有望」「暫定」「不明」の区分が適切か（`docs/EVIDENCE_POLICY.md` 参照）
- [ ] 新しいSR・メタ解析・ガイドラインが出ていないか
- [ ] 引用している研究が撤回・修正されていないか
- [ ] 言い切りすぎていないか・根拠のない主張が含まれていないか

### 3. 臨床的整合性

- [ ] 禁忌を無視した記述になっていないか
- [ ] 特定機器を過剰に推薦・批判していないか
- [ ] 「やってみればわかる」的な根拠なき記述がないか
- [ ] 患者説明文例が誤解を生む表現になっていないか

### 4. 有害情報のチェック

- [ ] 過剰な出力・不適切な使用を促すような記述がないか
- [ ] 「このプロトコルで必ず治る」的な保証表現がないか
- [ ] 医師との連携・受診勧奨が必要な場面を省略していないか

## note 公開前レビューの手順

1. `note/drafts/` の記事草稿を開く
2. `prompts/ai_review_prompt.md` を使って Claude に検証させる
3. AIの指摘を確認し、以下を判断する
   - 修正必要 → 草稿を修正する
   - 問題なし → `note/free/` または `note/paid/` に移動する
   - 判断保留 → 文献確認後に再レビュー
4. レビュー結果を記事ファイル冒頭にコメントとして残す（任意）
5. 公開後も変更履歴を `logs/` に記録する

---

## 月次ドキュメントレビューの手順

1. 月1回、`docs/` 配下の主要ファイルを対象にレビューを実施する
2. `prompts/ai_review_prompt.md` に対象ファイルの内容をペーストして検証する
3. 指摘を確認し、修正・保留・棄却を判断する
4. 修正した場合は `PROJECT_STATUS.md` の更新履歴に記録する
5. 「修正不要と判断した理由」も記録に残す

## AIの回答の使い方（注意点）

AI（Claude 等）の検証は参考情報であり、**最終判断は人間が行う**。

| AIの指摘 | 対応 |
|---|---|
| 明確な事実誤りの指摘 | 文献で確認してから修正 |
| 「研究がない」という指摘 | 追加調査して不確実性を明示するに留める |
| 新しい研究の示唆 | DOI・論文名を確認して、実際に内容を確認してから使う |
| 「これは危険かもしれない」という指摘 | 必ず専門家に確認する。AIの指摘だけで判断しない |

**AIが生成した情報を無確認でコピーしない。**
AI検証はあくまで「見落としのスクリーニング」として使う。

## 更新履歴の管理

検証・修正した場合は以下のフォーマットで `logs/` に記録する。

```
【日付】YYYY-MM-DD
【対象ファイル】例: docs/KNOWLEDGE_MAP.md
【検証方法】AI（Claude）月次レビュー
【指摘内容】例: パルス幅の閾値の表現が古いとの指摘
【対応】修正 / 保留 / 棄却
【修正内容】（修正した場合）例: 感覚刺激のパルス幅を <100μs に明確化
【根拠】（修正した場合）例: 確認した文献
```
