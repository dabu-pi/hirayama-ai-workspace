### 論点 P1 — `氏名カナ` の実意味と `フリガナ` との関係

#### 分析

マイナ確認アプリの標準出力には以下のフィールドが含まれる。

| マイナ確認出力ラベル | 意味 |
|---|---|
| `氏名` | 患者氏名（漢字） |
| `氏名カナ` | 患者氏名（カタカナ）— マイナ確認側の公式フィールド名 |
| `被保険者氏名` | 主被保険者（家族加入時は世帯主）の氏名 |

一方、実機Logger出力の結果：

| ラベル | Logger結果 | 解釈 |
|---|---|---|
| `フリガナ` | ✅ 転記成功（患者マスタに `フリガナ` 列あり） | 保険者情報シートに `フリガナ` 行が存在し、患者マスタ列名と一致した |
| `氏名カナ` | ❌ 患者マスタに列なし | 保険者情報シートに `氏名カナ` 行も存在するが、患者マスタに同名列がない |

**両ラベルが共存する理由:** 保険者情報シートは手動運用を含む縦型（A列=ラベル、B列=値）のため、「院内慣例の `フリガナ`」と「マイナ確認公式フィールドの `氏名カナ`」が両方並列して存在している可能性が高い。

**被保険者カナの可能性を検討:**
- `被保険者氏名`（漢字）は転記成功している
- 被保険者のカナ版は標準マイナ確認出力に含まれない（被保険者は氏名のみ）
- よって `氏名カナ` が被保険者カナである可能性は低い

#### 判定結果（確定）

**`氏名カナ` = 患者本人カナ = `フリガナ` と同一概念。**

- マイナ確認公式フィールド名が `氏名カナ`、院内慣例が `フリガナ`
- どちらも患者本人のカタカナ氏名を指す
- `フリガナ` で転記が成功しているため、実運用上の問題はない

#### 結論: **`氏名カナ` は不要。`フリガナ` に統一する。**

| 対応 | 内容 |
|---|---|
| 患者マスタ | 変更なし。`フリガナ` 列を維持する |
| 保険者情報シート | `氏名カナ` 行は放置してよい（転記スコープ外になるだけで害はない） |
| コード変更 | 不要 |
| 将来の `被保険者フリガナ` 列 | 追加不要（マイナ確認は被保険者カナを出力しない） |

### 論点 P2 — `資格情報(該当保険)` の実意味と代替可能性

#### 分析

マイナ確認アプリの `資格情報(該当保険)` は保険制度の種別を文字列で示す公式フィールド。

| `資格情報(該当保険)` の典型値 | 対応する院内コード |
|---|---|
| "健康保険"（協会けんぽ・組合健保） | `保険種別` = "協会けんぽ" / "組合" |
| "国民健康保険" | `保険種別` = "国保" |
| "後期高齢者医療" | `保険種別` = "後期高齢" |
| "共済組合" | `保険種別` = "共済" |
| "船員保険" | `保険種別` = "退職"に近い |

現行コードの `INS_TYPE_NAME_MAP_`（`Ver3_transferData.js` L769）:
```javascript
"協会けんぽ": 1, "組合": 2, "共済": 3, "国保": 4, "退職": 5, "後期高齢": 6
```

また `区分`（転記成功）は保険区分の短縮形（"健保" / "国保" 等）を保持している。

**代替可能性の評価:**

| 用途 | `資格情報(該当保険)` で得られる情報 | 既存列での代替 |
|---|---|---|
| 本家区分の判定（U5） | 後期高齢かどうか | `保険種別` + `一部負担金割合` で完全代替済み |
| 申請書 U3 保険種別欄 | 健保/国保等の区分 | `保険者番号` から自動判定または `保険種別` |
| 目視確認・監査 | 長文の公式保険名 | `区分` + `保険者名` で実用上十分 |

**差分の有無:**
- `資格情報(該当保険)` が "健康保険" でも、患者マスタの `保険種別` が "協会けんぽ" と "組合" を区別する
- 仮に `資格情報(該当保険)` = "健康保険" を保存しても、協会けんぽと組合の判別には `保険者番号` が必要
- 保存する意味が薄い

#### 判定結果（確定）

**`資格情報(該当保険)` は不要。`保険種別` + `区分` で運用上の要件はすべて満たせる。**

#### 結論: **患者マスタへの独立列追加は不要。現行構成を維持する。**

| 対応 | 内容 |
|---|---|
| 患者マスタ | 変更なし。`保険種別` + `区分` の2列を維持する |
| 保険者情報シート | `資格情報(該当保険)` 行は放置してよい（転記スコープ外で害なし） |
| コード変更 | 不要 |

## 3. 不一致・不足項目の評価

### A. `氏名カナ` と `フリガナ` の関係

**現状:**
- マイナ確認アプリの保険者情報シートに `氏名カナ` ラベルが存在する
- 患者マスタは `フリガナ` として列を持ち、転記成功
- `氏名カナ` は患者マスタに列がなく転記不可

**仮説:**
- `フリガナ` = 患者本人のカナ氏名（被保険者本人の場合）
- `氏名カナ` = 別の被保険者（家族加入時の主被保険者）のカナ氏名

**評価:**

| 選択肢 | 内容 | 推奨度 |
|---|---|---|
| 案1: 現状維持 | `フリガナ` 転記を使い続け、`氏名カナ` は無視 | ★★★ 当面はこれ |
| 案2: 列名統一 | 患者マスタを `フリガナ` から `氏名カナ` に改名 | ★☆☆ コード変更が多い。V3TR 全体に影響 |
| 案3: 列追加 | `被保険者フリガナ`（または `氏名カナ`）列を患者マスタに追加 | ★★☆ 家族加入者がいる場合に有効 |

**推奨:** 案1（現状維持）。`氏名カナ` が何を指すか（被保険者カナか患者カナか）を実際の保険者情報シートで目視確認してから判断する。

### B. `資格情報(該当保険)` の取り扱い

**現状:**
- マイナ確認アプリが出力するラベル
- 内容は「健康保険」「国民健康保険」「後期高齢者医療」等の文字列と推定
- 患者マスタに `保険種別` 列があり、`保険者番号` から算定済み

**評価:**

| 選択肢 | 内容 | 推奨度 |
|---|---|---|
| 案1: 保留 | 現行の `保険種別`（算定値）で十分とみなし、追加しない | ★★★ 推奨 |
| 案2: 列追加 | `資格情報(該当保険)` 列を患者マスタに追加し、マイナ確認の生テキストを保持 | ★★☆ 保険種別の確認用に有用な可能性 |

**推奨:** 案1（保留）。`保険者番号` ベースの `保険種別` 算定が正しく機能していれば不要。ただし `資格情報(該当保険)` の実値を一度目視確認し、`保険種別` との整合を取ること。

### C. `要配慮情報` と `備考` の関係

**現状:**
- 患者マスタに `要配慮情報（備考）` 列が存在（転記対象・列あり・値空）
- 「備考」という一般的な列は患者マスタに別途なし

**評価:**
- `要配慮情報（備考）` はマイナ確認専用の列として位置付ける
- 患者固有のメモ（アレルギー・担当者メモ等）は別途 `運用メモ` 列を設ける候補がある
- 現時点では現状維持。`要配慮情報（備考）` を「マイナ確認からの要配慮フラグ」として固定する

### D. `一部負担金割合` と `負担割合` の関係

**現状:**
- `負担割合`: テキスト（例: "3割"）。保険者情報から転記
- `一部負担金割合`: 数値（1/2/3）。申請書 DP45 用。コード内算出（転記されない）

**評価:**
- 2列の役割が明確に分離されており、現状のまま維持してよい
- ただし列名だけを見ると「どちらが主か」が分かりにくい。列の注釈（シート上のコメント）推奨
