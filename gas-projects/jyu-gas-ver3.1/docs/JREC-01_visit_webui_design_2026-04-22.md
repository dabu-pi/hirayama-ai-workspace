作成日: 2026-04-22

## 0. 前提と制約（絶対に変えてはいけないもの）

| 制約 | 内容 |
|---|---|
| 計算・制度判定はGAS側に残す | 30日ルール・区分判定・多部位逓減・月上限・長期減額はすべてGASが担う |
| 保存はGAS側に残す | `saveVisit_V3` / `saveSelfPayDetailsFromDialog_V3` を使う。WebがDBに直書きしない |
| visitKey / caseKey は変更しない | 既存の来院ヘッダ・ケース管理構造を崩さない |
| 帳票出力は変更しない | 申請書・施術録の生成ロジックは触れない |
| 集計構造は変更しない | 月次集計・窓口負担・保険請求の計算は既存GAS/Sheetsに任せる |
| 既存シートUIと並行運用 | Web UIが動かない場合でも既存メニュー+シートで業務継続できること |

## 1. Web化の設計原則（責務分離）

### 1-1 三層の責務定義

```
┌──────────────────────────────────────────────┐
│  Web UI（ブラウザ）                           │
│  ・入力フォーム表示                           │
│  ・バリデーション（形式チェックのみ）         │
│  ・確認画面表示                               │
│  ・GAS関数の呼び出し（google.script.run）     │
│  ★ 区分判定・金額計算・制度解釈は絶対しない ★  │
└─────────────────┬────────────────────────────┘
                  │ google.script.run（RPC）
┌─────────────────▼────────────────────────────┐
│  GAS（バックエンド）                          │
│  ・入力値のバリデーション（制度ルール）       │
│  ・区分判定（30日ルール / 受傷日経過 etc）    │
│  ・金額計算（単価・逓減・長期減額）           │
│  ・Sheetsへの書き込み                        │
│  ・呼び出し結果をJSONで返す                  │
└─────────────────┬────────────────────────────┘
                  │ SpreadsheetApp
┌─────────────────▼────────────────────────────┐
│  Google Sheets（データ・ロジック正本）        │
│  ・来院ヘッダ・ケース・自費明細の永続保存     │
│  ・数式（C2患者ID抽出 / B3患者名VLOOKUP等）  │
│  ・月次集計・帳票出力の元データ              │
└──────────────────────────────────────────────┘
```

### 1-2 WebがやってよいことYES / やってはいけないことNO

| 操作 | 可否 | 理由 |
|---|---|---|
| 入力値の空チェック・形式チェック | ✅ YES | UI上の誤入力防止。GAS側でも再チェックする |
| 受傷日が来院日より未来かチェック | ✅ YES | 明らかな入力ミス防止。制度判断ではない |
| 区分（初検/再検/後療）の表示 | ✅ YES（GAS計算結果を表示するのみ） | GASが返した値を見せるだけ |
| 区分をWebで判定してGASに送る | ❌ NO | 30日ルール等の制度解釈がWebに漏れる |
| 月上限に達しているか判断する | ❌ NO | GAS側が唯一の判断者 |
| 金額をWebで計算する | ❌ NO | 単価・逓減率・消費税はGAS側で管理 |
| `saveVisit_V3` の代替保存を自作する | ❌ NO | 保存経路の分岐はデータ不整合の原因 |
| 申請書・施術録の生成を変更する | ❌ NO | 帳票生成ロジックは触れない |
| データをWebサーバー・外部DBに保存する | ❌ NO | SSOTはSheets |

## 2. 対象業務の分解

### 2-1 来院〜会計の業務フロー

```
① 受付（患者特定）
     ↓ Web UI 実装済み（Phase 1-2-3）
② 来院情報入力（部位・傷病・受傷日・治療法）
     ↓ ← ここが最大の課題
③ 区分確認（GAS算定結果の確認）
     ↓ GAS側に任せる
④ 自費明細入力（メニュー・数量）
     ↓ Web UI 実装済み（selfPayWeb）
⑤ 会計確認（請求額・窓口負担 確認）
     ↓
⑥ 決済・領収（現金/カード・領収書印刷）
```

### 2-2 各ステップの現状と課題

| ステップ | 現状 | Web化状況 | 課題 |
|---|---|---|---|
| ① 受付 | シートB2プルダウン / Web患者検索 | **実装済み** | 安定運用 待ち |
| ② 来院入力 | セル直接入力（A12-H13, A36-H37） | **未実装** | `readRowNewUI_` 依存が最大ボトルネック |
| ③ 区分確認 | E2-E6セルに自動表示 | **未実装**（読み取りUI） | GAS計算結果をWebで見せるだけ → 難度低 |
| ④ 自費明細 | selfPayDialog → selfPayWeb | **実装済み** | B7(会計区分)の渡し方を改善余地あり |
| ⑤ 会計確認 | シートE2-E6を施術者が確認 | **未実装** | Web表示はシンプル。保存済みデータ読み取りのみ |
| ⑥ 決済・領収 | 手作業（現金のみ）| **対象外** | 既存運用で問題なし |

### 2-3 ステップ② 来院入力の詳細分解

来院入力は「UIシート依存」と「制度判定」が混在しており、Web化のボトルネックになる。

```
saveVisit_V3（現行の保存関数）
  └── readRowNewUI_()
        └── シートA1定数・行番号定数を読んでセル値を取得
              ├── B2: 患者表示文字列 → C2数式でID抽出
              ├── B4: 来院日
              ├── B5: ジム会員フラグ
              ├── B7: 会計区分
              ├── B8: 慢性候補フラグ
              ├── D8: 次回予約フラグ
              ├── F8: 新規区分
              ├── A12:H13: ケース1（部位/傷病/受傷日/治療法/終了日/転帰）
              └── A36:H37: ケース2（同上）
```

**Web化の選択肢:**

| 案 | 内容 | リスク |
|---|---|---|
| **A案（推奨）: シート経由** | WebからGASを呼び、GASがシートの該当セルに書き込んでから `saveVisit_V3` を呼ぶ | セル書き込み → 保存の2段階で整合が必要 |
| B案: JSON経路追加 | `saveVisit_V3` をラップする `saveVisitFromWeb_V3(json)` を新設。`readRowNewUI_` を迂回 | `readRowNewUI_` と同等のマッピングを新設する工数。乖離リスク |
| C案: そのままシート入力 | Web化しない。「シートをGoogleスプレッドシートアプリで開く」運用継続 | 現状維持。スマホ操作は煩雑なまま |

**推奨は A案（シート経由）:**  
既存の `saveVisit_V3` / `readRowNewUI_` を変更せず、WebからGASにUI値を渡してGASがシートセルを更新 → その後 `saveVisit_V3` を呼ぶ。既存保存経路を1本に保てる。

### 3-4 保存の分離（下書き vs 確定）

| ボタン | 動作 | GAS関数（案） | 備考 |
|---|---|---|---|
| 下書き保存 | UIシートのセルに値を書き込む。`saveVisit_V3` は呼ばない | `writeVisitDraftToUI_V3(json)` | 施術途中・中断時に使う |
| 確定保存 | UIシートへ書き込み → `saveVisit_V3` を呼ぶ | `saveVisitFromWebUI_V3(json)` | 最終確定時のみ |

**下書き保存の意義:**
- 施術者がフォームを書きかけで離席しても来院データが消えない
- UIシートが「正の途中状態」として機能する（既存動作と整合）
- 確定保存時は下書きを読んで `saveVisit_V3` → 既存パスを変えない

**「確定済み」チェック（二重保存防止）:**
- `saveVisit_V3` 成功後は H8（ステータス）に保存済みマーカーが書き込まれる
- 確定保存前に H8 を読んで「既に保存済みです」を表示する

## 4. 危険ポイント整理

### 4-1 ロジック崩壊リスク（絶対に踏んではいけない）

| リスク | 内容 | 対策 |
|---|---|---|
| **WebでKubun（区分）を独自判定する** | 30日ルール・複数ケースの期間判断をWebで実装すると、GAS側と二重定義になる。将来の制度改正で一方だけ更新されてサイレントに不整合が生じる | Webは区分を「表示する」だけ。計算は常にGASに投げる。絶対に自前で判定しない |
| **新しい保存経路を作る** | `saveVisit_V3` を迂回してSheets APIで直接書く実装は、ロギング・金額計算・H8更新が漏れる | 保存は必ず `saveVisit_V3` か `saveSelfPayDetailsFromDialog_V3` 経由にする |
| **単価・料金をWebに持つ** | メニュー料金をHTML/JSにハードコードすると設定シートとの乖離が起きる | `getSelfPayMenuMaster_V3` で毎回取得する |
| **visitKey をWebで変形する** | WebでvisitKeyを変換・再生成すると既存の来院ヘッダと対応が取れなくなる | visitKeyはP01で確定したものをそのまま引き回す。加工しない |
| **readRowNewUI_ の読み取りをWebで代替する** | WebがUIシートのセル配置を「知っている」状態にすると、シート構造変更時にWeb側も修正が必要になる | Webはセル番地を知らない。値を渡すだけ。GASがUIシートに書いて、GASが `readRowNewUI_` で読む |

### 4-2 Webが判断してはいけない項目（GAS専管）

| 項目 | 理由 |
|---|---|
| 初検 / 再検 / 後療の区分 | 30日ルール + 受傷日経過日数 + 月の来院状態が必要 |
| 月上限に達しているか | 当月の既存来院ヘッダを集計する必要がある |
| 多部位逓減の適用 | ケース数・傷病の組み合わせルールが複雑 |
| 長期減額の適用 | 受傷日からの経過月数が必要 |
| 保険請求金額 | 単価×算定ルール（設定シート依存） |
| 窓口負担割合 | 患者マスタの保険種別・高齢者区分 |
| 申請書への算定内容 | 帳票生成ロジックは既存GASのみが担う |

### 4-3 監査・確認ログとして必要な項目

Web UI 経由での保存は「誰がいつWebで入力したか」の追跡が難しい。以下を `saveVisitFromWebUI_V3` で必ずGASログ出力する:

| ログ項目 | 内容 |
|---|---|
| visitKey | 対象来院 |
| 実行者 | `Session.getActiveUser().getEmail()` |
| タイムスタンプ | `new Date()` |
| 送信JSON（要約） | 受け取った入力値（患者ID / ケース数 / 会計区分） |
| 保存結果 | `{ok: true/false, caseKeys: [...]}` |
| GASのLogger出力 | `[saveVisitFromWebUI] visitKey=... result=...` |

### 4-4 段階導入時の警告

| 警告 | 内容 |
|---|---|
| **P02（来院入力）は他のページより実装コストが10倍高い** | `readRowNewUI_` 依存の解体、下書き/確定分離、セル書き込み→GAS保存の2段階が必要 |
| **P02を先に実装してはいけない** | P01（患者検索）とP04（自費明細）が安定稼働する前にP02を実装すると、問題の切り分けが困難になる |
| **「シートUIを廃止したい」という欲を出さない** | WebとシートUIの並行期間は最低3ヶ月取ること。シートUIは生命線 |
| **GASの応答時間に注意** | 来院入力フォームは「下書き保存」が遅いと入力体験が壊れる。GASの冷間起動（数秒）を考慮してローディング表示を必ず出す |

## 5. 推奨導入順

### 5-1 フェーズ区分

```
現在地 → Phase A → Phase B → Phase C → Phase D（任意）
```

| Phase | 内容 | 前提条件 | 優先度 |
|---|---|---|---|
| **現在地** | P01（患者検索・選択）+ P04（自費明細）実装済み | — | 完了 |
| **Phase A** | 実運用テスト（コード変更なし） | P01/P04 が安定稼働 | **最高（今すぐ）** |
| **Phase B** | P05（会計確認）読み取りUI | Phase A で2週間以上問題なし | 中 |
| **Phase C** | P02（来院入力フォーム）下書きのみ | Phase B 完了 + `readRowNewUI_` 依存コスト調査完了 | 低 |
| **Phase D** | P02 確定保存（`saveVisit_V3` 連動） | Phase C で下書きが安定 | 低（慎重に） |

### 5-2 Phase A 実運用テスト（最優先・コード変更なし）

**目的:** 今あるWebUIを実際の受付業務で使い、問題点を洗い出す

| 確認項目 | 方法 |
|---|---|
| 患者検索の速度 | 実際の患者マスタ件数で体感確認。3秒超えたら改善を検討 |
| スマホ/タブレット操作性 | iPhoneまたはAndroidで患者選択→自費入力の一連を試す |
| B7（会計区分）の渡り方 | selfPayWebがB7を正しく読めているか確認 |
| GASログのエラー率 | 1週間後に `[getSelfPayDataByVisitKey]` エラーが何件か確認 |
| 従来シートUIとの競合 | Web UIで選択後にシートUI（プルダウン）を使った場合の挙動を確認 |

**Phase A の判定基準（Phase B への進み方）:**
- 1〜2週間で重大バグが出ない
- 受付スタッフが「シートを開かずに自費入力できた」を報告できる
- B7（会計区分）の読み取りが実用範囲で正確

### 5-3 Phase B: P05 会計確認ページ（読み取りのみ）

**実装範囲（小さく）:**

```javascript
// GAS側: 来院保存済みデータを読んで金額情報を返す
function getAccountingSummary_V3(visitKey) {
  // 来院ヘッダから: 窓口負担 / 保険請求 / 自費合計 / 会計区分
  // 患者マスタから: 氏名 / 保険種別
  // return { patientName, visitDate, accountingType, selfPayTotal,
  //          windowBurden, insuranceClaim, statusText }
}
```

**UI:**
- visitKey を受け取り、金額サマリを表示するだけ
- 「保存する」ボタンはない（読み取り専用）
- 「患者検索に戻る」リンクのみ

**なぜ Phase B で入れるか:**
- 施術者がスマホで「今日の会計確認」ができるようになる
- 実装コストが低い（読み取りのみ）
- P04（自費明細）保存後の遷移先として自然

### 5-4 Phase C: P02 来院入力フォーム（下書きのみ）

**実装前に必ず調査する事項:**

| 調査項目 | 確認方法 |
|---|---|
| `readRowNewUI_` のセル番地マッピング全量 | `Ver3_core.js` の `readRowNewUI_` を読んで全入力項目を洗い出す |
| `autofillFromPreviousVisit_V3` の UI書き込みパス | UIシートへの書き込み範囲を特定する |
| 部位/傷病/治療法のマスタリスト取得方法 | 設定シートの列名とリスト生成関数を確認する |
| `writeVisitDraftToUI_V3` の実装可能性 | UIシートへの書き込みでバリデーション（セル保護等）が邪魔しないか確認 |

**Phase C の成功基準:**
- Webフォームから下書き保存 → UIシートに値が入る
- UIシートに値が入った状態で既存の「保存」ボタン（シートUI）で `saveVisit_V3` が正常動作する
- Web経由の下書きが既存フローを壊さない

### 5-5 Phase D: P02 確定保存連動（慎重に）
