## 4. 観察項目

### 4-1 検索速度

- 患者マスタ件数が増えた状態での体感速度を確認する
- 目安: キーワード入力後 3 秒以内に候補が表示されること
- 3 秒を超える場合は記録欄に件数・端末・回線環境を記録する

### 4-2 保存成功 / 失敗

- 自費明細の保存が正常完了するか確認する（「N 件の自費明細を保存しました」表示）
- 失敗した場合はエラーメッセージ・操作手順・visitKey を記録する
- GAS ログ（Apps Script エディタ > 実行数）でエラー率も確認する（1 週間後メド）
- 確認対象関数: `getSelfPayDataByVisitKey_V3` / `saveSelfPayDetailsFromDialog_V3`

### 4-3 操作の迷い

- 「次に何をすればいいかわからない」場面があれば記録する
- 特に「患者選択後の遷移」「自費明細保存後の戻り先」を重点確認する

### 4-4 スマートフォン操作性

- iPhone または Android の実機で以下の一連を試す
  1. Web App URL を開く
  2. 患者をキーワード検索
  3. 患者カードをタップ
  4. 「自費明細入力 →」をタップ
  5. メニュー選択・数量入力・保存
- タップ対象が小さすぎる / スクロール不要な部分がスクロールされる / キーボードが出てUIが隠れる などを確認する

### 4-5 受付業務への適合性

- B7（会計区分）が selfPayWeb に正しく引き継がれているか確認する
- 既存スプレッドシートメニュー（来院登録・算定実行）と Web UI を同日混在使用した場合の挙動を確認する
- 「シートを開かずに自費入力を完結できたか」を記録する

## 8-A. 2026-04-25 Phase A 試用記録追記

- Phase A は 2026-04-23 初回通過から継続中（コード変更なし）
- 今日は主に「前回施術再読込（自費明細）」の実装前調査を Phase A と並行して実施
- `openSelfPayDialog_V3` のシグネチャ・データフロー・HTML初期化ロジックをコード精査（コード変更なし）
- 調査結果を `docs/JREC-01_prev_selfpay_reload_investigation_2026-04-25.md` に記録済み
- 既存 P01/P04 の保存経路（`saveSelfPayDetailsFromDialog_V3`）に変更なし・継続稼働中

### 結果

- **PASS:**
  - Phase A 継続稼働（2026-04-23 から問題報告なし）
  - 実装前調査が Phase A の動作に干渉しなかったこと（コード変更なし）
  - 前回施術再読込の未解決論点①②がともに解消できる見通しが立ったこと

- **OBSERVE:**
  - ダイアログ起動時に `google.script.run` を2回（`getCurrentVisitKey_V3` → `getSelfPayMenuMaster_V3`）連鎖呼び出しする構造のため、起動後に一瞬「読み込み中」表示が出る。現時点で業務上の問題にはなっていないが、患者マスタが増えた際の体感速度は引き続き観察対象
  - 自費明細の「前回引き継ぎ」が現状は手入力。来院ごとに同じメニューを再入力している場面があり得るため、Phase A 完了後の前回再読込実装は現場ニーズに合致している

- **NG:** なし

- **未確認:**
  - 2026-04-25 当日の実受付での直接試用（次回確認）
  - GAS 実行ログのエラー率（Phase A 開始から約2日経過。1週間後メドで確認）
  - スマートフォン実機での連続操作（タップ精度・キーボード表示時のUI隠れ）
  - 患者マスタ件数が増えた際の検索レスポンス（目安: 3秒以内）

### 実運用上の気づき

- Web UI の入力補助層としての役割は現時点で機能している。Sheets 側の算定ロジック・保存処理は一切変更していないため、既存の保険請求フローへの影響なし

- Phase A 中に発見した「自費明細前回引き継ぎ」のニーズは、設計・実装方針まで到達した。Phase B 移行判断（2026-05-07 目安）と並行して実装準備を進める状態に入った
