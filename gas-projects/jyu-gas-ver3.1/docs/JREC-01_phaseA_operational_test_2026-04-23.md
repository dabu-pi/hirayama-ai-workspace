## 1. 目的

実装済みの Web UI（患者検索 + 自費明細）を実際の受付業務で試用し、
Phase B（会計確認ページ）への移行判断に必要な問題点・使い勝手データを収集する。

コードは変更しない。あくまで「動くものを使ってみる」フェーズ。

## 4. 観察項目

### 4-1 検索速度

- 患者マスタ件数が増えた状態での体感速度を確認する
- 目安: キーワード入力後 3 秒以内に候補が表示されること
- 3 秒を超える場合は記録欄に件数・端末・回線環境を記録する

### 4-2 保存成功 / 失敗

- 自費明細の保存が正常完了するか確認する（「N 件の自費明細を保存しました」表示）
- 失敗した場合はエラーメッセージ・操作手順・visitKey を記録する
- GAS ログ（Apps Script エディタ > 実行数）でエラー率も確認する（1 週間後メド）
- 確認対象関数: `getSelfPayDataByVisitKey_V3` / `saveSelfPayDetailsFromDialog_V3`

### 4-3 操作の迷い

- 「次に何をすればいいかわからない」場面があれば記録する
- 特に「患者選択後の遷移」「自費明細保存後の戻り先」を重点確認する

### 4-4 スマートフォン操作性

- iPhone または Android の実機で以下の一連を試す
  1. Web App URL を開く
  2. 患者をキーワード検索
  3. 患者カードをタップ
  4. 「自費明細入力 →」をタップ
  5. メニュー選択・数量入力・保存
- タップ対象が小さすぎる / スクロール不要な部分がスクロールされる / キーボードが出てUIが隠れる などを確認する

### 4-5 受付業務への適合性

- B7（会計区分）が selfPayWeb に正しく引き継がれているか確認する
- 既存スプレッドシートメニュー（来院登録・算定実行）と Web UI を同日混在使用した場合の挙動を確認する
- 「シートを開かずに自費入力を完結できたか」を記録する

## 5. 記録テンプレート

試用のたびに以下を記録する（問題がなかった日も「問題なし」と記録する）。

```
日付: YYYY-MM-DD
利用場面: （例: 朝の受付 / 自費のみ患者）
問題: （なし / 内容を簡潔に）
重要度: 高（業務継続不可） / 中（不便だが回避可能） / 低（改善希望）
再現性: 毎回 / 特定条件下 / 1回のみ
備考: （端末・ブラウザ・患者件数・エラーメッセージ等）
```

## 6. Phase A 完了判定基準

以下を**すべて**満たした時点で Phase A 完了とし、Phase B 設計に進む。

| 判定基準 | 確認方法 |
|---|---|
| 1〜2 週間で**重大バグが出ない**（業務継続不可レベルの問題なし） | §5 記録テンプレートで高重要度がゼロ |
| 受付スタッフが「シートを開かずに自費入力できた」を確認できる | §5 記録テンプレートの利用場面欄 |
| B7（会計区分）の読み取りが実用範囲で正確 | 保存後に D7/F7/H8 が正しく更新されているか Sheets で確認 |
