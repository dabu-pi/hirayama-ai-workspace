## 1. 生成前確認（毎月必須）

### 1-1. データ確認チェックリスト

前月の来院データがスプレッドシートに全件登録されているか確認する。

| 確認項目 | 確認方法 | 判断基準 |
|---|---|---|
| 対象月の来院行が全件ある | 「来院ヘッダ」シートで対象月の行数を目視 | 通院日数と一致すること |
| 患者マスタの保険情報が最新か | 「患者マスタ」シートで各患者の保険者番号・負担割合を確認 | 保険証更新がある場合は要更新 |
| 施術明細が生成されているか | 「施術明細」シートに対象月の行があること | 来院登録後に自動生成される |
| 転記データシートが汚染されていないか | 「申請書_転記データ」シートの最終行が前月以前のデータか確認 | B案は毎回クリーン生成するため問題なし |

### 1-2. Cloud Run ヘルスチェック（月次 1 回）

**Apps Script エディタ（スプレッドシート > 拡張機能 > Apps Script）で実行:**

1. 関数プルダウンで **`V3TR_smokeHealth`** を選択
2. 「実行」をクリック
3. 期待値: `✅ /health OK / HTTP: 200 / {"status":"ok"}`

> **NG の場合:** §5「Cloud Run 再デプロイが必要な条件」を確認する。

### 1-3. スクリプトプロパティ確認（初回 or PC変更時のみ）

Apps Script > プロジェクトの設定 > スクリプトプロパティ:

| プロパティ名 | 期待値 |
|---|---|
| `APPGEN_ENDPOINT` | `https://jrec-appgen-server-xxxxxxxxxx-an.a.run.app`（末尾スラッシュなし）|
| `APPGEN_SECRET` | Secret Manager `JREC_APPGEN_SECRET_KEY` と同じ値 |

## 2. 実行手順

### STEP 1: スプレッドシートを開く

- スプレッドシートID: `1rXWkfAc_ppOfMV5Dxmb3maX9ORVrZbpSOX2Lz7RouZM`
- アカウント: 柔整業務用 Google アカウントで開く

### STEP 2: メニューから生成を開始する

1. メニューバー「**柔整ツール**」をクリック
2. 「**【B案】申請書を生成して Drive に保存**」をクリック
3. 月指定ダイアログが出たら対象月を入力（例: `2026-02`）
4. **OK** をクリック

> デフォルトは当月。先月分は必ず明示指定する。

### STEP 3: プリフライト確認

処理前に GAS がデータを検証し、問題があればダイアログを表示する。

**hard error（除外対象）:**

| エラー内容 | 対応 |
|---|---|
| 患者氏名が空 | 患者マスタを確認・修正してから再実行 |
| 保険者番号が空 | 患者マスタを確認・修正してから再実行 |
| 必須項目が空欄 | ダイアログ内容を確認 → 問題患者を除外して続行 or 修正後に再実行 |

> hard error 患者を除外して続行する場合は「続行」を選択する。残患者は生成される。

**warning（警告のみ・続行可能）:**

| 警告内容 | 対応 |
|---|---|
| 一部負担金割合 = 0 | 患者マスタの「負担割合」を確認（0.1 / 0.2 / 0.3 のいずれかが正常値）|
| 窓口負担額 = 0 | 金額算定が正しいか来院明細を確認 |
| 請求金額 = 0 | 当月の来院がないか算定漏れか確認 |
| 合計金額不一致 | 施術明細の再計算が必要な可能性（来院登録し直し）|

> warning が出た場合: 内容を確認してから「OK」（続行）または「いいえ」（中断して修正）を選ぶ。

### STEP 4: 生成完了を確認する

処理中トースト:
1. `NDJSON 生成中...`
2. `Cloud Run に送信中...`
3. `Drive に保存中...`

完了アラートに以下が表示されれば成功:
- `保存: X 件`（患者数と一致）
- `エラー: 0 件`
- 保存先フォルダ URL が表示

## 3. 生成後確認

### 3-1. Drive ファイル確認

1. 完了アラートに表示された Drive フォルダ URL を開く
2. `output/YYYY-MM/` フォルダに患者ごとの xlsx が保存されていることを確認
3. ファイル命名: `申請書_{患者ID}_{YYYY-MM}_{HHMMSS}.xlsx`

### 3-2. 目視確認項目

各 xlsx を開いて以下を確認する:

| 確認欄 | 確認内容 |
|---|---|
| 患者氏名（U1 相当）| 正しい氏名が入っているか |
| 保険者番号 | 患者マスタと一致しているか |
| 保険種別（U5）| 後期高齢者は「⑥.後期」、一般は「①.本人」or「④.家族」 |
| 給付割合（U6）| 9割給付=「10・⑨」、8割=「⑧・７」、7割=「８・⑦」|
| 登録記号番号 | 左欄（契XXXXXXX）・中欄・右欄に分割されているか |
| 施術料金合計 | 来院明細の金額と一致しているか |
| 負傷の原因（D4）| 文字化けや日付の書き方が正しいか（英語日付が混入していないか）|
| 楕円（丸付け）| 選択肢が1箇所だけ丸付けされているか（二重丸や位置ずれがないか）|

### 3-3. 生成ログ確認

スプレッドシートの「**申請書_生成ログ**」シートで:
- 対象月の行に OK が記録されているか
- エラー行がないか

## 4. 再生成が必要な条件

以下のいずれかに該当する場合は再生成する。同じ月を再生成するとタイムスタンプ付きで上書きせず新規保存される（旧ファイルは手動削除する）。

| 条件 | 対応 |
|---|---|
| 生成後に患者マスタ（保険情報）を修正した | 修正後に B案を再実行 |
| 目視確認で氏名・金額・欄が間違っていた | GAS / Python / 設定シートのどこが誤りかを確認してから再実行 |
| プリフライト warning を無視して生成したが内容に問題があった | 元データを修正してから再実行 |
| 来院登録を後から追加・修正した | 施術明細を再生成してから B案を再実行 |
| Cloud Run に新 Revision をデプロイした | 必ず再生成して帳票が正しいことを目視確認する |
| 旧ファイルを参照していた（deploy 前のイメージで生成されたもの）| 新 Revision deploy 後に再生成 |

> **注意:** 再生成してもスプレッドシートのデータは変わらない。Drive に新しいファイルが追加されるだけ。不要な旧ファイルは Drive から手動削除する。

## 5. Cloud Run 再デプロイが必要な条件

以下に該当する場合は Cloud Run の再デプロイが必要。

### 再デプロイが必要

| 条件 | 理由 |
|---|---|
| `write_application.py` を修正した | Python コードは Docker イメージに焼き込まれている。clasp push では反映されない。 |
| `server.py` を修正した | 同上 |
| `application_template.xlsx`（申請書テンプレ）を修正した | Docker イメージに同梱されている。|
| `requirements.txt` を修正した（ライブラリ追加等）| 同上 |
| Dockerfile を修正した | 同上 |
| `/health` が 200 以外を返す | 起動失敗。ログ確認後に再デプロイ。 |

### 再デプロイ不要（clasp push のみでよい）

| 条件 | 理由 |
|---|---|
| `Ver3_core.js` / `Ver3_amounts.js` / `Ver3_transferData.js` 等を修正 | GAS のみ。`clasp push` で反映される。 |
| スプレッドシートの設定シートを変更 | GAS が動的に読む。コード変更なし。 |
| スクリプトプロパティを変更 | GAS 設定。コード変更なし。 |

### 再デプロイ手順（簡易版）

```bash
# jyu-gas-ver3.1 ディレクトリで実行
cd C:\hirayama-ai-workspace\workspace\gas-projects\jyu-gas-ver3.1

# Cloud Build でイメージビルド & Cloud Run へデプロイ
gcloud builds submit --tag asia-northeast1-docker.pkg.dev/hirayama-jrec-appgen/jrec-appgen/server:latest .

gcloud run deploy jrec-appgen-server \
  --image=asia-northeast1-docker.pkg.dev/hirayama-jrec-appgen/jrec-appgen/server:latest \
  --platform=managed \
  --region=asia-northeast1

# デプロイ後に必ずヘルスチェック
curl $(gcloud run services describe jrec-appgen-server --platform=managed --region=asia-northeast1 --format="value(status.url)")/health
# 期待: {"status":"ok"}
```

詳細手順: `docs/JREC-01_CloudRun_デプロイ手順.md`

## 6. 手書き対応が残る項目

以下の欄は**現在システムで自動生成されない**。申請書 xlsx を印刷した後に手書きで記入する。

| 欄 | 内容 | システム対応予定 |
|---|---|---|
| **D5 施術証明欄** | 施術者の署名・捺印欄 | 未定（手書き運用継続）|
| **D5 委任欄** | 患者の委任状記名・捺印 | 未定（手書き運用継続）|
| **D2 継続月数（摘要欄）** | 「施術開始 YYYY年MM月 継続X月目」等の記載 | 実装済みだが、**最終確認・加筆は手動**（摘要欄の出力内容を目視確認すること）|
| **D2 頻回確認（長期欄）** | 5ヶ月超・月16回超の場合の記載 | 実装済みだが、**最終確認・加筆は手動** |

> **手書き記入タイミング:** Drive から xlsx を印刷した後、提出前に記入する。

## 7. 保存先・管理方法

### Drive 保存先

| 項目 | 値 |
|---|---|
| 保存場所 | Google Drive `output/YYYY-MM/` フォルダ（スプレッドシートと同一アカウントの Drive）|
| ファイル命名 | `申請書_{患者ID}_{YYYY-MM}_{HHMMSS}.xlsx` |
| 上書きルール | 同名ファイルは上書きしない。再生成すると別ファイルが追加される。 |

### 月別管理方針

| 操作 | タイミング |
|---|---|
| 旧ファイル（再生成前のもの）を削除 | 最終版の帳票が確定したタイミングで手動削除 |
| 提出済みの管理 | Drive フォルダに残す（提出月フォルダで管理）|
| 再生成後の旧ファイル整理 | `申請書_{患者ID}_{YYYY-MM}_旧.xlsx` などにリネームして保管 or 削除 |

### スプレッドシート

| シート名 | 役割 |
|---|---|
| 来院ヘッダ | 日次入力の正本（B案生成の元データ）|
| 施術明細 | 来院登録時に自動生成（申請書欄の詳細）|
| 申請書_転記データ | GAS が生成する中間データ（B案実行時に毎回再生成）|
| 申請書_生成ログ | B案の生成ログ（成功/エラー記録）|
| 患者マスタ | 患者情報（保険者番号・負担割合 等）|

## 8. トラブルシューティング早見表

| 症状 | 最初に確認する場所 | 対応 |
|---|---|---|
| `APPGEN_ENDPOINT が未設定` エラー | Script Properties | プロパティを再確認・再設定 |
| HTTP 401 | Secret Key の一致確認 | `APPGEN_SECRET` と GCP Secret Manager が一致しているか確認 |
| HTTP 400 | 対象月の来院データ・患者データ | 来院ヘッダに対象月のデータがあるか確認 |
| HTTP 500 | GCP Console > Cloud Logging | スタックトレースを確認。Python 側エラー。|
| 接続タイムアウト | Cloud Run URL の確認 | `gcloud run services describe jrec-appgen-server ...` で URL 再確認 |
| Drive 保存失敗 | Drive の権限 | スプレッドシートを開いているアカウントの Drive 書込み権限を確認 |
| 楕円がずれる / テンプレ文字が消える | Cloud Run Revision | `write_application.py` の SELECTION_SPLIT_MAP 方式が反映されているか（Revision 確認）|
| 英語日付が D4 に混入 | GAS 側の日付型セル | GAS の `get()` が `Utilities.formatDate` を使っているか。または Python の `_normalize_date_str()` が動いているか確認 |
| `/health` が 200 以外 | Cloud Logging | gunicorn 起動ログを確認。Revision 00019-w8n が動いているか確認 |
