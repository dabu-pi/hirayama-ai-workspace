# JREC-01 mixed case 表示改善メモ

- 作成日: 2026-03-16
- 対象: JREC-01（柔整毎日記録システム）
- 対象領域: 来院ヘッダ表示 / mixed case 説明性改善
- 位置づけ: 実装判断メモ（制度SPEC本文の補助メモ）

---

## 1. 背景

JREC-01 では、1来院（visitKey）に対して複数ケース（case1 / case2）が同時に存在しうる。
この mixed case において、来院ヘッダ1行だけでは **「表示区分」と「実際に課金した区分」** が一致しないことがあり、後から見返したときに意味が分かりにくい。

現状確認できた mixed case の例:

- `hirayamaka_2026-03-02`
  - case1 = 再検
  - case2 = 初検
  - 来院ヘッダでは `区分=再検 / 初検料=0 / 再検料=410 / caseKey2あり / 初検抑制理由あり`

- `touji_2026-03-03`
  - case1 = 後療
  - case2 = 初検
  - 来院ヘッダでは `区分=後療 / 初検料=0 / 再検料=0 / caseKey2あり`

- `mineo_2026-03-03`
  - case1 = 後療のみ
  - 来院ヘッダでは `区分=後療 / 初検料=0 / 再検料=0 / 明細合計=505 / 来院合計=505`

このため、mixed case に対しては、既存の `区分` 列だけでは説明性が不足する。

---

## 2. 今回の設計判断

### 2-1. 基本方針

既存の `区分` 列は消さずに残す。
ただし、mixed case の説明性を高めるため、**説明用の追加列** を来院ヘッダに持たせる。

### 2-2. 考え方

- `区分` は当面、**表示区分**（来院登録時の kubun1 or kubun2 の先頭値）
- 新設する `算定区分` は、**実際に課金した区分**
- mixed case の有無は `Mixed区分`
- case1 / case2 の中身は `case1要約` / `case2要約`
- なぜその算定になったかは `課金理由要約`

### 2-3. 目的

- 来院ヘッダ1行だけ見ても mixed case の意味が分かる
- 金額ロジックはそのままに、説明可能性だけを強化する
- 監査時・見返し時に「なぜこの金額なのか」を説明しやすくする

---

## 3. 追加する列

来院ヘッダに以下の5列を追加する。

1. `算定区分`
2. `Mixed区分`
3. `case1要約`
4. `case2要約`
5. `課金理由要約`

### 3-1. 追加位置

`HEADER_COLS` の `caseKey2` の**直後**に追加する（末尾追記）。
理由:
- 既存の数式・転記・列参照を壊しにくい
- 既存の金額列を左側に残せる
- 説明用の列を右側にまとめられる

---

## 4. 各列の意味

### 4-1. 算定区分

実際に料金計上した区分。
値候補:

- `初検`
- `再検`
- `後療`
- `算定なし`

### 4-2. Mixed区分

来院内に複数ケースがあるかどうか。
値候補:

- `通常`
- `Mixed`

### 4-3. case1要約

caseNo=1 の区分を短い文で表す。
例:

- `case1:初検`
- `case1:再検`
- `case1:後療`
- `case1:なし`

### 4-4. case2要約

caseNo=2 の区分と抑制状況を短い文で表す。
例:

- `case2:なし`
- `case2:初検`
- `case2:初検(抑制)`
- `case2:再検`
- `case2:後療`

### 4-5. 課金理由要約

なぜその `算定区分` になったかを短く表す。
例:

- `初検のみ`
- `再検のみ`
- `後療のみ`
- `初検抑制のため再検採用`
- `算定可能な初検ありのため初検採用`
- `初検抑制かつ再検対象なし`

---

## 5. 値の生成ルール

### 5-1. Mixed区分

- `kubun2`（または caseKey2）が空でない → `Mixed`
- それ以外 → `通常`

### 5-2. 算定区分

優先順位は次のとおり。

1. `initFee > 0` → `初検`
2. それ以外で `reFee > 0` → `再検`
3. それ以外で case に後療区分がある → `後療`
4. それ以外 → `算定なし`

### 5-3. case1要約

caseNo=1 の区分（kubun1）をそのまま要約にする。

- `kubun1` が `初検` → `case1:初検`
- `kubun1` が `再検` → `case1:再検`
- `kubun1` が `後療` → `case1:後療`
- `kubun1` が空 → `case1:なし`

### 5-4. case2要約

- `kubun2` が空 → `case2:なし`
- `kubun2` が `初検` かつ `reasons` に初検抑制あり → `case2:初検(抑制)`
- `kubun2` が `初検` かつ抑制なし → `case2:初検`
- `kubun2` が `再検` → `case2:再検`
- `kubun2` が `後療` → `case2:後療`

初検抑制の判定: `reasons` 配列に `"初検抑制"` を含む文字列がある場合。

### 5-5. 課金理由要約

パターン判定の優先順位:

1. `hasBillableInitial && !mixedCase` → `初検のみ`
2. `hasBillableInitial && mixedCase` → `算定可能な初検ありのため初検採用`
3. `!hasBillableInitial && reFee > 0 && mixedCase && initSuppressed` → `初検抑制のため再検採用`
4. `!hasBillableInitial && reFee > 0 && !mixedCase` → `再検のみ`
5. `!hasBillableInitial && reFee === 0 && hasKoryo && mixedCase && initSuppressed` → `初検抑制かつ再検対象なし`
6. `!hasBillableInitial && reFee === 0 && hasKoryo` → `後療のみ`
7. その他 → `算定なし`

補助フラグ `initSuppressed`:
- `reasons` 配列に `"初検抑制"` を含む文字列がある場合 `true`

---

## 6. mixed case の代表パターン

### M01

- case1 = 再検、case2 = 初検（抑制）
- 期待値:
  - `算定区分 = 再検`
  - `Mixed区分 = Mixed`
  - `case1要約 = case1:再検`
  - `case2要約 = case2:初検(抑制)`
  - `課金理由要約 = 初検抑制のため再検採用`

### M02

- case1 = 再検、case2 = 初検（算定可）
- 期待値:
  - `算定区分 = 初検`
  - `Mixed区分 = Mixed`
  - `case1要約 = case1:再検`
  - `case2要約 = case2:初検`
  - `課金理由要約 = 算定可能な初検ありのため初検採用`

### M03

- case1 = 後療、case2 = 初検（抑制）
- 期待値:
  - `算定区分 = 後療`
  - `Mixed区分 = Mixed`
  - `case1要約 = case1:後療`
  - `case2要約 = case2:初検(抑制)`
  - `課金理由要約 = 初検抑制かつ再検対象なし`

### TC03（参考: 単独後療）

- case1 = 後療のみ、case2 なし
- 期待値:
  - `算定区分 = 後療`
  - `Mixed区分 = 通常`
  - `case1要約 = case1:後療`
  - `case2要約 = case2:なし`
  - `課金理由要約 = 後療のみ`

## 7. 実装方針

### 7-1. 実装の中心

今回の主実装箇所は **Ver3_amounts.js** の `calcHeaderAmountsByVisitKey_V3_` とする。
理由:
- `initFee / reFee / detailSum` と整合しやすい
- `hasBillableInitial / hasReexam / hasKoryo / reasons` が同関数内にすでに存在する
- 金額確定後に判断した方が安全

### 7-2. 補助的に触る箇所

- **Ver3_core.js** の `HEADER_COLS` に5キー追加
- **Ver3_core.js** の `appendHeaderRow_V3_` に5行 `setByName_` 追加
- **Ver3_core.js** の `saveVisit_V3` の `appendHeaderRow_V3_` 呼び出し部に5プロパティ追加

### 7-3. 書込み方法

- 列名ベースで書き込む（`setByName_` 既存方式）
- 既存列の位置変更は行わない

---

## 8. 今回やらないこと

- 既存 `区分` 列の名称変更（`表示区分` へのリネーム）
- 既存データの一括再計算
- 帳票・申請書側への新列反映
- `exportHeaderFromCases_V3` への新5列反映（空欄扱いで可）
- 大規模リファクタ・dead code 削除

---

## 9. 受け入れ条件

次が確認できれば受け入れとする。

1. M01: `算定区分=再検 / Mixed区分=Mixed / case2要約=case2:初検(抑制)` が入る
2. M02: `算定区分=初検 / Mixed区分=Mixed / case1要約=case1:再検 / case2要約=case2:初検` が入る
3. M03: `算定区分=後療 / 課金理由要約=初検抑制かつ再検対象なし` が入る
4. TC03: `算定区分=後療 / Mixed区分=通常 / case2要約=case2:なし` が入る
5. 既存の `visitTotal / 窓口負担 / 保険請求額` は変わらない
6. 月次転記・transfer 処理に影響を出さない
