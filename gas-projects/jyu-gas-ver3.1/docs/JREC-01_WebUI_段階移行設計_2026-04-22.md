# JREC-01 Web UI 段階移行設計

作成日: 2026-04-22  
対象ブランチ: `feature/auto-dev-phase3-loop`

---

## 設計方針

- **全面移行ではなく段階移行** — 制度ロジック・算定ロジック・帳票出力・シート保存は資産として維持
- **UI を先に Web 化** — GAS 側の関数群はそのまま使い、フロントだけを差し替える
- **構成は Web UI + GAS + Sheets のまま継続** — Next.js / React 等への全面移植はしない
- **壊れたら戻せる** — 既存のシート UI は削除せず、Web UI と並行運用できる期間を設ける

## 2. 各UIの評価

| UI | Web化効果 | リスク | 制度ロジック影響 | 難易度 | 向け |
|---|---|---|---|---|---|
| **患者検索・選択** | ★★★ スマホ・タブレット対応・操作高速化 | 低（検索のみ。書き込みなし） | なし | 低 | 受付 |
| **来院入力フォーム** | ★★★ 入力ミス削減・バリデーション強化 | 高（保存関数と密結合。区分判定UI連動） | あり（区分・kubun 判定をUI依存） | 高 | 施術者 |
| **自費明細入力** | ★★ 既にダイアログ実装済み | 低（独立した HTML dialog として分離済み） | なし | 中 | 受付 |
| **帳票出力起点** | ★ 月・患者選択をWeb化 | 中（出力先はGASドキュメント） | なし（出力トリガーのみ） | 中 | 施術者 |
| **保険者情報転記** | ★ マイナ確認後の操作を簡略化 | 低（1機能のみ） | なし（転記のみ） | 低 | 事務 |
| **患者マスタ登録** | ★★ 初回登録を専用フォームで | 中（マスタ書き込み） | なし | 中 | 事務 |

### 評価詳細

#### 患者検索・選択（★★★ 最優先）
- 現状: Google Sheets のプルダウン検索 → B2 に表示文字列が入り C2 数式で ID 抽出
- 問題: スマホ・タブレット非対応。プルダウン件数が増えると重い。IME 依存。
- Web化: 名前・フリガナ・患者ID で絞り込み → 選択 → シート B2/C2 に反映 or 直接 visitKey 発行
- 制度影響: ゼロ。患者IDを特定するだけ。
- 依存関数: `PatientPicker_findDisplayCol_` / `PatientPicker_refreshDisplayCol_` — 読み取り専用でよい

#### 来院入力フォーム（★★★ 効果大・難易度高）
- 現状: セル直接入力 + `readRowNewUI_` で A1 定数読み取り → `saveVisit_V3` で保存
- 問題: セルが見づらい。入力行が離れている（ケース1=行12-13, ケース2=行36-37）。
- 問題: Web化すると `readRowNewUI_` が UI シートを前提にしているため影響大。
- Web化コスト: `saveVisit_V3` に Web からの JSON 入力経路を追加するか、別関数が必要。
- 制度影響: あり。`autofillFromPreviousVisit_V3` が UI シートに書き込む前提のため変更が必要。

#### 自費明細入力（★★ 既に半 Web 化済み）
- 現状: `selfPayDialog.html` という HtmlService 実装が存在し `google.script.run` で保存。
- Web 化の方向: 独立した URL に移設するだけ。`saveSelfPayDetailsFromDialog_V3` を Web から呼べばよい。

## 4. 最初に着手する1画面: 患者検索・患者選択 Web UI

### 選定理由

1. **制度ロジックに触れない** — 患者IDを「探す」だけ。保険算定・来院保存に影響ゼロ。
2. **失敗しても即戻せる** — 既存のプルダウン検索は削除しない。並行運用可能。
3. **受付が毎日使う** — 患者選択を速く・確実にできれば院全体の業務が軽くなる。
4. **スマホ対応の恩恵が大きい** — 問診・受付でタブレット活用が見込める。
5. **既存関数を読み取りで使える** — `PatientPicker_refreshDisplayCol_` が生成した「検索用」列をそのまま参照可能。

## 5. 患者検索・患者選択 Web UI 設計

### 5-1 画面の目的

患者名・フリガナ・患者ID のいずれかをキーワードで部分一致検索し、
対象患者を選択して来院日とともに確定する。

確定後、来院ヘッダの visitKey 発行に必要な `{ patientId, treatDate }` を返す。
（将来的には `saveVisit_V3` 相当の処理への入口にもなる）

### 5-2 利用者

- **受付スタッフ**: 来院時の患者特定（メイン用途）
- **施術者**: 患者確認・当日履歴参照

### 5-3 最小機能（Phase 1 スコープ）

| 機能 | 内容 |
|---|---|
| キーワード検索 | 患者名・フリガナ・患者ID で前方一致/部分一致 |
| 候補表示 | 患者ID / 氏名 / フリガナ / 生年月日 を一覧表示 |
| 選択 | タップ/クリックで患者を選択 |
| 来院日入力 | デフォルト = 今日。手動変更可。 |
| 確定 | 選択内容を GAS 関数に渡し、既存シート B2/C2/B4 に反映 |

**Phase 1 でやらないこと:**
- 来院ケース入力（保存処理）
- 自費入力
- 帳票出力

### 5-4 入力項目

| 項目 | 型 | 必須 | 補足 |
|---|---|---|---|
| 検索キーワード | テキスト | 任意 | 空欄で全件表示（件数が多い場合は先頭N件） |
| 来院日 | 日付 | 必須 | デフォルト: 今日 |

### 5-5 表示項目（検索結果リスト）

| 項目 | ソース |
|---|---|
| 患者ID | 患者マスタ「患者ID」列 |
| 氏名 | 患者マスタ「氏名」列 |
| フリガナ | 患者マスタ「フリガナ」列 |
| 生年月日 | 患者マスタ「生年月日」列 |
| 最終来院日 | 来院ヘッダ「最終来院日」列（任意。Phase 1 では省略可） |

### 5-6 GAS 側で使う既存関数 / 新設が必要な関数

#### 既存関数（再利用）

| 関数 | 場所 | 用途 |
|---|---|---|
| `buildHeaderColMap_` | `Ver3_core.js` | 患者マスタのヘッダマップ取得 |
| `getMasterHeaderMap_` | `Ver3_core.js` | 同上（別名版） |
| `PatientPicker_refreshDisplayCol_` | `Ver3_patientPicker.js` | 検索用文字列の最新化（任意） |

#### 新設が必要な関数（推定）

| 関数（案） | 役割 |
|---|---|
| `searchPatients_V3(keyword)` | キーワードで患者マスタを検索し候補リストを返す |
| `setPatientAndDate_V3(patientId, treatDate)` | 患者IDと来院日をシート B2/C2/B4 に書き込む |

`searchPatients_V3` の実装イメージ:
```javascript
function searchPatients_V3(keyword) {
  var ss = SpreadsheetApp.getActive();
  var masterSh = ss.getSheetByName(SHEETS.master);
  var map = getMasterHeaderMap_(masterSh);
  // 患者マスタ全行を読んでキーワード部分一致フィルタ
  // 返却: [{patientId, name, furi, birthday}, ...]
}
```

`setPatientAndDate_V3` の実装イメージ:
```javascript
function setPatientAndDate_V3(patientId, treatDateStr) {
  // B2 に "P001｜氏名｜フリガナ｜生年月日" 形式を書き込む
  // C2 は数式のままで自動抽出
  // B4 に日付を書き込む
  // autofillFromPreviousVisit_V3 を実行して前回データを引継ぎ

```

### 5-7 Sheets 側の保存先 / 参照先

| 操作 | 対象シート | 列 |
|---|---|---|
| 検索（読み取り） | 患者マスタ | 患者ID / 氏名 / フリガナ / 生年月日 |
| 選択確定（書き込み） | 患者画面 | B2（表示文字列） / B4（来院日） |
| C2 / B3 | 患者画面 | 数式のまま（書き込み不要） |

### 5-8 既存運用を壊さない移行方法

| ルール | 内容 |
|---|---|
| 並行運用 | 既存のシート B2 プルダウンは削除しない。Web UI とどちらでも選択できる期間を設ける |
| 書き込み先は同じ | Web UI も既存シートと同じ B2/B4 に書き込む。以降の `saveVisit_V3` は変更不要 |
| 検索用列は維持 | `PatientPicker_refreshDisplayCol_` が作る「検索用」列はそのまま残す（既存プルダウン用） |
| ロールバック | Web UI が動かない場合でも既存の GAS メニュー + シート操作で業務継続可能 |

## 6. 段階移行方針

### Phase 1: Web UI 試作（患者検索）

**目標:** 患者検索だけが動く最小 Web ページを作り、内部で使えるか検証する

| 項目 | 内容 |
|---|---|
| 実装内容 | `searchPatients_V3` + 検索 UI（HTML/CSS/JS） |
| デプロイ方法 | `HtmlService.createHtmlOutput` → GAS Web App として公開（doGet） |
| 保存処理 | Phase 1 では「シートに書き込まず候補を表示するだけ」でも可 |
| 確認基準 | キーワード検索で正しい候補が返ること |
| 並行運用 | 既存シートUI は手を付けない |
| 撤退基準 | 動作しない / 認証が複雑すぎる → 既存シート継続 |

### Phase 2: 一部運用（患者選択 + 来院日確定）

**目標:** 受付スタッフが実際の来院受付で使い始める

| 項目 | 内容 |
|---|---|
| 実装内容 | `setPatientAndDate_V3` 追加 + `autofillFromPreviousVisit_V3` 連動 |
| 保存処理 | 選択確定で B2/B4 を書き込み。それ以降は既存フロー |
| 確認基準 | Web UI で選択 → シート B2/C2/B4 に値が入り → 来院入力・保存が従来通り動くこと |
| 並行運用 | シートのプルダウンも残す（施術者は引き続き使える） |
| 撤退基準 | B2/B4 への書き込みでバグが出る → Web UI の選択確定のみ中断 |

### Phase 3: 本運用判断（来院入力フォーム検討）

**目標:** 来院入力フォームの Web 化可否を判断する

| 項目 | 内容 |
|---|---|
| 前提 | Phase 2 の患者選択が安定稼働していること |
| 評価内容 | `readRowNewUI_` 依存の解体コスト / `autofillFromPreviousVisit_V3` の UI 書き込み経路の変更コスト |
| 判断基準 | `saveVisit_V3` に JSON 入力経路を追加するコストが許容範囲か |
| 移行方法 | `saveVisit_V3` をラップする `saveVisitFromWeb_V3(json)` を新設。既存関数は変更しない |
| 撤退基準 | 算定ロジックの再現性が担保できない → シート入力を本線維持 |

## 7. 技術スタック（候補）

| レイヤ | 案 | 理由 |
|---|---|---|
| フロントエンド | Vanilla HTML + CSS + JavaScript | GAS の `HtmlService` と直接統合可能。依存ゼロ |
| または | React + Vite（外部ホスト） | より高度な UI。ただし CORS / OAuth 設定が必要 |
| バックエンド API | GAS Web App（`doGet` / `doPost`） | 既存ロジックをそのまま公開 |
| 認証 | GAS の「ユーザーアクセス制限」（Googleログイン） | 追加実装不要。スプレッドシートと同じアカウント |
| デプロイ | `clasp deploy` | 既存ワークフローに統合 |

**Phase 1 推奨:** Vanilla HTML + GAS Web App。依存ライブラリなし。HtmlService に直接埋め込み可能。外部ホストが不要。

## 8. リスクと対策

| リスク | 対策 |
|---|---|
| GAS Web App の応答が遅い（初回 cold start）| `UrlFetchApp` ではなく直接呼び出し（HtmlService 内の `google.script.run`）でレイテンシを最小化 |
| 患者マスタが大きくなると `searchPatients_V3` が遅い | ヘッダ1行+データ全行を1回読んでフィルタ（1回のシート読み取りで完結） |
| `setPatientAndDate_V3` が B2 数式を破壊する | B2 はバリデーション文字列として書き込む（数式ではないため上書き安全）。C2・B3 の数式は触れない |
| 認証・権限の複雑化 | GAS Web App を「自分のみ」または「組織内全員」公開に限定。外部公開しない |
| 既存プルダウンとの競合 | B2 の値フォーマット（`"P001｜氏名｜..."` 形式）をWeb UIでも同一にすることで `readRowNewUI_` / C2 数式が壊れない |

## 次に実装する場合のプロンプト案

```
JREC-01 で、患者検索 Web UI の Phase 1 を実装してください。

【目標】
GAS Web App (doGet) として公開できる患者検索ページを作る。
検索結果を表示するだけ。シートへの書き込みは Phase 2 で行う。

【実装内容】
1. Ver3_core.js に searchPatients_V3(keyword) を追加する
   - 患者マスタ（SHEETS.master）を全行読む
   - keyword を患者ID / 氏名 / フリガナ で部分一致フィルタ
   - 返却: [{patientId, name, furi, birthday}, ...] (最大50件)
   - keyword が空文字の場合は全件返す（最大100件）

2. 患者検索 HTML ファイル（patientSearch.html）を新規作成する
   - 検索テキストボックス（リアルタイム or ボタン押下検索）
   - 候補一覧テーブル（患者ID / 氏名 / フリガナ / 生年月日）
   - google.script.run.searchPatients_V3(keyword) で非同期呼び出し
   - スマホで使えるレスポンシブレイアウト（Bootstrap不要 / CSS変数で実装）

3. GAS Web App エントリポイント doGet を Ver3_core.js に追加する
   - `HtmlService.createHtmlOutputFromFile("patientSearch")` を返す
   - タイトル: "患者検索 — JREC-01"

【制約】
- Ver3_patientPicker.js・Ver3_core.js の既存関数を変更しない
- シートへの書き込みはしない（Phase 1 スコープ外）
- clasp push まで実施して完了とする
- 実装後、doGet の URL を報告する手順を示すこと

【参照】
docs/JREC-01_WebUI_段階移行設計_2026-04-22.md §5 患者検索・患者選択 Web UI 設計
```
