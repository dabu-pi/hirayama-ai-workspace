## 4. 差し込みアプローチ比較

### アプローチ A: replaceText() プレースホルダー方式

```
body.replaceText('{{patient_name}}', '田中 花子')
```

| 項目 | 評価 |
|---|---|
| 実装容易度 | ○ シンプルで読みやすい |
| 前提条件 | テンプレートに `{{KEY}}` マーカーを挿入する必要あり |
| 固定欄への適用 | ◎ 最適（患者名・保険番号・負傷名等） |
| 繰り返し行への適用 | △ 行数が可変なので難しい |
| レイアウト影響 | ○ テキスト置換のみなのでレイアウト崩れリスク低 |

### アプローチ B: テーブルセル直接書き込み方式

```
tables[i].getRow(r).getCell(c).setText('値')
```

| 項目 | 評価 |
|---|---|
| 実装容易度 | △ テーブル Index / 行 / セルを事前に特定する必要あり |
| 前提条件 | テンプレートの構造が変わると壊れる |
| 固定欄への適用 | ○ 可能だが Index 管理が必要 |
| 繰り返し行への適用 | ◎ 最適（日別明細行への書き込みに必須） |
| レイアウト影響 | ○ setText() はフォント・サイズを保持 |

### 採用方針（暫定）

| 対象 | 採用アプローチ |
|---|---|
| 表面 固定欄（患者名・保険番号・負傷名等） | **A: replaceText** （テンプレートにプレースホルダーを追加後） |
| 裏面 日別明細行 | **B: テーブルセル直接書き込み** |
| 裏面 ①②③ 月次集計行 | **B: テーブルセル直接書き込み** |

## 8. テーブル検索ロジック（ハードコードIndex を避ける設計）

sandbox 実行で判明した重要事実:
> **表面テーブルは複数存在し、Word → GDocs 変換で Index が変動しやすい。**
> → ハードコード Index は採用せず、**セル内容で対象テーブルを特定するロジック**を本実装に採用する。

### 表面: replaceText 方式（Index 不要）

```javascript
// プレースホルダーをテキスト置換するだけ — テーブル位置を知る必要なし
body.replaceText('{{患者氏名}}', '田中 花子');
body.replaceText('{{被保険者記号}}', 'テスト-1234');
```

→ テンプレートの各セルに `{{KEY}}` を手入力しておくことで全欄を網羅できる。

### 裏面: content-based テーブル検索

```javascript
function srFindUrameTable_(body) {
  var tables = body.getTables();
  for (var i = 0; i < tables.length; i++) {
    // ヘッダ行に「後療料」を含む = 裏面日別テーブル
    if (tables[i].getNumRows() > 0 &&
        tables[i].getRow(0).getText().indexOf('後療料') >= 0) {
      return tables[i];
    }
  }
  throw new Error('裏面テーブルが見つかりません');
}
```

### 裏面: ①②③ 集計行の特定

```javascript
function srFindSummaryRows_(urameTable) {
  var summaryRows = { 1: -1, 2: -1, 3: -1 };
  for (var r = 0; r < urameTable.getNumRows(); r++) {
    var cellText = urameTable.getCell(r, 0).getText();
    if (cellText.indexOf('①') >= 0) summaryRows[1] = r;
    if (cellText.indexOf('②') >= 0) summaryRows[2] = r;
    if (cellText.indexOf('③') >= 0) summaryRows[3] = r;
  }
  return summaryRows;  // { 1: row_idx, 2: row_idx, 3: row_idx }
}
```
