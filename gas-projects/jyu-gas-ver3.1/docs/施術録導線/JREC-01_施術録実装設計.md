## 1. 概要

JREC の日次入力データを集約し、柔道整復師療養費支給申請書に添付する
「施術録」を Google Docs テンプレートから月次で自動生成・PDF 化する機能。

| 項目 | 内容 |
|---|---|
| 入力 | 患者ID（`patientId`） + 対象年月（`YYYY-MM`） |
| 出力 | Google Docs コピー + PDF（Drive 保存） |
| トリガー | スプレッドシート カスタムメニュー「施術録を出力」→ ダイアログで月・患者を指定 |
| 実装場所 | `gas-projects/jyu-gas-ver3.1/Ver3_shuRecorder.js`（新規ファイル）|

> **T-SR-10v4 修正（2026-04-02）:** 実テンプレ export を踏まえ、裏面は `月/日` 左端1セル、`初検料等` grouped cell、`後療料`、`冷温`、`電療料`、`一部負担金`、`所見` を優先して再マッピングする。申請書側の算定判定ロジックには触れず、施術録側の表示 / 転記処理のみで吸収する。

> **基本料表示方針（2026-04-03 v11 整理）:**
> `initialAmount`（初検料等欄）= 来院ヘッダの `initFee + supportFee + reFee`（申請書算定済み値）。
> `shiryoOut`（施療料欄）= 施術明細のうち `kubun=初検` の `baseOut`。

> 実テンプレートの `施療料はここ` / `後療料はここ` 目印を別々に採用し、列混入を防ぐ。

> **裏面 所見列の主転記列: `CASE_COLS.shoken`（= "所見"）を優先。空の場合は `keikaNow`（= "経過_今回"）を補完。**

## 4. 初版スコープ確定

### 自動転記（Phase 1 v1）

| 対象 | 内容 |
|---|---|
| 表面 患者・保険情報 | 患者マスタから `replaceText` で全欄 |
| 表面 負傷名一覧 | 最大2件（部位1・部位2） |
| 表面 負傷記録 | 初検情報履歴から日時・場所・状況・初検所見 |
| 裏面 日別明細 | 後療料・冷温・電療・一部負担金・所見（対象月全来院日） |
| 裏面 ①月次集計 | 回数・合計金額・一部負担額・期間を集計して記入 |

### 手入力で残す（Phase 1）

| 対象 | 理由 |
|---|---|
| 一部負担割合チェックマーク | チェックボックス操作が GAS では困難 |
| 裏面 初検料・整復料列 | 初版は実装コスト大・頻度を見て判断 |
| 裏面 明細書発行体制加算 | 月次固定値 → 手入力後で確認 |
| 同意医師氏名・同意年月日 | 署名欄のため手書きが主 |
| 労務不能欄 | まれなケース |
| 受傷部位図解 | 手書きスタンプ等で対応 |
| 初検時相談支援 ①②③ チェック | チェックボックス操作が困難 |
| ②③月次集計行 | 複数月対応は Phase 2 以降 |

### Phase 1 スコープ外

| 対象 | 理由 |
|---|---|
| 市町村番号・受給者番号・公費番号 | 公費対応は別フェーズ |
| 自費混在ケース | データ構造が別ルート |
| 3部位以上の負傷名 | 実運用上まれなケース |
| 保険者所在地 | 患者マスタに列がない場合 |

## 5. 出力ファイル管理

### 命名規則

```
Google Docs: 施術録_{氏名}_{YYYY}年{MM}月
PDF:         施術録_{氏名}_{YYYY}年{MM}月.pdf
```

例: `施術録_田中花子_2026年04月` / `施術録_田中花子_2026年04月.pdf`

### 出力先フォルダ構造

```
平山接骨院再開プロジェクト2026（ID: 1XMx2e1ufCRqp7bhpfRRjvPDyXCESL83V）
└── 施術録/
    ├── 2026-04/
    │   ├── 施術録_田中花子_2026年04月       ← Google Docs（修正可能版）
    │   └── 施術録_田中花子_2026年04月.pdf  ← 確定提出版
    └── 2026-05/
        └── ...
```

> `施術録/YYYY-MM/` フォルダがなければ GAS が自動作成する。

| 状況 | 挙動 |

| はい | 既存ファイルをトラッシュ → 再生成 |

## 11. T-SR-10 実機NG記録（2026-04-02）

T-SR-10（clasp push + 手動テスト）で以下の5件の不具合を確認。同日 T-SR-10v2 として修正を実施。

| # | 不具合内容 | 原因 | 修正箇所 |
|---|---|---|---|
| NG-01 | **裏面に自費のみ日が混入**（例: 3/2, 3/10） | `srGetVisitRows_` が会計区分チェックなしで全来院日を対象にしていた | `srGetVisitRows_`: `会計区分 = 自費のみ` の行をスキップ |
| NG-02 | **裏面の冷/温/電/一部負担金が不正** | `coldWarm`（col6）に冷+温を合算して書き込んでいた。GDocs テンプレートは冷・温を別列として持つ | `SR_URAME_COL`を分離: cold→col6, warm→col7, elec→col8, copay→col10, notes→col11 |
| NG-03 | **表面の施術回数が月全体件数**（自費込み） | `visitRows.length`（来院ヘッダ全件数）を施術回数として使用 | `V3TR_aggregateDetailMonthly_` の `visitDays` に変更 |
| NG-04 | **表面の日数が「受傷日〜月末」の日数差** | `srCalcNissuu_`（inj → monthEnd）で計算 | `V3TR_aggregateDetailMonthly_` の `visitDays` に変更（実来院日数） |
| NG-05 | **表面の負傷名が傷病のみ**（部位なし） | `srGetCaseData_` が `傷病_部位1`（d1）だけ取得し、`部位_部位1`（p1）を含めていなかった | `srGetCaseData_` で p1 + d1 を `部位 傷病` 形式で組み合わせ |
| NG-06 | **表面の case2 が月途中から存在する月に欠落** | `srGetCaseData_` が月内最初の行だけで `p2 / d2 / inj2 / start2` を固定し、後続行の case2 を補完していなかった | `srGetCaseData_` で空欄項目を月内後続行から backfill |
| NG-07 | **裏面で 505 が後療料欄に着地せず、1265 が不自然に見える** | `srGetVisitRows_` が `baseOut` を `kubun` 無視で合算し、`初検 760` まで `後療料` 側へ流していた | `srGetVisitRows_` で `kubun=初検` を `初検料等`、`kubun!=初検` を `後療料` として表示用分離 |

**修正ファイル:** `Ver3_shuRecorder.js`（全5件対応済み）

**次テスト時の確認ポイント:**
- `SR_URAME_COL` の cold/warm/elec/copay/notes 列インデックスが実テンプレートと一致するか確認
