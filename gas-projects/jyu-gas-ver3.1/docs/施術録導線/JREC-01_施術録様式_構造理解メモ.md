## 2. Word様式（施術録様式参考例.docx）の構造

### PDF様式との相違点・追加点

| 項目 | PDF様式 | Word様式 |
|---|---|---|
| 基本構造 | 同一 | 同一 |
| ページ構成 | 2ページ | 3ページ（表面・空白ページ・裏面） |
| 初検時相談支援欄 | 空欄 | **3項目の記載例あり** |
| 裏面列 | 明細書発行体制等加算が赤字 | 同様 |
| 編集性 | 不可（PDF固定） | **可（Word表セルが編集可能）** |

### Word様式の初検時相談支援記載例

Word様式には以下の記載例が印刷済み（ガイドとして）:
```
① 日常生活動作上での励行事項や禁止事項【入浴、歩行、安静期間、保護、運動制限等】
② 患部の状況や選択される施術方法などの詳細な説明【施術方針等】
③ 受療委任の取扱いについての説明
```

> 制度SPEC（§3-5）と整合している。自動転記時はこの形式に合わせて出力するのが自然。

### Word様式の編集性評価

| 観点 | 評価 | 備考 |
|---|---|---|
| テキスト入力 | ✅ 可能 | 各欄が表セル構造 |
| 転記・修正 | ✅ 容易 | セル単位で書き換え可 |
| 月次保存 | ✅ 実用的 | ファイル単位で月ごとに保存 |
| 日別行の追記 | ✅ 可能 | 行に直接入力 |
| PDF変換 | ✅ 可能 | 印刷→PDF保存またはエクスポート |
| GAS自動入力 | ⚠️ 要確認 | Docs API でセル操作が可能か要検証 |
| 骨格図（受傷部位） | ❌ 自動化困難 | 図解は手書きが現実的 |

## 3. 1患者×月出力の可否判断

### 判断結果: **現時点方針として「1患者×月」出力を採用する**

### 根拠

| 観点 | 内容 |
|---|---|
| 様式の月欄構造 | 裏面に①②③の月集計欄がある。「月ごとに集計する」設計思想が様式に内包されている |
| 実務の使いやすさ | 月ごとにファイルが分かれていると、特定月の確認・提出・Archive管理がしやすい |
| 3か月上限との整合 | 1枚で最大3か月分。「1患者×月」方式なら月集計欄①のみ使い、残り②③は未記入。上限内に常に収まる |
| 継続患者への対応 | 長期継続患者（3か月超）でも、月ごとにファイルが分かれているため遡り確認が容易 |

### 「1患者×月」出力の具体的運用イメージ

```
患者A 2026年4月分
└─ 施術録_患者A_2026-04.docx（または .pdf）
    ├─ 表面: 患者基本情報・負傷記録・初検情報（月によって変化なし）
    └─ 裏面: 4月分の来院日のみ記入 → 月集計①のみ使用

患者A 2026年5月分
└─ 施術録_患者A_2026-05.docx
    ├─ 表面: 同上（テンプレート流用）
    └─ 裏面: 5月分の来院日のみ記入 → 月集計①のみ使用
```

## 4. 内部管理単位 と 帳票出力単位 の分離整理

### 考え方の分離

| 概念 | 定義 | 目的 |
|---|---|---|
| **内部管理単位** | 1患者 × 1負傷系列（caseKey） | JREC内での算定・集計の整合性を保つ |
| **帳票出力単位** | 1患者 × 1月 | 実務上の施術録ファイル管理・提出・Archive |

### なぜ2つを分けるか

- **内部管理単位（caseKey）** はJRECの算定ロジック（30日ルール・月上限・多部位逓減）の前提。これを崩すと算定が壊れる
- **帳票出力単位（月）** は実務上の利便性。月ごとに施術録ファイルを出すほうが確認・保管・提出しやすい
- 同一患者に複数負傷がある場合、内部ではcaseKey別に管理しつつ、帳票は「月単位で1ファイルに合算」する形が自然

### 同月内複数負傷がある場合の帳票上の扱い

**パターン別の方針（現時点方針）:**

| パターン | 内部管理 | 帳票出力単位 |
|---|---|---|
| 単独負傷（通常） | caseKey 1件 | 1患者×月で1ファイル |
| 同月内に別負傷が発生（継続中） | caseKey 2件（制度SPECの Mixed [A]） | **同じ月ファイルに合算**（表面の負傷記録テーブルに2行） |
| 前負傷治癒後に新規負傷（同月） | caseKey 2件（制度SPECの [B]） | **同じ月ファイルに合算**（表面の負傷記録テーブルに2行） |
| 翌月以降も継続 | 同一caseKey継続 | 翌月分は新しい月ファイル（表面テンプレ流用） |

> 様式の表面「負傷記録テーブル（5行）」は、同一患者の複数負傷を同一施術録に記録するための構造。
> 月単位出力に複数負傷を含めることは、様式の設計思想と整合している。

## 6. 出力方式の比較と推奨

### 方式比較

| 方式 | 編集性 | 転記自動化 | 月次保存 | 監査対応 | 実務運用 | 総合 |
|---|---|---|---|---|---|---|
| **PDF直打ち** | ❌ 不可 | ❌ 不可 | △ Drive保存 | ✅ 確定感 | ❌ 修正不可 | **不採用** |
| **Word方式（docx）** | ✅ 可 | ✅ Docs API / テンプレ置換 | ✅ ファイル単位 | ✅ PDF変換で確定 | ✅ 実用的 | **最有力候補** |
| **Excel / Sheets** | ✅ 可 | ✅ GAS連携しやすい | ✅ ファイル単位 | △ 改変リスク | △ 縦書き・図解が不向き | **部分採用（裏面のみ検討）** |
| **Google Docs（docxをGDoc化）** | ✅ 可 | ✅ Docs API | ✅ Drive管理 | ✅ PDF変換で確定 | ✅ PC不問でアクセス可 | **Word方式と同等・検討対象** |

### 推奨: Word方式（または同等のGoogleドキュメント）

- **docxテンプレートを1枚用意**し、月ごとに複製→転記→確定保存 の流れが最も自然
- GASからの自動入力は「Google Drive API でdocxを複製 → Docs API でテキスト置換」が実現可能
- 最終確定後は PDF変換して Archive保存
