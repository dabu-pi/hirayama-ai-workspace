# JREC-01 申請書メニュー経路整理

- 日付: 2026-04-06
- 対象ブック: `【毎日記録】来店管理施術録ver3.1`
- 対象UI: 柔整ツール > 柔整管理

---

## 背景・問題の経緯

### 発生した問題

負傷名・負傷の原因の出力仕様を修正（9af6fc4, 7dadd97）したが、
現物の申請書テンプレートシートに変更が反映されなかった。

**原因:** 申請書の出力経路が2系統あり、テンプレートシートを直接更新する経路
（`V3TR_menuWriteApplication`）がメニューに未登録だった。

ユーザーが実行できるメニューはどちらもテンプレートシートを更新しない経路のみだった。

---

## 申請書系メニュー 3経路の役割

### 経路A: 申請書テンプレに反映（画面確認用）

| 項目 | 内容 |
|---|---|
| メニュー配置 | 柔整管理 > **申請書テンプレに反映（画面確認用）** |
| 関数名 | `V3TR_menuWriteApplication` |
| 追加日 | 2026-04-06（今回追加） |
| 出力先 | 申請書テンプレートシート（現物・直接書き込み） |
| 処理フロー | 患者ID + 対象月 入力 → 転記データ生成 → テンプレートシートに直接書き込み |
| 用途 | **現物確認・印刷前確認**。セル単位で正しく反映されているかを目視確認するための経路 |
| 注意 | テンプレートシートを上書きするため、再実行は冪等（同じデータを再書き込みするだけ） |

---

### 経路B: 申請書転記データを作成

| 項目 | 内容 |
|---|---|
| メニュー配置 | 柔整管理 > **申請書転記データを作成** |
| 関数名 | `V3TR_menuBuildTransferData` |
| 出力先 | 「申請書_転記データ」シートのみ（テンプレートシートは変更しない） |
| 処理フロー | 患者ID + 対象月 入力 → 「申請書_転記データ」シートを upsert |
| 用途 | 転記データの準備・確認。経路AとCの前段としても使える |

---

### 経路C: 申請書を生成して Drive に保存

| 項目 | 内容 |
|---|---|
| メニュー配置 | 管理者用 > **申請書を生成して Drive に保存** |
| 関数名 | `V3TR_menuGenerateApplication_B` |
| 出力先 | Google Drive（月別フォルダに xlsx 保存）・テンプレートシートは変更しない |
| 処理フロー | 対象月 入力 → NDJSON 生成 → Cloud Run POST → base64 xlsx → Drive 保存 |
| 用途 | **本番提出用 xlsx 生成**。印刷・提出する最終ファイルを Drive に保存する |
| 依存 | スクリプトプロパティ `APPGEN_ENDPOINT` / `APPGEN_SECRET` が必要 |

---

## 経路フロー図

```
[柔整管理]
  申請書転記データを作成       → 転記データシート更新のみ
  申請書テンプレに反映（確認用）→ テンプレートシート直接書き込み ← 今回追加
                                  ↑ 負傷名・原因の修正が反映される経路

[管理者用]
  申請書を生成してDriveに保存  → Cloud Run → xlsx → Drive保存
```

## 今後の検討事項

### 通常運用の一発生成化

現在は「転記データ作成」→「テンプレ確認」→「Drive保存生成」の3ステップ。
今後の検討として、月次処理を一発で完結させる案がある。

```
[案] 一発生成メニュー（将来）
  対象月入力
  → 全患者の転記データ生成
  → テンプレ確認スキップ or ログ出力
  → Cloud Run で全患者 xlsx 生成
  → Drive 保存
  → 完了レポート表示
```

**現状の判断:** 経路A（テンプレ確認）は確認目的として残す価値がある。
通常運用は経路C（Drive保存）を中心に寄せる案が現実的。
経路Aは確認・デバッグ用途として柔整管理に置く。

### テンプレートシートのクリア処理

現在 `V3TR_writeToApplication_` はセルへの上書きのみで、前回のデータをクリアしない。
別の患者・月に切り替える際に前のデータが残る可能性があるため、
必要に応じてテンプレクリア処理の追加を検討する。
