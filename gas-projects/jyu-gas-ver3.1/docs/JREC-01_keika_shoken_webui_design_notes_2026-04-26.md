# JREC-01 keikaNow / shoken / 経過履歴 / 施術録裏面 設計課題メモ

記録日: 2026-04-26  
ステータス: **設計課題記録済み / Phase A は現状維持 / Phase B で見直し予定**

---

## 0. 背景

Phase A 実運用テスト中に、以下の運用上の分かりにくさが確認された。

- A16:B20 に入力した「今回の経過」が D23:G28 の経過履歴に出る
- しかし同じ内容が施術録裏面に出ない可能性がある
- A23:B28 の「所見」が施術録裏面に優先して出るが、これは初検日の所見が繰り返される

---

## 1. 現在の構造と流れ先

```
[UI入力欄]            [保存先]           [出力先]
A16:B20（今回の経過）→ CASES.keikaNow → D23:G28 経過履歴（最新5件）
                                       → 施術録裏面 notes（shoken が空の場合のみ）

A23:B28（所見）       → CASES.shoken   → 施術録裏面 notes（優先）
                                       → D23:G28 には出ない

D23:G28（経過履歴）   → 表示専用        → 画面上の経過確認のみ
```

### 施術録裏面 notes の生成ロジック（Ver3_shuRecorder.js:548-551）

```javascript
var sho = String(crow[cci(cc.shoken)]   || '');
var kei = String(crow[cci(cc.keikaNow)] || '');
if (!notesMap[vk3] || sho) notesMap[vk3] = sho || kei;
// → shoken が非空であれば shoken を使い、keikaNow を無視する
```

## 2. 確認された問題点

### 問題1: 入力先と帳票出力先の不一致

| 入力欄 | D23:G28（画面）| 施術録裏面 notes |
|---|---|---|
| A16:B20（keikaNow） | ✅ 出る | ❌ 出ない（shoken があるため） |
| A23:B28（shoken） | ❌ 出ない | ✅ 出る（優先） |

→ 「画面で確認できる経過記録」と「施術録に印刷される経過記録」が別の欄に依存している。

### 問題2: 初検日の所見が毎日繰り返される

`autofillFromPreviousVisit_V3` は「エピソード開始日（初検日）の CASES.shoken」を A23:B28 に書き戻す。

→ 施術者が A23:B28 を更新しなければ、施術録裏面に初検日の所見が毎日同じテキストで出力される。

```
施術録裏面 イメージ（問題のある状態）:
  4/23（初検） | ... | 「腰部疼痛著明・ROM制限あり」（手入力）
  4/25         | ... | 「腰部疼痛著明・ROM制限あり」（autofill のまま）
  4/26         | ... | 「腰部疼痛著明・ROM制限あり」（autofill のまま）
```

### 問題3: keikaNow に毎日入力しても施術録裏面に出ない

```
施術者が A16:B20 に「4/25 改善傾向あり」と入力
  → CASES.keikaNow に保存 → D23:G28 の画面履歴に表示 → OK
  → 施術録裏面には出ない（shoken が非空なため）← 問題
```

## 3. 改善候補

### Phase A 中（コード変更なし）

**案D: ラベル変更のみ（最低リスク）**

- A16:B20 → 「今回の経過（経過履歴用）」
- A23:B28 → 「所見（施術録裏面）」
- D23:G28 → 「経過履歴」

目的: どこに入力した内容がどこに出るかを明示する。  
コード変更なし。スプレッドシートのラベルセルを手動変更するだけ。

### Phase B（コード変更可）

**案A: 施術録裏面 notes の出力方針を keikaNow 優先に変更（推奨）**

```javascript
// Ver3_shuRecorder.js:551 の 1 行を変更
// 変更前: if (!notesMap[vk3] || sho) notesMap[vk3] = sho || kei;
// 変更後: if (!notesMap[vk3] || kei) notesMap[vk3] = kei || sho;
```

- keikaNow = 毎日入力する今日の経過 → これが施術録裏面に出るべき
- shoken = 初検所見・補助 → keikaNow がない場合のフォールバック

効果: A16:B20 に入力した内容が施術録裏面にも出るようになる。

注意: 既存データを対象に施術録を出力すると、各日の notes 表示が変わる可能性がある。  
→ Phase B 実施前に既存データでの出力テストが必要。

**案B（補助）: autofill の shoken 引継ぎ挙動を見直す**

現在: 毎回「エピソード開始日（初検日）の shoken」を引き継ぐ  
→ 2日目以降は「常に初検日の所見がセットされた状態から始まる」

見直し案:
- 2日目以降は shoken を引き継がず空欄で始める
- 施術者が今日の所見を入力したいときだけ A23:B28 を使う
- または shoken を廃止して keikaNow に一本化する（大きな変更）

## 4. Web UI 移行時の入力欄整理方針（将来候補）

Web UI 移行時には、入力欄を以下のように整理する案を検討する。

### 入力欄の整理候補

| 欄 | 内容 | 保存先 | 出力先 |
|---|---|---|---|
| **初検時情報** | 負傷日時・場所・状況・初検時所見・相談支援内容 | 初検情報履歴シート | 申請書・施術録 **表面** |
| **本日の経過** | 患者の訴え・前回からの変化・痛み・可動域 | CASES.keikaNow | 施術録 **裏面**・経過履歴表示 |
| **本日の所見** | 施術者の客観所見・検査所見・触診・圧痛 | CASES.shoken | 施術録 **裏面**（本日の経過と分けて表示または同欄） |
| **経過履歴** | 過去の「本日の経過」を表示 | 保存なし（参照のみ） | 画面表示のみ |

### 設計方針

- **「画面で見ている経過」と「施術録裏面に出る記録」を一致させる**
- 初検時情報 → 申請書・施術録表面（現行と同じ役割・出力先）
- 本日の経過 → 施術録裏面（keikaNow 優先の場合の実装に対応）
- 本日の所見 → 施術録裏面の補足欄または別列（役割分担の場合）
- 経過履歴 → 参照専用。入力欄ではない
