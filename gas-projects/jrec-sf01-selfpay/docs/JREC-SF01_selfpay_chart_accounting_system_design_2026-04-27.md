# JREC-SF01 自費カルテ・会計システム 設計ドキュメント

**作成日:** 2026-04-27
**バージョン:** 0.1（初期設計）
**ステータス:** 設計段階（コード実装前）

---

## 1. このシステムの目的

JREC-SF01 は、接骨院・整骨院における**自費施術専用**のカルテ・会計・売上管理システムである。

| 目的 | 内容 |
|---|---|
| 自費カルテ管理 | 自費施術の評価・所見・施術内容を記録する |
| 自費会計 | 施術メニュー・金額・支払方法を管理する |
| 領収書発行 | 番号管理付きの領収書を発行・再発行する |
| 売上集計 | 日次・月次の自費売上を集計する |
| 保険請求との完全分離 | 保険JRECとは別スプレッドシート・別GASプロジェクトで管理する |
| 将来の外販対応 | 保険JRECを含まないパッケージとして外販可能な設計にする |

---

## 2. なぜJREC本体と分けるのか

### 保険請求ロジックを守るため

保険JREC本体（`jyu-gas-ver3.1`）は柔整療養費制度に準拠した複雑なロジック（30日ルール・逓減・長期減額等）を持つ。自費ロジックが混入すると制度準拠性・監査耐性が低下するリスクがある。

### clasp push 対象を分けるため

GASプロジェクトを分けることで、自費側のデプロイが保険JREC側に誤って影響することを防ぐ。

### スプレッドシート構造を分けるため

保険JRECのシート構造は制度要件に縛られている。自費専用シートを分けることで、自費特有の列・シートを自由に追加できる。

### 自費特有の機能を自由に追加するため

予約管理・施術プロトコル・機器記録・説明トークなど、自費ならではの機能を保険JRECに影響なく追加できる。

### 将来Webアプリ化しやすくするため

自費システムを独立させることで、Next.js / Supabase / Vercel 等によるWebアプリ化の際に保険JRECを巻き込まずに移行できる。

### 外販時に保険JRECと切り離せるようにするため

他院への外販時、保険請求機能は院ごとにカスタマイズが必要であるため切り離して提供できる構成にする。

## 3. JREC本体との関係

### JREC本体（`jyu-gas-ver3.1`）が担うこと

```text
保険施術の来院記録
療養費支給申請書の生成
施術録の管理
制度準拠ロジック（30日ルール・逓減・長期減額等）
監査耐性を持つ記録管理
```

### JREC-SF01（本プロジェクト）が担うこと

```text
自費患者管理
自費カルテ（評価・所見・施術内容・機器使用）
自費メニュー・料金管理
自費会計（明細・合計・支払方法・未収管理）
領収書（番号管理・再発行）
自費売上集計（日次・月次）
患者別自費履歴
```

### 連携設計方針

- 患者IDは将来的にJREC本体・柔整ストラテジーと連携可能な形式で設計する（例: `P0001`）
- 保険患者と自費患者が同一人物の場合、`jrecPatientId` カラムで紐づける設計とする
- 当面は独立運用。連携は将来フェーズで実装する

## 4. 柔整ストラテジー（`hirayama-jyusei-strategy`）との関係

`hirayama-jyusei-strategy` は JREC-SF01 の設計元である。

| 柔整ストラテジーが持つもの | JREC-SF01 での活用 |
|---|---|
| 価格設計（単価・コース料金） | `MenuMaster` シートの初期データ |
| 自費メニュー定義 | メニューコード・カテゴリ設計 |
| 施術プロトコル | `SelfPayChart` の施術内容・評価項目 |
| 説明トーク | カルテの説明内容欄の構成参考 |
| KPI設計 | `DailySales` の集計指標 |
| 経営戦略（慢性疼痛強化） | Phase 設計の優先順位 |

## 7. ID設計案

| ID種別 | フォーマット | 例 |
|---|---|---|
| patientId | `P` + 4桁連番 | `P0001` |
| selfPayVisitKey | `SPV_YYYYMMDD_patientId_3桁連番` | `SPV_20260427_P0001_001` |
| chartId | `SPC_YYYYMMDD_patientId_3桁連番` | `SPC_20260427_P0001_001` |
| itemId | `SPI_YYYYMMDD_selfPayVisitKey_3桁連番` | `SPI_20260427_P0001_001_001` |
| paymentId | `SPP_YYYYMMDD_selfPayVisitKey` | `SPP_20260427_P0001_001` |
| receiptId | `SPR_YYYYMMDD_通し連番` | `SPR_20260427_001` |

**ID設計の方針:**

- 来院日が含まれるため、IDを見るだけでいつの記録か判断できる
- JREC本体の caseKey / visitKey と形式を区別するため `SP` プレフィックスを使う
- `jrecPatientId` は任意項目とし、同一患者を保険JRECと紐づける際に使う

## 8. 会計設計

### 基本方針

| 方針 | 内容 |
|---|---|
| 税別管理 | データは常に税別で保持する。税込は計算で導出する |
| 税込表示 | 患者向けの表示・領収書は税込で表示する |
| 支払方法 | 現金・カード・電子マネー・未収を記録する |
| 未収管理 | 未収発生・回収を `Payments` シートで追跡する |
| 領収書番号 | 年間通し番号で管理。再発行時は `再発行回数` を加算する |
| selfPayVisitKey による紐づけ | カルテ・明細・支払・領収書はすべて `selfPayVisitKey` で紐づく |

### 税計算方針

```text
小計（税別）= 単価 × 数量
消費税額 = 小計 × 税率（Settings シートから取得）
合計（税込）= 小計 + 消費税額
```

- 端数処理: 切り捨て（Settings で変更可）
- 軽減税率対象メニューは `MenuMaster` の `税区分` で管理する

## 9. 将来Webアプリ化を見据えた注意点

| 注意点 | 理由 |
|---|---|
| 1行1レコード形式を守る | スプレッドシートをDBとして扱う前提。帳票構造にしない |
| セル結合を使わない | データ読み込み・エクスポート時にセル結合は障害になる |
| IDで紐づける | 患者・来院・カルテ・会計・領収書を外部キーで関連付ける |
| テーブル間の正規化 | 同じデータを複数シートに重複して持たない |
| 将来DB移行しやすい構成 | スプレッドシートのシート = DBのテーブルとして設計する |
| GAS WebアプリのUI分離 | データ操作はGASのサーバーサイド関数に集中させ、UIはHTML側で分離する |
| Supabase移行前提の命名 | カラム名は英語スネークケースを原則とする（日本語は表示ラベルとして別管理） |
