# JREC-SF01 会計方針 v1

**作成日:** 2026-04-28
**バージョン:** v1.0
**ステータス:** 確定

---

## 1. 基本方針：来院・カルテ入力と会計入力は分離する

### 目的の違い

| 入力種別 | 目的 | 保存先 |
|---|---|---|
| 来院・カルテ入力 | **施術事実を記録する** | SelfPayVisits / SelfPayChart |
| 会計入力 | **請求・領収書用の会計項目を選択する** | SelfPayItems / Payments |

### カルテで記録するもの

- 来院日・来院区分・担当者
- 主訴（患者の訴え）
- 評価（ROMテスト・筋力・姿勢など）
- 所見（触診・圧痛部位など）
- 施術内容（干渉波・手技など）
- 使用機器
- 生活指導
- 次回予定
- VAS（痛みスコア）

### 会計で選択するもの

- MenuMaster に登録されたメニューコード・名称・価格
- 数量
- 税区分（課税 / 非課税）
- 支払方法（現金 / カード / 電子マネー / 未収）
- 入金状態（入金済 / 未収 / 一部入金）

## 2. カルテ内容を会計明細へ自動変換しない理由

### 2-1. 記録の性質が異なる

カルテの「施術内容」フィールドは施術者が自由記述で書く施術記録であり、
MenuMaster の「メニュー名」とは1対1で対応しない。

例:
```
カルテ施術内容: 「干渉波10分・ハイボルテージ5分・手技（軟膏塗布込み）」
→ これを会計明細に自動変換すると...
  - 干渉波 (SELFPAY_IFC10) ¥1,000
  - ハイボルテージ (SELFPAY_HIGHVOLTAGE) ¥500
  - 手技 (SELFPAY_MANUAL3) ¥500
  の3行になるが、実際には継続標準施術パッケージ ¥3,500 1本で請求することが多い
```

### 2-2. 事故リスク

完全自動連動させると以下のリスクがある：

| リスク | 内容 |
|---|---|
| 単価誤り | MenuMaster の単価が改定されても、過去のカルテ記述が引っ張ってしまう可能性 |
| 税区分誤り | 施術パッケージと個別パーツで税区分が異なる場合、自動変換が誤ることがある |
| 領収書表示の誤り | 施術記録の表現と領収書の項目名が一致しないと患者・院長が混乱する |
| メニュー改定時の事故 | メニュー統廃合・価格変更時に過去のカルテ文字列が新メニューに誤マッチする |

### 2-3. 外販モデルでの要件

外販（他院への提供）を前提とすると、以下の説明責任が必要：

- **算定事実**（どの施術を何分行ったか）= SelfPayChart に記録
- **請求事実**（何を何円で請求したか）= SelfPayItems / Payments に記録

両者を分けて説明できる構造は、保険外施術における透明性・監査対応の基礎となる。

## 3. 将来的な自動候補セットの検討

### 検討は認める、実装は慎重に

将来的に、以下の情報を元に **会計メニューの初期候補** を自動セットすることは検討対象：

- 来院区分（初診 → 初回標準施術 SELFPAY_INITIAL_FULL を候補）
- VAS スコア
- 前回の会計実績

ただし：

| 条件 | 内容 |
|---|---|
| 初期候補の提示まで | 人が確認・変更できる状態で表示する |
| 自動保存は禁止 | 人がフォームを確認・送信するまで保存しない |
| 候補の上書き可能 | 提示された候補を削除・変更できる UI を必ず用意する |

## 4. 会計入力フローの確定仕様

```
①来院・カルテ入力（SelfPayVisits + SelfPayChart に保存）
                    ↓
②患者詳細で「会計入力」ボタンをクリック（Phase 4 Step 5 で実装）
                    ↓
③billing-form.html で MenuMaster からメニューを選択して入力・確認
                    ↓
④「保存して領収書へ」ボタンを押す（人が確認してから実行）
                    ↓
⑤GAS savePaymentWithItems():
    SelfPayItems に明細を保存
    Payments に入金情報を保存
    SelfPayVisits.会計状態 を更新
                    ↓
⑥receipt.html に遷移 → 「領収書を発行する」ボタンを押す
                    ↓
⑦GAS issueReceipt():
    Receipts に保存・receiptNo 採番
                    ↓
⑧領収書プレビュー表示 → 印刷ボタン
                    ↓
⑨患者詳細へ戻る
```

## 5. 今回実装しないこと（明示的に除外）

| 機能 | 除外理由 |
|---|---|
| 会計の取消・返金 | 不可逆操作。設計フェーズで別途検討 |
| 領収書の再発行 | 再発行回数の管理が必要。Phase 4 後半で設計 |
| DailySales 日次集計更新 | 集計ロジックが複雑。Phase 5 で実装 |
| 未収回収処理 | 支払状態更新ロジックが必要。Phase 4 後半で設計 |
| カルテ→会計の自動変換 | 本方針により現フェーズでは実装しない |
| 分割払い対応 | 設計未確定 |
