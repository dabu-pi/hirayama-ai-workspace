## 全体シート構成

Google Sheets 1ファイルに以下のシートを作成する。

| シート名 | 役割 | 優先度 |
|---|---|---|
| **① 候補管理** | AI提案・手動追加銘柄の一元管理（スクリーニング前段） | **最初に作る** |
| **② スクリーニング記録** | 候補をSCREENING_RULESで条件チェックし判定 | **最初に作る** |
| **③ 取引記録** | エントリー・エグジットの全取引履歴（ペーパー含む） | Phase 1 開始時 |
| **④ チャート型記録** | 勝ちパターン／負けパターンの蓄積・評価データベース | Phase 1 開始時 |
| 保有一覧 | 現在の保有銘柄・評価損益の確認 | Phase 2 以降 |
| 損益サマリー | 月次・累計の損益集計 | Phase 2 以降 |

## 候補→スクリーニング→取引 の流れ

```
[AI提案銘柄]  ─┐
               ├──→ ① 候補管理シート ──→ ② スクリーニング記録 ──→ ③ 取引記録
[手動追加銘柄] ─┘    （登録・一次整理）    （条件チェック・判定）    （売買・損益管理）

                      ↓ 除外                ↓ 除外
                   （ここで弾く）         （ここで弾く）
```

**原則: AI提案銘柄も手動追加銘柄も、①から③まで同じ経路・同じ基準で評価する。特別扱いしない。**

## シート①: 候補管理（前段管理）

### 目的

スクリーニングを実施する前の「候補プール」を管理する。
AIが提案した銘柄も、自分で気になった銘柄も、すべてここに登録してから評価を始める。
「なぜこの銘柄を候補にしたのか」の根拠をブラックボックスにしない。

### 列設計（A列〜J列）

| 列 | 列名 | 入力内容 | 入力方法 |
|---|---|---|---|
| A | 登録日 | 候補として登録した日付（YYYY-MM-DD） | 手入力 |
| B | コード | 証券コード（例: 6861） | 手入力 |
| C | 銘柄名 | 銘柄名（例: キーエンス） | 手入力 |
| D | 分類 | A / B / C / D / E（UNIVERSE.md の分類）または TBD | 手入力 |
| E | 登録元 | 下記の値から選択 | 手入力 |
| F | 登録理由 | なぜこの銘柄を候補に上げたかを一言で | 手入力 |
| G | AI根拠ソース | 下記の値から選択（複数ある場合はカンマ区切り） | 手入力 |
| H | 一次コメント | 最初の印象・気になる点・懸念点など | 手入力 |
| I | 判定ステータス | 未判定 / 判定済み | 手入力 |
| J | 最終アクション | 監視継続 / 保留 / 除外 | 手入力 |

### 登録元（E列）の値

| 値 | 意味 |
|---|---|
| `AI_UNIVERSE` | UNIVERSE.md に登録済みの銘柄からAIが候補化した |
| `AI_SCREENING` | AIのスクリーニング補助でSCREENING_RULES条件通過を確認した |
| `AI_NEWS` | AIがニュース・決算材料を分析して候補として提案した |
| `MANUAL_USER` | ユーザー自身が気になって手動で追加した |

### AI根拠ソース（G列）の値

| 値 | 意味 |
|---|---|
| `UNIVERSE登録済み` | UNIVERSE.mdに正規に記載されている対象銘柄 |
| `SCREENING条件通過` | 5ステップチェックリストの条件を確認済み |
| `決算材料` | 直近の好決算・業績上方修正・増配など |
| `ニュース材料` | 新規事業・大型契約・受注・提携などのIRニュース |
| `ユーザー注目銘柄` | ユーザーが自分で気になって登録。AI根拠なし |

> **なぜ根拠ソースを残すか:** 後から「なぜこれを買おうとしたのか」を振り返るため。特にAI提案銘柄は、根拠が曖昧なまま流されてエントリーするリスクがある。ソースを明記することでブラックボックスを防ぐ。

### 運用ルール

- 銘柄を候補にしたいと思ったら、まずここに登録する（スクリーニングより前）
- 登録しただけでは「エントリー候補」ではない。スクリーニング通過後に初めてエントリー検討対象になる
- `MANUAL_USER` の銘柄も `AI_UNIVERSE` の銘柄も、スクリーニング（②）では同じ条件で評価する
- 判定済みになったら J列に最終アクションを記録し、I列を「判定済み」に更新する
- 除外になった銘柄は削除せず残しておく（同じ銘柄を何度も再評価する手間を省くため）

### シートのヘッダー行イメージ

```
A: 登録日 | B: コード | C: 銘柄名 | D: 分類 | E: 登録元 |
F: 登録理由 | G: AI根拠ソース | H: 一次コメント | I: 判定ステータス | J: 最終アクション
```

## シート②: スクリーニング記録

### 目的

候補管理シートで「監視継続」になった銘柄を、SCREENING_RULES.md のチェックリスト（5ステップ）に沿って手動確認し、「通過 / 保留 / 除外」に判定するための記録表。
月に1回程度を目安に実施する（定期スクリーニング）。

### 列設計（A列〜R列）

| 列 | 列名 | 入力内容 | 入力方法 |
|---|---|---|---|
| A | 確認日 | スクリーニングを実施した日付（YYYY-MM-DD） | 手入力 |
| B | コード | 証券コード（例: 6861） | 手入力 |
| C | 銘柄名 | 銘柄名（例: キーエンス） | 手入力 |
| D | 分類 | A / B / C / D / E | 手入力 |
| E | 株価>25日MA | ○ / ✕ | 手入力（チャートで目視確認） |
| F | 株価>75日MA | ○ / ✕ | 手入力（チャートで目視確認） |
| G | 1か月騰落率>+5% | ○ / ✕ | 手入力（始値・終値で計算） |
| H | 出来高条件OK | ○ / ✕ | 手入力（5日平均 ≥ 20日平均×80%） |
| I | 売上成長OK | ○ / ✕ | 手入力（直近決算で前年比プラスか） |
| J | 営業利益成長OK | ○ / ✕ | 手入力（直近決算で前年比プラスか） |
| K | PER≦50 | ○ / ✕ | 手入力（Yahoo!ファイナンス等で確認） |
| L | 自己資本比率≧30% | ○ / ✕ | 手入力（決算短信・四季報で確認） |
| M | 総合判定 | **通過 / 保留 / 除外** | 手入力（下記ルールで判定） |
| N | 保留・除外理由 | 満たさなかった条件を簡潔に記入 | 手入力（例:「75日MA下」「PER62倍」） |
| O | 監視メモ | 次回確認時のポイント・気づきなど | 手入力 |
| P | 登録元 | 候補管理シートの登録元と同じ値 | 手入力（AI_UNIVERSE等） |
| Q | データ確認日 | ファンダデータ（I〜L列）を確認した日付 | 手入力（30日ルールの管理用） |
| R | AIコメント | AIに確認補助を依頼した場合の要点メモ | 手入力（任意） |

> **P〜R列は補助列。手入力の負担が大きい場合は最初は空欄でよい。慣れてきたら埋めていく。**

### 総合判定ルール

| 判定 | 条件 |
|---|---|
| **通過** | E〜L の8項目すべてが ○ |
| **保留** | 7項目が ○（1項目のみ ✕）かつ、✕の項目が「出来高」または「騰落率」のみ |
| **除外** | 2項目以上が ✕、またはテクニカル（E/F）・ファンダ主要（I/J）が ✕ |

> 「惜しい」は保留ではなく除外に回すのが基本。保留銘柄は翌月に再チェックする。

### 入力の手順（1銘柄あたり5〜10分）

1. Yahoo!ファイナンスで対象銘柄のページを開く
2. チャートで 25日・75日移動平均線を表示し、E・F列を入力
3. 1ヶ月前の株価と現在値を比較し G列を入力（（現在値÷1か月前値−1）×100 が +5% 超か）
4. 出来高グラフで 5日平均と 20日平均を比べ H列を入力
5. IRページ・決算短信で最新決算を開き I・J列を入力（決算発表日を確認。30日以内か → Q列に記録）
6. Yahoo!ファイナンスの「財務」タブで PER・自己資本比率を確認し K・L列を入力
7. M・N・O列に判定とメモを入力

### 運用ルール

- 実施頻度: **月1回**（相場急変時は追加実施してよい）
- 記録はその月のスクリーニング結果を1行1銘柄で追記していく（上書きしない）
- 「通過」の銘柄が実際のエントリー候補になる。エントリーは③取引記録シートに移行して管理

## シート③: 取引記録

### 目的

エントリー・エグジットの全取引履歴を記録し、損益管理と振り返りに使う。
ペーパートレード（Phase 1）も同じシートに記録し、Phase 2 以降の実取引と区別する列を設ける。

### 列設計（A列〜O列）

| 列 | 列名 | 入力内容 |
|---|---|---|
| A | 日付 | 取引日（YYYY-MM-DD） |
| B | 種別 | **ペーパー / 実取引** |
| C | コード | 証券コード |
| D | 銘柄名 | |
| E | 売買区分 | 買 / 売（部分） / 売（全） |
| F | 取引単価 | |
| G | 数量 | |
| H | 取引金額 | F×G で自動計算可 |
| I | 損切りライン | エントリー時に設定した価格（買い時のみ） |
| J | 部分利確ライン | エントリー価格×1.10（買い時のみ） |
| K | 全利確ライン | エントリー価格×1.20（買い時のみ） |
| L | 損益（円） | エグジット時に記入（売り時のみ） |
| M | 損益（%） | エグジット単価÷エントリー単価−1（売り時のみ） |
| N | エグジット理由 | 損切り / 部分利確 / 全利確 / 期間利確 / 根拠消滅 |
| O | エントリー根拠・メモ | スクリーニング通過の主な根拠を一言で |

### 取引記録の運用ルール

- エントリー時は B〜K・O列を必ず記入してからエントリー（ペーパーも実取引も同じ）
- エグジット時は L・M・N列を記入する
- 1銘柄で部分利確（+10%で50%売却）するときは、売却分と残り保有分を別行で管理する
- 損益（%）の列は月次レビュー時に合計を確認する

## シート④: チャート型記録シート

### 目的

エントリーした「チャート型（パターン）」を記録し、将来的に「勝ちパターン / 負けパターン」を定量的に評価できるデータベースを構築する。感覚での売買を防ぐための記録装置として機能させる。

「なぜこのタイミングでエントリーしたか」を言語化する習慣を積み重ねることで、条件ベースの判断力を高める。

> **Phase 1 の最優先目標は「記録を貯めること」。型別の分析・評価は50件以上蓄積してから行う。**

### 列設計（A列〜N列）

| 列 | 列名 | 入力内容 | 入力方法 |
|---|---|---|---|
| A | 記録日 | エントリーした日付（YYYY-MM-DD） | 手入力 |
| B | コード | 証券コード | 手入力 |
| C | 銘柄名 | 銘柄名 | 手入力 |
| D | 分類 | A / B / C / D / E（UNIVERSE.mdの分類） | 手入力 |
| E | チャート型 | 下記の型リストから選択 | 手入力（選択式推奨） |
| F | エントリー理由 | なぜこのタイミングでエントリーしたかを一言で | 手入力 |
| G | ルール通過状況 | 通過 / 一部未確認 / 特例（理由を記入） | 手入力 |
| H | エントリー価格 | 実際のエントリー単価（円） | 手入力 |
| I | 損切りライン | エントリー価格×0.95（-5%の価格） | 手入力 |
| J | 利確ルール | 部分: +10%で50%売 / 全: +20%で全売（標準値を記録） | 手入力 |
| K | 結果 | **勝ち / 負け / 保留（保有中）** | エグジット後に手入力 |
| L | 損益率（%） | エグジット時の損益率（例: +12.3% / -5.0%） | エグジット後に手入力 |
| M | 失敗理由 / 成功理由 | 結果の要因を一言で | エグジット後に手入力 |
| N | 振り返りメモ | 次回に活かすべき気づき・反省・発見 | エグジット後に手入力 |

### チャート型（E列）の値リスト

| 型 | 説明 |
|---|---|
| `25日線上の押し目` | 25日MAが上向き、一時的な押し後に再上昇を確認してエントリー |
| `75日線上抜け後の初押し` | 75日MAを上抜けた後、初めての押し局面でエントリー |
| `高値更新ブレイク` | 直近の高値を出来高を伴って明確に上抜けたタイミングでエントリー |
| `出来高を伴うボックス上抜け` | 横ばいレンジを出来高の増加を伴って上抜けした場面でエントリー |
| `決算後ギャップアップ継続` | 好決算で窓を開けて上昇後、翌日以降も継続上昇を確認してエントリー |
| `25日線割れ反発失敗` | 25日MAを割り込んだ局面でのエントリー（ルール外・失敗パターンの記録用） |
| `その他` | 上記に当てはまらない場合。F列に詳細を記入 |

> **「25日線割れ反発失敗」をあえて型として残す理由:** ルール違反や条件未満でのエントリーも記録することで、後から「どんな失敗をしたか」を定量的に振り返れる。記録が証拠になり、同じ失敗の繰り返しを防ぐ。

### 運用ルール

- **エントリー直前（H〜J列）をその場で記入してからポジションを持つ。後入力は禁止**
- エグジット後に K〜N列を記入して1行を完結させる
- 記録の「精度」より「続けること」を優先する。振り返りメモは短い一言で十分
- 月次レビュー時にチャート型別の勝率を確認する（Phase 2 以降でGAS自動集計も可）
- ペーパートレード（Phase 1）も実取引と全く同じ列構成で記録する

### ヘッダー行イメージ

```
A: 記録日 | B: コード | C: 銘柄名 | D: 分類 | E: チャート型 |
F: エントリー理由 | G: ルール通過状況 | H: エントリー価格 | I: 損切りライン | J: 利確ルール |
K: 結果 | L: 損益率(%) | M: 失敗/成功理由 | N: 振り返りメモ
```

## AI連携方針

### 基本原則

| 原則 | 内容 |
|---|---|
| **開発ツール** | 実装は Claude Code（ローカルGAS・スクリプト開発）を活用 |
| **判定AIの一本化** | スプレッドシート内でAI判定を使う場合は1系統に固定する。複数のAIを混在させると基準がぶれる |
| **同一基準の徹底** | AI提案銘柄も手動追加銘柄も、候補管理→スクリーニング→取引の同じ経路を通す。特別扱いなし |
| **最終判断は人間** | AIは整理・補助のみ。エントリー・エグジットの決定は必ず人間が行う |

### API設計方針（Phase 2以降向け）

- 候補判定の自動化は **JSON形式で返却する前提** で設計する
- 例: AIに銘柄データを渡すと `{"code": "6861", "judgment": "通過", "reason": "全条件クリア", "source": "SCREENING条件通過"}` のように返す
- これにより GAS でシートに自動書き込みしやすくなる
- Phase 1 では手動で同じ内容を入力するだけでよい（設計は統一しておく）

**AI問い合わせテンプレート（手動版・Phase 1用）:**
```
銘柄: ○○（コード: xxxx）
登録元: AI_UNIVERSE / MANUAL_USER など

以下の情報を確認して、SCREENING_RULES.mdの条件に対するコメントをJSON形式で返してください。
判断はしないでください。情報の整理のみお願いします。

{
  "code": "xxxx",
  "name": "銘柄名",
  "per": "確認したPER",
  "equity_ratio": "自己資本比率",
  "revenue_growth": "増収か",
  "op_profit_growth": "増益か",
  "latest_financial_date": "直近決算日",
  "news_summary": "直近2週間の重大ニュース有無",
  "comment": "気になる点があれば一言"
}
```

### フェーズ別 自動化ロードマップ

| フェーズ | AI連携の状態 | 実装内容 |
|---|---|---|
| **Phase 1（今）** | 手動のみ | AIに情報整理を依頼し、結果を手入力。シート設計はAPI対応の列構成にしておく |
| **Phase 2** | Apps Script + API | GASでAPIを叩き、候補管理・スクリーニング記録を半自動化。判定結果は自動入力、最終確認は人間 |
| **Phase 3** | 半自動化 | 定期スクリーニングをGASで自動実行。通過銘柄をSlack等で通知。エントリー判断は人間 |

## AI連携最小構成（Phase 1〜3 拡張設計）

### 設計思想

初心者が手動運用から始め、慣れたら段階的にAIと自動化を組み合わせられる構成にする。いきなり自動化を目指さない。まず「記録が貯まる仕組み」を作ることが先決。

| 原則 | 内容 |
|---|---|
| **記録が先** | 自動化より記録の習慣化を優先する。Phase 1 は記録装置を完成させる期間 |
| **感覚売買を防ぐ** | チャート型記録シートに「なぜエントリーしたか」を強制的に言語化させる構造にする |
| **条件ベース運用** | 「なんとなく上がりそう」でのエントリーをなくす。型・理由・ラインをすべて先に記録してからポジションを持つ |
| **損失制御重視** | 損益率より損切りラインの遵守を最優先とする。記録が「守れたか」の証拠になる |
| **AIは1系統** | 判定AIは gpt系（OpenAI API）1モデルに固定。Claude は開発支援（コード・設計）に使用。複数AIの混在は評価基準のブレを生む |
| **同一基準の徹底** | AI提案銘柄も手動追加銘柄も、候補管理→スクリーニング→チャート型記録の同じ経路を通す。特別扱いなし |
| **最終判断は人間** | AIは情報整理・補助のみ。エントリー・エグジットの決定は必ず人間が行う |

### Phase 3: 勝敗蓄積後の評価ループ強化

**目的:** チャート型記録が50件以上蓄積されたら、型別の勝率を評価しルールを改良する。

| 評価ループ | 内容 |
|---|---|
| **型別勝率の算出** | チャート型記録シートで型ごとの（勝ち数 ÷ 全試行数）を算出 |
| **勝率60%未満の型は警戒** | 以後そのチャート型でのエントリーは追加確認を必須とする |
| **勝率30%未満の型は一時停止** | 十分なデータが揃うまでそのパターンでのエントリーを禁止 |
| **AIに型の分析を依頼** | 蓄積データをAIに渡し「このパターンで勝てない理由は何か」を問い合わせ、改善案を検討 |
| **ルール改訂** | 20〜30トレード後の INVESTMENT_POLICY.md 見直しタイミングに合わせて実施 |

> **注意:** AIによる「チャート型の評価」はあくまで参考情報。ルール改訂の最終判断は人間が行う。AIの提案をそのまま採用しない。

### チャート記録 → AI連携 の全体フロー（Phase 3 完成形）

```
[候補登録] AI提案 / 手動追加
     ↓
① 候補管理シート（登録元・根拠ソースを明記）
     ↓
② スクリーニング記録（テクニカル + ファンダ 8条件チェック）
     ↓ 通過のみ
③ 取引記録 + ④ チャート型記録シート
  （エントリー時: 型・理由・ラインを先に記録）
  （エグジット後: 結果・損益率・振り返りを記録）
     ↓ 50件以上蓄積
⑤ 型別勝率評価（Phase 3〜）
  （GAS集計 → AI分析補助 → ルール改訂判断（人間））
```
