## 1. なぜ「腰痛専用」から「運動器全体基盤」に変えたか

### 問題意識

最初に `low-back-assessment/` を作った時点では、腰痛評価のシート1枚を作ることが目的だった。
しかし、実際に設計を進めると以下のことが明確になった：

- NRS・赤旗・PSFS・患者基本情報・コメント出力・AI判定連携は、腰痛だけでなく**どの運動器疾患でも使う共通基盤**である
- 接骨院で評価が必要な症状は腰痛だけではない（頸部・肩・膝・姿勢・高齢者機能）
- 「腰痛専用」で作ると、次に首肩評価を作るときに**ゼロから設計し直すことになる**
- 接骨院の治療家として育つためにも、**運動器疾患を局所だけでなく全体から評価する姿勢**が重要

### 変更の判断

- 今は腰痛しか実装しないが、**器は広く作っておく**
- 共通部分（赤旗・NRS・PSFS・コメント出力・AI連携）は基盤として分離する
- 症状固有部分（RMDQ・STarT・腰部動作評価）は腰痛モジュールとして分離する
- このことにより、将来のモジュール追加が「コピー&修正」ではなく「モジュール追加」になる

### 変更しなかったこと

- **Phase 1 の実装範囲は腰痛評価のみ** — 首肩・膝の詳細実装は今はしない
- 既存の評価項目・シート設計・GASコードはそのまま活かす
- 構造を変えるだけで、実装内容は増やさない

## 2. なぜこの命名にしたか

| 項目 | 選択肢 | 採用 | 理由 |

| プロジェクトID | JEVAL-01 / JASSESS-01 | **JASSESS-01** | JEVAL は「評価（Evaluation）」で腰痛専用感がある。JASSESS は「Assessment（評価・査定）」で運動器全般に使えるニュアンス |

## 3. 共通基盤と症状別モジュールの境界

### 共通基盤に含めるもの

以下は「どの症状でも使う」評価要素。スプレッドシートのシート構造では「共通シート」として実装する。

| 評価項目 | 理由 |
|---|---|
| 患者基本情報（氏名・年齢・性別・職業） | 全症状共通 |
| 赤旗スクリーニング | 全症状で必須（重篤疾患除外） |
| NRS（痛みの強度） | 全症状共通の痛み指標 |
| PSFS（患者特定機能評価） | 症状ごとの目標設定に使えるが評価方法は共通 |
| 評価区分（初回・中間・最終） | 全症状共通 |
| 再評価予定日 | 全症状共通 |
| 医療連携を考えるべき条件（一般） | 共通テンプレートが使える |
| 患者向け説明文（構造） | フォーマットは共通。内容は症状別 |
| AI判定用出力（JSON構造） | 共通フォーマット。症状別フィールドを追加する形 |
| スタッフマスタ | 全症状共通 |
| 評価履歴（転記構造） | 共通列 + 症状別列の構成 |

### 症状別モジュールに含めるもの

以下は「腰痛特有」または「症状ごとに異なる」評価要素。

| 評価項目 | 腰痛 | 将来: 頸肩 | 将来: 膝 |
|---|---|---|---|
| 機能障害評価 | RMDQ-10 | NDI or NPQ | KOOS or LEFS |
| 慢性化リスク | STarT簡易版 | STarT頸部版（調整） | （TBD） |
| 動作評価 | 前後屈・側屈 | 頸部ROM | 膝屈伸・歩行 |
| 神経症状 | SLR・デルマトームL/S | 上肢放散・腕神経叢 | 大腿神経・閉鎖神経 |
| 移乗動作 | 腰部起因の移乗困難 | （腰より少ない） | 膝起因の移乗困難 |
| 症状固有コメント | 腰痛コメントセット | 頸肩コメントセット | 膝コメントセット |
| ルールベース判定フラグ | 腰痛フラグ群 | 頸肩フラグ群 | 膝フラグ群 |

## 4. 今なぜ腰痛モジュールだけに絞るか

### 根拠

1. **臨床現場での導入ハードルを下げるため** — 一度にすべての症状評価を実装すると、シートが複雑になりすぎて使われなくなる
2. **フィードバックを早く得るため** — 腰痛評価で実際に使ってみて、評価項目・コメント・判定ロジックの改善点を洗い出す。この経験が他モジュールの設計に活きる
3. **エビデンスの確認** — 腰痛評価ツール（NRS・RMDQ・STarT・PSFS）は研究での有効性が確認されているが、頸肩・膝では使うツールの選定に時間をかけたい
4. **治療家としての習熟** — 平山先生が腰痛評価の考え方を身につけてから他症状へ展開する方が、評価者としての理解が深まる

## 5. 将来どう拡張するか

### 拡張の設計方針

```
msk-assessment-platform/
├── 共通基盤（gas/setup_sheets.js の共通シート群）
│   ├── 患者マスタ
│   ├── 評価履歴（共通列 + 症状別列）
│   ├── コメントマスタ（共通 + 症状別テンプレート）
│   └── AI判定用出力
│
└── modules/               ← 症状別モジュール（仕様書）
    ├── low-back/          ← Phase 1（実装済み設計）
    ├── neck-shoulder/     ← Phase 2（仕様書のみ、将来）
    ├── knee/              ← Phase 3（仕様書のみ、将来）
    ├── posture/           ← Phase 4（仕様書のみ、将来）
    └── elderly-function/  ← Phase 4（仕様書のみ、将来）
```

### モジュール追加時の手順（将来）

1. `modules/<module-name>/README.md` に評価項目・ロジック・コメントを記述
2. `gas/setup_sheets.js` に症状別入力シートと判定ロジックシートを追加する関数を実装
3. 評価履歴シートに症状別列を追加（既存列は維持）
4. コメントマスタに症状別テンプレートを追加
5. AI判定用JSONに症状別フィールドを追加

## 6. このシステムが「治療家育成の基盤」である理由

評価シートを使いながら施術者が身につけることを設計に組み込んでいる：

| 評価項目 | 施術者が学ぶこと |
|---|---|
| 赤旗スクリーニング | 見逃してはいけない重篤疾患の知識と判断基準 |
| STarT簡易版 | 慢性化リスクの概念と心理社会的因子の影響 |
| RMDQ | 「痛みがあるか」だけでなく「動けるか」の視点 |
| PSFS | 患者さんの目標を引き出す聴き方と目標設定 |
| ルールベース判定 | 評価結果をどう施術方針に変換するかの思考回路 |
| コメント自動生成 | 評価→説明の一貫性。説明の型を身につける |
| 再評価比較 | 改善を数値で可視化し患者・施術者双方が確認する習慣 |

評価シートを使い続けることが、**「評価できる治療家」から「説明できる治療家」へ、さらに「再発予防まで設計できる治療家」へ**の成長を支援する。
