## 5. 既存腰痛版に触らない範囲の明確化

### 変更しないもの（保護対象）

| 対象 | 理由 |
|---|---|
| `腰痛評価入力` シートの全セル・数式 | 稼働中の評価入力画面。触ると現在の運用が止まる |
| `判定ロジック` シートの全計算式 | `logic_engine.js` の依存先。変更すると判定が壊れる |
| `コメントマスタ` シートの既存行 | 腰痛コメントが格納されている。行削除・移動禁止 |
| `logic_engine.js` の既存コード | 腰痛用の onEdit・フラグ・コメント生成がある |
| `CELLS` 定数 | 腰痛シートのセル位置が定義されている |
| `TRIGGER_CELLS` 定数 | 腰痛シートの監視対象セルが定義されている |

### 変更可（追加のみ）

| 対象 | 変更内容 | 安全性 |
|---|---|---|
| `評価履歴` シート | 右端に列追加 | 既存列に影響なし |
| `設定` シート | 末尾に行追加 | 既存設定に影響なし |
| `SHEET_NAMES` 定数 | 頚肩こり用シート名を追記 | 既存定数はそのまま残す |
| `setupAllSheets()` | 呼び出し行を追加（既存呼び出しはそのまま） | 実行時の順序を後方に追加 |

## 9. 変更理由の記録

### なぜ共通_初期評価を新設するか
患者基本情報・赤旗・NRS・PSFS は腰痛・頚肩こり双方で重複するため、1回入力で両モジュールに流用できる構造にする。これにより、同日に複数部位を評価する場合の二重入力を防ぐ。

### なぜ頚肩こり_コメントマスタを別シートにするか
既存 `コメントマスタ` に頚肩こりコメントを追加すると、将来のモジュール削減時に関連コメントを特定しにくくなる。分離することで、モジュール単位での管理・削除・修正が容易になる。

### なぜ logic_engine_neck_shoulder.js を別ファイルにするか
既存 `logic_engine.js` に直接追記すると、腰痛ロジックと頚肩こりロジックが混在し、どちらかの修正時にもう一方に影響するリスクがある。ファイルを分けることで、互いのロジックを独立させる。

### なぜ評価履歴は列追加で対応するか
シートを分けると、過去の評価履歴を一覧比較する際に複数シートをまたぐ必要が生じる。列追加方式なら1シートで全モジュールの履歴を管理でき、患者ごとの経過確認が容易になる。
