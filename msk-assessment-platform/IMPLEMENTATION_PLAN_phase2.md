## 1. 方針サマリー

| 原則 | 内容 |
|---|---|
| 既存腰痛版は温存 | `腰痛評価入力` / `判定ロジック` / `コメントマスタ` の既存構造は変更しない |
| 追加のみで対応 | 新しいシートは新規追加。既存シートへの変更は「後方追加」のみ |
| GAS は別ファイル化 | 頚肩こり用ロジックは `logic_engine_neck_shoulder.js` として独立させる |
| 既存 onEdit は変更しない | 既存 `onEdit` は腰痛シートのみ対象。頚肩こり用は別トリガーを追加 |
| 評価履歴は列追加 | 既存列はそのまま保持し、右端に頚肩こり固有列を追加する |
| 設定シートは行追加 | 既存設定行はそのまま保持し、末尾にセクション区切りを入れて追加する |

## 5. 既存腰痛版に触らない範囲の明確化

### 変更しないもの（保護対象）

| 対象 | 理由 |
|---|---|
| `腰痛評価入力` シートの全セル・数式 | 稼働中の評価入力画面。触ると現在の運用が止まる |
| `判定ロジック` シートの全計算式 | `logic_engine.js` の依存先。変更すると判定が壊れる |
| `コメントマスタ` シートの既存行 | 腰痛コメントが格納されている。行削除・移動禁止 |
| `logic_engine.js` の既存コード | 腰痛用の onEdit・フラグ・コメント生成がある |
| `CELLS` 定数 | 腰痛シートのセル位置が定義されている |
| `TRIGGER_CELLS` 定数 | 腰痛シートの監視対象セルが定義されている |

### 変更可（追加のみ）

| 対象 | 変更内容 | 安全性 |
|---|---|---|
| `評価履歴` シート | 右端に列追加 | 既存列に影響なし |
| `設定` シート | 末尾に行追加 | 既存設定に影響なし |
| `SHEET_NAMES` 定数 | 頚肩こり用シート名を追記 | 既存定数はそのまま残す |
| `setupAllSheets()` | 呼び出し行を追加（既存呼び出しはそのまま） | 実行時の順序を後方に追加 |

## 6. 評価履歴に追加する列

### 既存列（変更しない）

現在の評価履歴シートには腰痛モジュール用の列が並んでいる。
これらの列番号は変更せず、右端に続けて頚肩こり固有列を追加する。

### 追加する列（末尾追加）

| 追加列 | 項目名 | 内容 |
|---|---|---|
| +1 | モジュール種別 | 腰痛 / 頚肩こり / 複合（既存行は空欄のまま） |
| +2 | NS-NRS | 頚肩こり評価時のNRS |
| +3 | NS-神経症状レベル | なし / 神経根性 / 頚髄症疑い |
| +4 | NS-ROM制限型 | 正常 / 単方向 / 複合 / 全方向 / 疼痛誘発 |
| +5 | NS-生活負荷フラグ | 標準 / 高 |
| +6 | NS-総合方針 | GAS 判定テキスト |
| +7 | NS-次の介入先 | 施術 / 運動療法 / 医療連携 |

> **注意:** モジュール種別列は既存の腰痛評価行に「腰痛」と自動入力する処理を
> 新規 GAS で追加する（既存 GAS は触らない）。

## 7. 設定シートへの追加内容

既存設定行の固定セル番地には依存せず、`setup_neck_shoulder.js` では `lastRow + 2` を基準に末尾追加する。

```
[既存設定: 既存行は変更しない]

B(addAt):     ─────────────────────────
B(addAt + 1): [頚肩こりモジュール設定]
B(addAt + 2): NS: NRS高強度閾値               C(addAt + 2): 7
B(addAt + 3): NS: ROM制限（著明）判定値        C(addAt + 3): 著明制限
B(addAt + 4): NS: 生活負荷高リスク閾値（PC）   C(addAt + 4): 7時間以上
B(addAt + 5): NS: 放散方向 プルダウン選択肢    C(addAt + 5): なし,片側,両側
B(addAt + 6): NS: ROM評価 プルダウン選択肢     C(addAt + 6): 正常,軽度制限,著明制限,疼痛再現
B(addAt + 7): NS: 前傾頭位 プルダウン選択肢    C(addAt + 7): なし,軽度,中等度,著明
```

> 固定番地ではなく末尾追加のため、Phase C では設定値取得時もシート末尾追加を前提に扱うこと。

## 9. 変更理由の記録

### なぜ共通_初期評価を新設するか
患者基本情報・赤旗・NRS・PSFS は腰痛・頚肩こり双方で重複するため、1回入力で両モジュールに流用できる構造にする。これにより、同日に複数部位を評価する場合の二重入力を防ぐ。

### なぜ頚肩こり_コメントマスタを別シートにするか
既存 `コメントマスタ` に頚肩こりコメントを追加すると、将来のモジュール削減時に関連コメントを特定しにくくなる。分離することで、モジュール単位での管理・削除・修正が容易になる。

### なぜ logic_engine_neck_shoulder.js を別ファイルにするか
既存 `logic_engine.js` に直接追記すると、腰痛ロジックと頚肩こりロジックが混在し、どちらかの修正時にもう一方に影響するリスクがある。ファイルを分けることで、互いのロジックを独立させる。

### なぜ評価履歴は列追加で対応するか
シートを分けると、過去の評価履歴を一覧比較する際に複数シートをまたぐ必要が生じる。列追加方式なら1シートで全モジュールの履歴を管理でき、患者ごとの経過確認が容易になる。

### なぜ Phase C の正本を setup_neck_shoulder.js に寄せるか
Phase B でシート構造がすでに実装されているため、文書側の旧セル番地を残したまま Phase C を進めると、GAS の書き込み先や参照先がずれて不具合につながる。実装済みコードを正本に固定し、文書はそれを説明する役割にそろえる。
