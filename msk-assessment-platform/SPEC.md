最終更新: 2026-03-23（JEVAL-01 → JASSESS-01 再整理 / 運動器全体基盤として再定義）

## 1. プロジェクト目的

接骨院における運動器疾患の初回・再評価業務において：

1. **評価の標準化** — 施術者が変わっても同じ評価軸で記録できる
2. **評価者差の低減** — プルダウン・チェックボックス中心で主観ブレを最小化
3. **評価→方針の一貫化** — 入力すると評価まとめ・注意点・施術優先順位・セルフケア案が自動表示される
4. **治療家育成の基盤** — 「評価できる」→「説明できる」→「再発予防まで設計できる」成長を支援
5. **将来AI連携の基盤** — 施術者判定 / ルールベース / AI の3層判定構造を前提にしたデータ設計

### 現在の実装対象

**Phase 1 = 腰痛評価モジュール**
腰痛（慢性腰痛・急性腰痛・再燃例）の初回・再評価が最初の実装対象。

### 将来の拡張対象

| フェーズ | 対象症状 |
|---|---|
| Phase 2 | 頸部痛・肩こり（neck-shoulder） |
| Phase 3 | 膝慢性痛（knee） |
| Phase 4a | 姿勢評価（posture） |
| Phase 4b | 高齢者機能・移乗評価（elderly-function） |

## 2. 治療家育成との接続

このシステムは、評価→施術→セルフケア→再評価の一貫したフローを支援し、施術者が：

```
痛みの背景を整理できる
  → 原因を患者に説明できる
    → 回復の見通しを伝えられる
      → 施術と運動療法を組み合わせられる
        → 再発予防まで設計できる
          → 患者と二人三脚で改善を進められる
```

このサイクルを繰り返すことで、**結果が出せて、信頼され、紹介される治療家**への成長を支援する。

## 3. システム構成の設計思想

### 共通基盤と症状別モジュールの分離

```
共通基盤（全症状で使う）
  ├── 患者基本情報
  ├── 赤旗スクリーニング
  ├── NRS（痛みの強度）
  ├── PSFS（患者特定機能評価）
  ├── 評価区分・再評価予定
  ├── 医療連携判断基準
  ├── コメント出力（構造）
  ├── AI判定用出力（JSON構造）
  ├── スタッフマスタ
  └── 評価履歴（共通列）

症状別モジュール（症状ごとに異なる）
  ├── 機能障害評価スケール（腰痛: RMDQ / 頸部: NDI / 膝: KOOS等）
  ├── 慢性化リスクスクリーニング（腰痛: STarT / 頸肩版等）
  ├── 動作評価（腰痛: 前後屈・側屈 / 頸部: ROM / 膝: 屈伸）
  ├── 神経症状評価（部位別）
  └── 症状固有のコメント・判定ロジック
```

### Phase 1 での実装（腰痛モジュール）

共通基盤 + 腰痛固有項目 を含む8シート構成で実装する。

## 4. 評価項目の根拠（Phase 1: 腰痛モジュール）

### 採用した評価ツール

| 評価ツール | 略称 | 位置づけ | エビデンス概要 | 採用判断 |
|---|---|---|---|---|
| Numeric Rating Scale | NRS | **共通基盤** | 感度・再現性ともに高い（VASと同等）。患者・施術者双方に簡便 | ✅ 採用 |
| Roland Morris Disability Questionnaire（短縮版） | RMDQ-10 | **腰痛モジュール** | 腰痛機能障害の国際標準指標（Ostelo 2008） | ✅ 採用（10項目版） |
| STarT Back Screening Tool（日本語版相当） | STarT簡易版 | **腰痛モジュール** | 慢性化リスクスクリーニング。RCTで有効性確認（Hill 2011, Lancet） | ✅ 採用（9項目版） |
| Patient-Specific Functional Scale | PSFS | **共通基盤** | 患者個人の機能目標に対する達成度評価（Stratford 1995） | ✅ 採用 |
| 赤旗（Red Flags） | RF | **共通基盤** | 重篤疾患除外スクリーニング（NICE 2016等、全診療ガイドライン共通） | ✅ 採用 |
| 神経症状評価（SLR等） | — | **腰痛モジュール** | 神経根障害・馬尾症候群除外 | ✅ 採用（簡易版） |
| 動作評価（前後屈・側屈） | — | **腰痛モジュール** | 機能障害の客観的記録 | ✅ 採用 |
| 移乗動作評価 | — | **腰痛モジュール** | 高齢者・ADL低下例で必須。転倒リスクと連動 | ✅ 採用 |

### 採用しなかったもの・理由

| ツール | 除外理由 |
|---|---|
| FABQ（恐怖回避思考質問票） | 初版では複雑すぎる。STarT簡易版で心理社会的因子はカバー |
| ODI（Oswestry Disability Index） | RMDQより現場負担が大きい |
| SLR角度の精密計測 | 角度計不要で「陽性/陰性/疼痛誘発」の3段階で代替 |

## 5. シート構成（Phase 1: 腰痛モジュール実装）

### 共通基盤シート（6シート）

| シート名 | 位置づけ | 役割 |
|---|---|---|
| 設定 | **共通** | 施設情報・プルダウン選択肢マスタ・スコア閾値 |
| 患者マスタ | **共通** | 患者基本情報の一元管理 |
| 評価履歴 | **共通** | 評価結果の時系列記録（共通列 + 腰痛列） |
| コメントマスタ | **共通+腰痛** | 条件別コメントテンプレート（共通 + 腰痛固有） |
| AI判定用出力 | **共通** | AI連携時のJSON出力（将来用） |
| スタッフマスタ | **共通** | 施術者一覧・資格情報 |

### 腰痛モジュールシート（2シート）

| シート名 | 位置づけ | 役割 |
|---|---|---|
| 腰痛評価入力 | **腰痛固有** | 腰痛評価の全項目入力シート（共通項目 + 腰痛固有項目） |
| 判定ロジック | **腰痛固有** | ルールベース判定の中間計算（通常非表示） |

> 将来モジュール追加時は「{症状}評価入力」「{症状}判定ロジック」シートを追加する。

## 6. データフロー

```
施術者が「腰痛評価入力」シートに入力
    ↓
onEdit トリガー（GAS）が発火
    ↓
「判定ロジック」シートで自動計算（腰痛ルール）
    ↓
評価コメント・注意点・方向性が入力シートに自動表示
    ↓
「記録確定」ボタン押下
    ↓
「評価履歴」シートへ転記
    ↓
（将来）「AI判定用出力」シートへJSON出力
    ↓
（将来）Claude API呼び出し → AI判定コメント返却
```

---

## 7. 判定3層構造（全フェーズ）

```
Layer 1: ルールベース判定（GAS自動）  ← Phase 1 で実装
Layer 2: 施術者判定（人間入力）        ← Phase 1 から運用
Layer 3: AI判定（Claude API）         ← Phase 5 で実装
         ↓
最終コメント出力（施術者が確認・補完）
```

## 8. 設計原則

| 原則 | 具体的実装 |
|---|---|
| 入力負荷最小 | プルダウン・チェックボックス中心。テキスト自由入力は所見欄のみ |
| 評価者差の低減 | 判定基準をセルコメントで常時表示 |
| 将来拡張性 | 共通基盤・症状別モジュールの分離。modules/ 配下に仕様を追加 |
| 単価・閾値の外部管理 | スコア閾値は「設定」シートで管理。コードにハードコードしない |
| 安全ネット | 赤旗・馬尾症候群項目は入力時に自動アラート（背景色変化） |
| 患者説明連動 | 評価結果から「患者向け説明文」を自動生成 |
| 治療家育成 | 評価項目の意味・基準・背景が学べるよう設計書と連動 |

## 10. 用語定義

| 用語 | 定義 |
|---|---|
| 共通基盤 | どの症状でも使う評価項目・シート・ロジック |
| 症状別モジュール | 腰痛・頸肩・膝など症状固有の評価項目・ロジック |
| 腰痛モジュール | Phase 1 で実装する症状別モジュール |
| 赤旗（Red Flags） | 重篤疾患（骨折・腫瘍・感染・馬尾症候群等）を示唆する所見 |
| 慢性化リスク | 腰痛が慢性化・遷延化する心理社会的・身体的リスク因子 |
| 施術者判定 | 施術者が総合的に入力する治療優先順位・方向性 |
| ルールベース判定 | GASのif/switch処理で自動生成される判定・コメント |
| AI判定 | Claude APIが評価データをもとに生成する判定・コメント（Phase 5〜） |
