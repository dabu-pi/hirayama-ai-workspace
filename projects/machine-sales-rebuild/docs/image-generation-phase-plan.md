最終更新: 2026-04-05

## 現在の判断

700x700 派生画像生成の実装には、まだ進まない。

理由:

- 現行 `画像1〜3` は正本URLとして使えない
- `source_image_count=0` が 924件すべて
- 自社商品画像の正本保管先が未確定

## 進行条件

| 条件 | 現状 | 判定 |
|---|---|---|
| 元画像の正本が分かる | 未確定 | 未達 |
| 元画像の取得方法が再現できる | 未確認 | 未達 |

## 再開時の入力

- 自社商品画像の正本一式
- 商品コードと画像ファイル名の対応ルール
- `product_master_v0.full.csv`
- 今後の保存先ルール

## 再開時の出力

- `data/derived-images/` 配下の 700x700 表示用画像
- 画像生成ログ
- `products.json.images[].displayUrl`

## 想定構造

- 元画像置き場: `data/raw-images/`
- 派生画像置き場: `data/derived-images/`
- 将来の表示用命名案: `{sd_product_code}-{sortOrder:02d}-700x700.jpg`

## 正本候補の現時点整理

- 自社商品画像:
  - 第一候補は旧 WordPress 反映系の保管資産
  - ただし workspace / OneDrive の手元確認では `generate.php` や media 控えは未回収
- 競合画像:
  - 既存 docs から Google Drive フォルダ `1Q0vGVu2N8Ouq8us0JIMSaH1oCdHVLiZl` に `商品ID_画像枝番` 形式で保存していた痕跡あり
  - 自社画像とは保管系統を分けて扱う前提が妥当

## 次に必要な調査

1. `strongdepot-product-manager` の PHP / WordPress 側実装を回収する
2. 過去の WordPress メディアまたはローカル控えに、商品コードを含む画像ファイルが残っているか確認する
3. Google Drive に自社商品画像フォルダが存在するか確認する
4. 競合画像と自社画像の正本を同じ設計に載せるか、別管理にするか決める

## 進める / 止める判断基準

進める:

- 元画像の実体が取得できる
- 商品コード単位で画像群を拾える
- 試験的に 5〜10商品で原本と派生画像を対応付けできる

止める:

- 正本が WordPress 内にしかなく回収経路が見つからない
- 商品コードと画像ファイルの対応が曖昧
- URLだけではなくログイン必須・手作業必須で再現不能
