# 700x700 派生画像生成フェーズ計画

## 目的

次フェーズで、元画像URLから 700x700 正方形の表示用画像を生成し、`products.json.images[].displayUrl` を埋められる状態にする。

## 入力

| 入力 | 内容 |
|---|---|
| `product_master_v0` | `sd_product_code`, `source_image_urls_json`, `main_image_index` |
| 元画像URL | 現時点では `ネットショップ商品一覧` の `画像1〜3` から取れないため、別ソース確定が先 |
| 画像生成設定 | 出力サイズ 700x700、収まり優先、余白背景で補完 |

## 出力

| 出力 | 内容 |
|---|---|
| 表示用画像ファイル | 700x700 正方形の派生画像 |
| 画像マニフェスト | `sd_product_code`, `sortOrder`, `sourceUrl`, `displayUrl`, `status`, `errorMessage` |
| 更新後 `product_master_v0` / `products.json` | `displayUrl` を反映した派生出力 |

## 派生画像の命名案

```text
{sd_product_code}-{sortOrder:02d}-700x700.jpg
```

例: `OOCY16003LG-01-700x700.jpg`

元画像の差し替え検知が必要なら、将来 `sourceHash` や `updatedAt` をマニフェストへ追加する。

## 保存先候補

| 候補 | メリット | 保留点 |
|---|---|---|
| Google Drive 派生画像フォルダ | 現行運用と近く、手動確認しやすい | 公開配信URL、キャッシュ、アクセス権設計が要確認 |
| オブジェクトストレージ | 静的配信と再生成管理がしやすい | 導入先サービス選定が必要 |
| リポジトリ管理外のローカル生成物 | まず試作しやすい | 本番配信にはそのまま使えない |

## 生成ルール案

- 元画像の縦横比は維持する
- 商品全体が収まるように内接リサイズする
- 足りない余白を背景色で埋めて 700x700 にする
- 過度な中央トリミングはしない
- 画像1枚目を代表画像初期値とし、`main_image_index` があればそれを優先する

## 失敗時の扱い

- 元画像URL取得失敗、画像デコード失敗、保存失敗を商品コード・画像順ごとに記録する
- 商品行自体は落とさず、失敗画像だけ `displayUrl=null` のまま残す
- 一定件数以上失敗したらバッチ終了ステータスを warning/error にする

## 再生成タイミング案

- 試作段階: 手動CLIで一括再生成
- 運用段階候補: 元画像URL変更行だけ差分再生成、または夜間バッチ再生成

## 今回まだ決めないこと

- 背景色を白固定にするか、サイト背景色に合わせるか
- JPEG / WebP / PNG の最終出力形式
- サムネイル専用派生を別途持つか
- Drive / CDN / オブジェクトストレージのどこを本番正とするか
