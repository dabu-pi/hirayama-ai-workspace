## もう設計に進んでよい領域

| 領域 | 進めてよい理由 | 次に作るもの |
|---|---|---|
| 商品マスタv0列設計 | 現行商品列、商品コード仕様、状態/カテゴリ/店舗/メーカー属性がかなり確定した | `商品マスタ`、`設定マスタ`、`サイト出力ビュー` の最小列確定版 |
| `products.json` データ契約 | WordPress非依存で必要な商品項目と公開制御の方向性が固まった | サンプルJSON 3〜5件、現行列→JSON項目マッピング |
| 商品コード新生成バリデーション | 現行仕様と例外/バグが把握できた | 未知マスタ時エラー、通し番号重複チェック、旧コード保持ルール |
| 競合データ新スキーマ | recyfit取得項目、画像保存、分類レビュー課題が見えている | `競合価格データ` タブのv0列確定、手動レビュー状態列 |
| WordPress切り離し方針 | taxonomy/post状態/`generate.php` を出力層へ隔離する方向は明確 | `site_category_map`、`legacy_wp_*` 隔離列の最小設計 |

## 新システムで先に切り離せる部分

| 先に切り離せる部分 | 理由 | 条件 |
|---|---|---|
| WordPress taxonomy文字列の中核商品マスタ混入 | 業務カテゴリとCMS slugを分ければ、現行反映を残したまま新サイト用ビューを並行生成できる | `設定マスタ` と `サイト出力ビュー` を別タブ化する |
| `products.json` 出力 | 現行サイト反映PHPを止めず、読み取り専用で並行生成できる | 現行商品マスタからの変換マッピングを確定する |
| 競合価格データの正規化ビュー | recyfit生データを壊さず、新タブ/新シートで分類レビューできる | 旧 `リサイフィット` は読み取り元として残す |
| 商品コード構成要素の独立列化 | 既存 `sd_product_code` を保持したまま、店舗/メーカー/年/部位を別列で管理できる | 既存コードを再生成で上書きしない |

## 並行運用が必要な部分

| 並行運用が必要な部分 | 理由 |
|---|---|
| 現行 `ネットショップ商品一覧` → WordPress/PHP反映 | `generate.php` の内部未回収で、いきなり止めるとサイト商品反映が止まる可能性が高い |
| 見積 `mitsumori` / 顧客別タブ / `2024長谷川さん` / freee-automation | 見積作成、案件台帳、freee/Gmailが現場業務に直結しており、正本フロー未確定のまま切替できない |
| recyfit競合収集 `リサイフィット` | 現行価格判断の入力なので、新分類ビューが安定するまでは旧収集を残す |
| BASE出力 | 現役度未確認のため、不要確定までは旧出力タブを残す |

## 一気に切替してはいけない部分

| 対象 | 理由 | 安全な進め方 |
|---|---|---|
| `sendHttpPost()` / `generate.php` | PHP内部副作用が見えず、WordPress商品DBを直接更新している可能性がある | PHPソース回収→影響範囲特定→新出力と並行稼働→段階停止 |
| 見積テンプレートGAS `myOnEdit` | 編集時自動発火のため、トリガーを誤って止めると現場入力補助が消える | 新見積入力プロトタイプ完成まで旧トリガー維持 |
| `sd_product_code` 再生成 | 既存見積・案件・サイトリンクの参照キーになっている可能性が高い | 既存行は保持、新規行だけ新バリデーションで生成 |
| `ルール` / GAS配列 / `Settings.php` のマスタ置換 | 三重管理のどれが現サイト本番表示に効いているか未確定 | 差分表を作り、読み取り互換を保ちながら単一マスタへ寄せる |

## 判断サマリ

| 判断項目 | 現時点の状態 |
|---|---|
| 新統合スプレッドシート v0 設計に進めるか | 商品・設定・サイト出力・競合は進めてよい。見積は正本フロー最終確認後に細部調整 |
| 新サイト `products.json` 設計に進めるか | 進めてよい |
| WordPress切替手順を確定できるか | `generate.php` / `Settings.php` 未回収のためまだ不可 |
| 現行GAS停止順序を確定できるか | 商品/見積/競合GASの scriptId と実トリガー未回収のためまだ不可 |

## 今回新たに確定したこと

- 設計に進める領域と、scriptId/PHPソース未回収のせいでまだ止めるべき領域を切り分けられた。
- 特に商品マスタv0と `products.json` は先行設計可能、WordPress/PHP反映停止順序と見積正本切替はまだ危険、という判断が明確になった。
