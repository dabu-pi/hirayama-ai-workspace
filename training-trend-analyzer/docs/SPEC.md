## 要件定義概要

MVP（自社利用）と公開版（将来）で要件が異なる部分は明記する。

## データ更新方針

| 項目 | 方針 |
|---|---|
| 更新頻度 | MVP: 週次。将来: 日次（急騰検知は随時） |
| 更新タイミング | 月曜AM（前週データを集計して月初/月中に参照） |
| 欠損時の扱い | 欠損スナップショットは NULL で記録。スコア計算からは除外 |
| 過去データの修正 | 原則禁止。修正が必要な場合は別レコードで上書きし履歴を残す |
| データ保持期間 | 無制限（ストレージ逼迫時に古い生データを圧縮） |

## ランキングロジックの考え方

### 基本方針

- **単一指標禁止**: 1つの指標だけでランキングすると偏りが大きい
- **合成スコア方式**: 複数指標を正規化してから加重平均
- **変化率重視**: 絶対値より「どれだけ動いたか」を重視
- **計算式の透明化**: ブラックボックス化しない

### スコア構成（初期案）

| 指標 | 重み（初期） | ソース例 |
|---|---|---|
| 検索ボリューム変化率 | 30% | Google Trends |
| SNS言及数変化率 | 25% | Twitter/Instagram |
| EC評価・レビュー数変化率 | 20% | Amazon, 楽天 |
| 業界メディア掲載回数 | 15% | 専門サイト |
| 問い合わせ・引き合い数 | 10% | 自社データ（将来） |

> 重み付けは実データ収集後に調整する。初期は均等（20%×5）でも可。

### 変化率の計算

```
trend_score = Σ (normalized_value_i × weight_i)
change_rate = (current_score - prev_score) / prev_score × 100
```

### ランクラベル基準（初期案）

| ラベル | 条件 |
|---|---|
| 急上昇 | change_rate > +30% |
| 上昇 | +10% < change_rate ≤ +30% |
| 安定 | -10% ≤ change_rate ≤ +10% |
| 下降 | -30% ≤ change_rate < -10% |
| 急下降 | change_rate < -30% |

> 閾値は実データを見てから調整する。

## データの信頼性に関する注意

| 注意事項 | 内容 |
|---|---|
| サンプルバイアス | 日本語ソースは国内偏重。グローバルトレンドは別途確認 |
| 季節変動 | 年明け・夏前はフィットネス全体が上がる。季節調整は将来対応 |
| バイラル補正 | 一時的な炎上・バズはトレンドスコアを歪める。外れ値除去を検討 |
| ソース依存 | 1ソースが欠損した週はスコアが変動。欠損フラグを付与する |
| 自社データの混在 | 自社の引き合いデータを混ぜる場合は社内用スコアと分離する |

| 価格情報 | 参照可 | 検討中 |
