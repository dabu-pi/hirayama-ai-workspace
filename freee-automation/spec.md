# freee 見積自動化 仕様書

作成日: 2026-03-03

---

## 1. 目的

`hawk@pop13.odn.ne.jp`（長谷川さん）からの見積依頼メールを起点に、以下を自動化して業務を省力化する。

1. メール内容を確認し必要情報を抽出
2. freee の見積書で見積を作成
3. 見積ができたら PDF を添付した返信メールを「下書き」作成（送信しない）
4. 案件管理用の Google スプレッドシートに進捗を自動記録（「2024長谷川さん」ページ）

---

## 2. 現行業務フロー

- 長谷川さん（hawk）が顧客に営業を行い、弊社へ見積・請求依頼メールが届く
- 弊社は見積書作成（freee）／請求書作成／入金管理などの事務を担当
- 進捗はスプレッドシートで管理

## 6. メールからの情報抽出

### 件名ルール

- 例: `見積もりの作成お願いします（ASRE様）`
- 括弧（）内の文字列を **お客様名**（B列）として優先使用

### 抽出対象フィールド

| フィールド | 用途 | 必須/任意 |
|-----------|------|----------|
| 取引先名（会社名/担当者名） | freee 見積の宛先 | 必須 |
| 見積内容（作業内容・機種・型番等） | D列 / 明細 | 必須 |
| 明細（品目/数量/単価 or 一式金額） | freee 明細行 | 必須 |
| 送料・出張費・消費税の扱い | freee 調整行 | 任意 |
| 納期/作業予定日 | 備考欄 | 任意 |

### メールテンプレ（推奨フォーマット）

長谷川さんへの依頼：メール本文をできる限りテンプレ化し、抽出精度を向上させる。

## 8. 事故防止（絶対条件）

1. **二重処理を絶対に避ける**
   - Gmail ラベル付与（処理済みマーク）
   - シートに Message-ID を保存し冪等性を担保
2. **勝手に送信しない**
   - 自動化は必ず「下書き保存」止まり
   - 人が最終確認して送信する

## 9. 実装フェーズ

### フェーズ 1（最優先・安定）: メール → シート記録

**目的**: 案件の取りこぼしゼロ

- 対象メール検知（時間トリガー 5〜15分）
- 「2024長谷川さん」へ追記: B/C/D/E/O 列
- 処理済みラベル付与
- Message-ID 保持で重複防止

**成果**: 台帳に自動で積まれる状態（見積作成はまだ）

**現状**: 動作し始めている ✅

---

### フェーズ 2（半自動 OK）: freee 見積作成

**目的**: 見積作成工数削減

- メールから情報抽出 → freee で見積作成
- 成功時 G 列に見積日を自動記録
- freee 見積 ID / URL をシートに保存（列追加 or メモ）
- 抽出不完全な場合は「要確認」フラグを立てて残す

**現状**: freee 本番 OAuth の `invalid_grant / 401` エラーで詰まり中 → 再構築中 🔧

---

### フェーズ 3: Gmail 下書き作成（PDF 添付）

**目的**: 返信作業を「確認だけ」に

- 見積作成できたら:
  - 見積 PDF を添付
  - 返信文を生成
  - 下書きに保存（送信しない）
- PDF 添付が難しい場合の暫定案: freee 見積 URL を本文に貼る

---

### フェーズ 4: 運用強化

- エラー通知（Slack / メール等）
- 取引先マスタ整備（顧客名 → freee 取引先 ID）
- 品目マスタ整備（定型品目）
- 重複検知（同一顧客・同一件名）

## 10. freee 本番 OAuth 再構築メモ

| ポイント | 内容 |
|---------|------|
| `redirect_uri` | GAS WebアプリURL `/exec` と完全一致が必要 |
| `code` | 使い捨て（再利用不可）|
| Client Secret 更新 | GAS 側の ScriptProperties も必ず更新する |
| Token 保存場所 | ScriptProperties（Token / RefreshToken）|
| リセット手順 | `reset()` 関数で ScriptProperties の OAuth 情報を全削除してからやり直す |

## 11. 技術スタック

- **実行環境**: Google Apps Script（GAS）
- **認証**: OAuth 2.0（freee）、Google APIs（Gmail / Sheets）
- **主要 API**:
  - Gmail API（メール取得・ラベル付与・下書き作成）
  - Google Sheets API（スプレッドシート書き込み）
  - freee API（見積書作成・PDF 取得）

### T列（要確認フラグ）仕様

| 値 | 意味 | 対応 |
|---|---|---|
| `要確認（明細なし：Q列手入力要）` | Phase1で明細自動抽出失敗 | Q列にlines_jsonを手入力 |
| `要確認（顧客名不明：B列確認要）` | Phase1で顧客名抽出失敗 | B列・P列を手動確認 |
| `要確認（取引先未解決：〇〇）` | Phase2で partner_id 解決失敗 | freeeで取引先確認またはP列手入力 |
| `要確認（明細不足：Q列にlines_jsonを入力）` | Phase2でQ列が空 | Q列にlines_jsonを入力 |
| `（空）` | 問題なし or 見積作成済み | - |

## NAME NORMALIZATION

B列（お客様名）を返信メールの宛名に使用する際は以下のルールで正規化する。

| 入力値 | 出力 |
|---|---|
| `長谷川` | `長谷川様` |
| `長谷川様` | `長谷川様`（重複しない） |
| `専修大学` | `専修大学様` |
| `（空）` | `要確認様` |

実装: `p3_normalizeName_(name)` — 末尾が「様」なら追加しない。

## PDF POLICY

- freee IV API は `/quotations/{id}` に `Accept: application/pdf` を送っても PDF を返さない（status 200 だが Content-Type が JSON）
- `/quotations/{id}/download` は 404
- **方針: PDF の自動取得・添付は行わない**
- 下書きには PDF 添付なしで「PDFを添付しておりますのでご確認ください」と記載し、人が手動で PDF を添付してから送信する
