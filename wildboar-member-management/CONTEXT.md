## 現状の業務

### 入会受付の流れ

1. 入会希望者が来店し、紙の入会申込書に記入する
2. スタッフが申込書を確認し、会員番号・鍵番号を手書きで記入する
3. スタッフがスプレッドシートに手入力する
4. 初回費用をスタッフが電卓で計算して請求する

### 月次業務の流れ

1. 月末にリコーリースへ集金代行データを提出する（現在は手作業でまとめる）
2. 休会中・退会済みの会員はリコーリースの対象から外す（手動で確認・除外）

### 課題

- 手入力ミスが発生する（スペルミス・数字の誤り）
- 計算ミスが発生する（特に月途中入会の日割り計算）
- 鍵番号の重複・欠番が発生しやすい
- 休会・退会の管理が属人的（スタッフによって処理方法が異なる）
- リコーリースへのデータ提出に毎月数時間かかっている

## 設計方針（重要）

### データ管理

- Google Sheets をデータベースとして使う（Phase 1〜9）
- 最終的には Supabase PostgreSQL に移行する（Phase 10〜12）
- 個人情報は Google Sheets 内にのみ保存し、外部サービスには送らない

### 金額・料金

- 金額はコードにハードコードしない
- Settings シートまたは FeeRules シートから取得する
- 料金変更時はシートを更新するだけで全体に反映される

### 操作ログ

- すべての会員情報変更・申込確認・ステータス変更は AuditLogs シートに記録する
- 誰がいつ何を変更したかを追跡できるようにする

### AIへの個人情報送信禁止

- 会員の氏名・住所・電話番号・メールアドレスをAIに入力しない
- 設計・コード生成の相談のみAIを使う
- AI_RULES.md を参照すること

## 技術スタック

### Phase 1〜9（GAS版）

- バックエンド：Google Apps Script（GAS）
- データベース：Google Sheets
- フロントエンド：GAS HTMLサービス（HTML + CSS + JavaScript）
- バージョン管理：clasp

### Phase 10〜12（Webアプリ版）

- フロントエンド：Next.js 14+（App Router、TypeScript）
- データベース：Supabase PostgreSQL
- 認証：Supabase Auth
- ホスティング：Vercel

## フェーズ移行の判断基準

| 移行タイミング | 条件 |
|---|---|
| Phase 0 → Phase 1 | 設計資料が完成し、オーナーの確認が取れた |
| Phase 9 → Phase 10 | GAS版が安定稼働し、会員数が50名を超えた（GASの処理速度が課題になり始めた） |
| Phase 11 → Phase 12 | Webアプリ版でのテスト期間（最低2週間）が完了した |
