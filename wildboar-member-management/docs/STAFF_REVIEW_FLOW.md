# スタッフ確認画面設計

作成日：2026-05-04

入会申込を確認し、会員登録を完了するためのスタッフ向け管理画面の設計。

---

## 概要

入会フォームで申し込まれた内容をスタッフが確認し、以下を行う。

1. 記入内容に問題がないか確認する
2. 会員番号・鍵番号を割り当てる
3. コースを確定する
4. 初回費用を確認する
5. 正式登録する（または却下・差し戻しする）

---

## 申込一覧画面（member-list.html）

### 表示概要

- ページタイトル：「入会申込一覧」
- 新規申込（未確認）と確認済みをタブで切り替える

### フィルタ・絞り込み

| フィルタ | 選択肢 |
|---|---|
| 確認ステータス | 未確認（pending）/ 承認済み（approved）/ 却下（rejected）/ すべて |
| 申込日 | 今日 / 今週 / 今月 / 全期間 |

### 申込一覧テーブルの列

| 列 | 内容 |
|---|---|
| 受付番号 | application_id |
| 申込日時 | application_date |
| 氏名 | family_name + given_name |
| 希望コース | plan_name |
| ステータス | review_status（バッジ表示） |
| 操作 | 「確認する」ボタン |

### 未確認件数バッジ

- ナビゲーションの「申込一覧」メニューに未確認件数を赤いバッジで表示する

---

## 申込詳細確認画面

### 表示概要

- ページタイトル：「入会申込 確認」
- 申込者の入力内容を全て表示する
- 右側のパネルにスタッフ入力エリアを配置する

### 申込内容表示エリア（左または上部）

以下の入力内容をラベル付きで表示する。

- 基本情報（氏名・フリガナ・生年月日・性別・職業）
- 住所（郵便番号・都道府県・市区町村・番地・建物名）
- 連絡先（携帯電話・自宅電話・メールアドレス）
- 緊急連絡先（氏名・続柄・電話番号）
- 希望コース
- 紹介者会員番号（入力があれば）
- 備考（入力があれば）
- 申込日時
- プライバシーポリシー同意状況

---

## スタッフ入力エリア

申込内容を確認したスタッフが以下を入力する。

### 確認時にスタッフが入力する項目

| 項目 | 入力方式 | 必須 | 説明 |
|---|---|---|---|
| 会員番号 | 自動採番（修正可） | 必須 | システムが提案する番号を確認・確定する |
| 鍵番号 | 空き番号から選択 | 必須 | KeyCardsシートの空き番号一覧から選択 |
| コース（確定） | ドロップダウン | 必須 | 希望コースを確認し確定する（変更可） |
| 入会日 | 日付ピッカー | 必須 | デフォルトは本日の日付 |
| 初回費用（自動計算） | 読み取り専用（確認のみ） | 確認必須 | 入会日・コースから自動計算。内訳も表示 |
| 紹介者確認 | チェックボックス | 任意 | 紹介者番号が正しいか確認したらチェック |
| スタッフメモ | テキストエリア | 任意 | 内部メモ（会員には非表示） |

### 初回費用内訳の表示

```
入会金：       設定値 円
カードキー発行料：設定値 円
入会月日割り（xx月yy日〜月末 / zz日間）：計算結果 円
翌月（xx月）月会費：設定値 円
─────────────────
合計：         計算結果 円
```

### 確定ボタン

- ボタン「正式登録する」（緑・強調）：全項目入力済みの場合のみ有効
- ボタン「差し戻す」：記入内容に問題がある場合
- ボタン「却下する」：入会不可の場合

### 正式登録時の確認ダイアログ

```
以下の内容で会員登録します。よろしいですか？

会員番号：W-xxxx
氏名：（氏名）
鍵番号：K-xxx
コース：（コース名）
入会日：yyyy年mm月dd日
初回費用：xx,xxx円

[キャンセル]  [登録する]
```

---

## 正式登録の処理フロー（バックエンド）

1. IntakeApplications の review_status を approved に更新する
2. reviewed_by・reviewed_at を記録する
3. Members シートに新規行を追加する（全フィールドを転記）
4. KeyCards シートの該当番号を in_use に更新し、member_id を記録する
5. Payments シートに初回費用の記録を追加する
6. AuditLogs に操作を記録する
7. 完了画面を表示する

---

## 差し戻し機能

### 差し戻し時のフロー

1. スタッフが「差し戻す」ボタンをクリックする
2. 差し戻し理由を入力するダイアログを表示する（例：「住所の番地が不明確です」）
3. IntakeApplications の review_status を rejected に更新する
4. rejection_reason を記録する
5. AuditLogs に操作を記録する

### 差し戻し後の対応

- スタッフが申込者に直接連絡して修正を依頼する（システム外の対応）
- 修正後は新たに申込フォームから再申込してもらうか、スタッフが直接修正する

---

## 操作ログ記録

以下の操作は必ず AuditLogs シートに記録する。

| 操作 | action値 | 記録内容 |
|---|---|---|
| 申込詳細を開く | view | application_id |
| 正式登録 | create | 会員番号・スタッフ名・日時 |
| 差し戻し | reject | application_id・理由・スタッフ名 |
| 却下 | reject | application_id・理由・スタッフ名 |
| 会員番号変更 | update | 変更前後の番号 |
| 鍵番号変更 | update | 変更前後の番号 |

---

## 画面設計補足

- スタッフはジム内のPCまたはiPadから操作する
- スタッフ認証：GASのWebアプリを「Googleアカウントでのログインが必要」に設定する
- 複数スタッフが同時に同じ申込を確認しようとした場合の競合処理は Phase 1 では考慮しない
