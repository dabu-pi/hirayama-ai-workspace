# GAS版からWebアプリ版への移行計画

作成日：2026-05-04
対象：Phase 11〜12（Supabase移行・本番リリース）

---

## 移行の概要

GAS版（Google Sheets + Google Apps Script）で蓄積した実運用データを、
Supabase PostgreSQL に移行し、Next.js Webアプリ版に切り替える。

---

## 移行のタイミングと判断基準

### 移行を開始するタイミング

以下の条件がすべて満たされた場合に Phase 11（移行作業）を開始する。

| 条件 | 確認方法 |
|---|---|
| GAS版が6ヶ月以上安定稼働している | バグ報告なし・データ不整合なし |
| 会員数が50名を超えた | GASの処理速度・割り当て上限が問題になり始めた |
| Next.js Webアプリ版が全機能実装・テスト済み | テスト環境での動作確認完了 |
| オーナーが移行を承認した | オーナーとの合意 |

### 移行を延期する条件

- GAS版で未解決のバグがある
- Next.js版のテストが完了していない
- オーナーが繁忙期（要確認）

---

## 移行手順

### Phase 11-1：Supabaseプロジェクト作成と初期設定

1. Supabase で本番用プロジェクトを作成する
2. DATABASE_SCHEMA_WEBAPP.md の SQL を順番に実行する
3. RLS ポリシーを設定する（DATABASE_SCHEMA_WEBAPP.md 参照）
4. スタッフのアカウントを Supabase Auth で作成する
5. Settings テーブルに初期設定値を投入する
6. MembershipPlans テーブルにコースマスタを投入する
7. FeeRules テーブルに料金ルールを投入する

### Phase 11-2：Google Sheetsデータのエクスポート

1. Google Sheets の全シートを CSV でエクスポートする
2. エクスポート時点の会員数・最終会員番号・最終鍵番号を記録する
3. エクスポートファイルを安全な場所に保存する（Google Drive の専用フォルダ）

エクスポートするシート：
- Members
- IntakeApplications
- StatusHistory
- KeyCards
- Payments
- Referrals

### Phase 11-3：データクレンジング

CSVをインポートする前に、以下の観点でデータをクリーニングする。

| チェック項目 | 対処方法 |
|---|---|
| 会員番号の重複・欠番 | 重複は手動確認・削除。欠番はそのまま |
| 鍵番号の重複 | 手動確認・修正 |
| 電話番号のフォーマット統一 | ハイフンを除去して数字のみに統一 |
| 郵便番号のフォーマット統一 | ハイフンを除去して7桁に統一 |
| ステータスの表記統一 | 「在籍」→ active、「休会」→ paused、「退会」→ withdrawn |
| フリガナの全角カタカナ統一 | ひらがな・半角カタカナを全角カタカナに変換 |
| 空欄フィールドの確認 | 必須フィールドが空欄の場合はスタッフが手動補完 |
| 日付フォーマットの統一 | YYYY-MM-DD 形式に統一 |

### Phase 11-4：データインポート

以下の順序でデータをインポートする（外部キー制約を考慮した順序）。

1. key_cards シートのデータをインポート（member_id は NULL で）
2. members シートのデータをインポート（key_card_id は NULL で）
3. key_cards の member_id を会員コードで更新する
4. members の key_card_id を鍵コードで更新する
5. status_histories のデータをインポート
6. payments のデータをインポート
7. intake_applications のデータをインポート
8. referrals のデータをインポート

#### インポートスクリプトの方針

- Python スクリプトまたは Supabase の CSV インポート機能を使用する
- GAS版の連番IDを Supabase の UUID に変換するマッピングテーブルを作成する
- インポートはテスト環境で先に実施し、問題なければ本番環境で実施する

### Phase 11-5：移行後の検証

インポート完了後に以下を確認する。

| 確認項目 | 確認方法 |
|---|---|
| 会員数の一致 | GAS版の会員数 = Supabase の会員数（ステータス別） |
| 最終会員番号の一致 | 最大の member_code が一致する |
| 鍵番号の対応確認 | 無作為に10件サンプリングして確認 |
| StatusHistory の件数 | GAS版とSupabase版の件数が一致する |
| Payments の合計金額 | GAS版とSupabase版の合計が一致する |
| 必須フィールドの空欄確認 | 必須フィールドに NULL がないか確認 |

## データクレンジングのポイント

### 住所

- 全角・半角の統一（全角スペース → 半角スペース）
- 都道府県の統一（「兵庫」→「兵庫県」等）

### 氏名・フリガナ

- フリガナの全角カタカナ統一
- 氏名のスペース統一（全角スペース・半角スペース）

### 電話番号

- ハイフン除去（090-1234-5678 → 09012345678）
- 先頭のゼロの有無確認

### 日付

- Excel の数値形式（シリアル値）になっている場合は日付に変換する
- フォーマット統一（令和○年→ YYYY-MM-DD）

## GAS版のバックアップ運用

### 移行期間中（Phase 11）

- GAS版をそのまま継続稼働させる
- Webアプリ版に移行後もGAS版は2ヶ月間は読み取り専用で残す

### 移行完了後（Phase 12 以降）

- GAS版の Webアプリ設定を「アクセス不可」に変更する
- スプレッドシートは削除せず、Google Drive にアーカイブとして保管する
- GASのコードは GitHub に保管したまま（削除しない）

---

## ロールバック計画

Webアプリ版に問題が発生した場合の切り戻し手順。

1. Vercel のデプロイを前バージョンにロールバックする
2. GAS版の Webアプリ設定を「アクセス可能」に戻す
3. 移行期間中の新規データ（Webアプリ版で入力されたデータ）を GAS版に手動で入力する
4. 問題を修正後、再度移行作業を実施する

### ロールバックが必要なケース

- データ消失・不整合が発覚した場合
- スタッフが使用できない重大な UI バグがある場合
- パフォーマンス問題でシステムが使い物にならない場合

---

## 並行運用期間のルール

Phase 11〜12の並行運用期間中は以下のルールを守る。

- 新規入会申込は Webアプリ版（テスト環境）を使用し、正式データは GAS版にも手動入力する
- 退会・休会処理は両方のシステムに反映する
- 毎週、両システムのデータを突き合わせて差分を確認する
- 並行運用期間は最大2ヶ月とする
