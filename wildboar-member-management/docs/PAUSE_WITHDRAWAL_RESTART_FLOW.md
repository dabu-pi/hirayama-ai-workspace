# 休会・退会・再開の処理フロー

作成日：2026-05-04

---

## 概要

会員ステータスの変更（休会・退会・再開）はすべてスタッフが管理画面から操作する。
すべての変更は StatusHistory シートに記録し、変更前後のステータスを追跡できるようにする。

---

## 休会処理フロー

### 申請受付

1. 会員が休会を申し出る（口頭または書面）
2. スタッフがシステムのステータス変更画面（status-change.html）を開く
3. 対象の会員を検索・選択する
4. 「休会申請を受け付ける」を選択する

### スタッフが入力する項目

| 項目 | 入力方式 | 必須 | 説明 |
|---|---|---|---|
| 休会開始日 | 日付ピッカー | 必須 | 翌月1日が多い（要ルール確認） |
| 休会終了予定日 | 日付ピッカー | 任意 | 未定の場合は空欄可 |
| 休会理由 | テキスト / 選択 | 任意 | 仕事 / 怪我 / 出産 / 引越 / その他 |
| 休会中の鍵扱い | 選択 | 必須 | 返却する / 保持する（要ルール確認） |
| メモ | テキストエリア | 任意 | 内部メモ |

### システムの処理

1. StatusHistory に記録を追加する
   - change_type: pause
   - previous_status: active
   - new_status: paused
   - effective_date: 入力された休会開始日
   - end_date: 入力された休会終了予定日
2. Members シートの status を paused に更新する
3. Members シートの updated_at を更新する
4. 鍵を返却する場合は KeyCards シートの status を available に変更する
5. AuditLogs に操作を記録する

### 月別集計への影響

- 休会開始月：設定によりリコーリースの対象から外す（または休会費のみ請求）
- 休会中：毎月のリコーリースデータ生成時に自動的に除外する
- 再開月：再開日から再びリコーリースの対象に含める

---

## 退会処理フロー

### 申請受付

1. 会員が退会を申し出る（口頭または書面）
2. スタッフがシステムのステータス変更画面を開く
3. 対象の会員を検索・選択する
4. 「退会申請を受け付ける」を選択する

### スタッフが入力する項目

| 項目 | 入力方式 | 必須 | 説明 |
|---|---|---|---|
| 退会日 | 日付ピッカー | 必須 | 通常は当月末または翌月末 |
| 退会理由 | 選択 / テキスト | 任意 | 引越 / 多忙 / 経済的理由 / 体調 / その他 |
| 鍵返却確認 | チェックボックス | 必須 | 鍵を受け取った場合にチェック |
| メモ | テキストエリア | 任意 | 内部メモ |

### 鍵返却未完了の場合

- 「鍵未返却」フラグを KeyCards シートに記録する
- 後から返却された場合に KeyCards シートを更新する

### システムの処理

1. StatusHistory に記録を追加する
   - change_type: withdraw
   - previous_status: active（または paused）
   - new_status: withdrawn
   - effective_date: 入力された退会日
2. Members シートの status を withdrawn に更新する
3. Members シートの updated_at を更新する
4. 鍵を返却した場合は KeyCards シートの status を available に変更し、returned_date を記録する
5. AuditLogs に操作を記録する

### 月別集計への影響

- 退会日の翌月以降：リコーリースデータから自動的に除外する
- 退会月：退会日によって請求するか設定（要ルール確認）

### 退会後のデータ管理

- 退会後も Members シートのデータは保持する（status = withdrawn）
- データ保持期間については PRIVACY_SECURITY_POLICY.md を参照する
- 保持期間終了後の処理は別途検討する

---

## 再開処理フロー

### 申請受付

1. 休会中の会員が再開を申し出る（口頭または電話）
2. スタッフがシステムのステータス変更画面を開く
3. 対象の会員を検索・選択する
4. 「再開申請を受け付ける」を選択する

### スタッフが入力する項目

| 項目 | 入力方式 | 必須 | 説明 |
|---|---|---|---|
| 再開日 | 日付ピッカー | 必須 | 翌月1日が多い（要ルール確認） |
| 鍵の扱い | 選択 | 必須 | 同じ番号を再割り当て / 新しい番号を割り当て |
| メモ | テキストエリア | 任意 | 内部メモ |

### システムの処理

1. StatusHistory に記録を追加する
   - change_type: restart
   - previous_status: paused
   - new_status: active
   - effective_date: 入力された再開日
2. Members シートの status を active に更新する
3. Members シートの updated_at を更新する
4. 鍵を新しく割り当てる場合は KeyCards シートを更新する
5. AuditLogs に操作を記録する

### 月別集計への影響

- 再開月：再開日から月末の日割り請求が発生するかどうかは要確認
- 再開翌月以降：通常の月会費請求に戻る

---

## StatusHistory への記録方法

StatusHistory シートへの記録フォーマット：

| フィールド | 値 |
|---|---|
| history_id | STATUS-yyyymmdd-XXXXX（自動採番） |
| member_id | 対象会員番号 |
| change_type | pause / withdraw / restart |
| previous_status | active / paused / withdrawn |
| new_status | active / paused / withdrawn |
| effective_date | 変更適用日 |
| end_date | 休会の終了予定日（休会の場合のみ） |
| reason | 理由（任意） |
| notes | メモ |
| created_at | 操作日時 |
| created_by | 操作したスタッフのGoogleアカウント |

---

## 月別集計への影響まとめ

| 変更種別 | 影響月 | リコーリース | 月会費 |
|---|---|---|---|
| 休会 | 休会開始月 | 除外（または休会費） | 設定による |
| 休会 | 休会継続月 | 除外 | なし（または休会費） |
| 退会 | 退会月 | 要確認 | 要確認 |
| 退会 | 退会翌月以降 | 除外 | なし |
| 再開 | 再開月 | 含める | 日割りまたは全額（要確認） |
| 再開 | 再開翌月以降 | 通常 | 通常 |

---

## 状態遷移図

```
[active] ---(休会申請)---> [paused] ---(再開申請)---> [active]
   |                          |
   +--------(退会申請)---------+-----(退会申請)--------> [withdrawn]
```

注意：
- withdrawn（退会済み）からの再入会は、新規の IntakeApplication として処理する（再開申請ではなく新規入会）
- withdrawn から active / paused には直接遷移しない

---

## エラー・例外ケース

| ケース | 対応 |
|---|---|
| 退会後に鍵が見つかった | KeyCards シートの returned_date と status を手動更新し、AuditLog に記録する |
| 休会中に退会を希望した | 休会状態から直接退会処理を行う（previous_status = paused） |
| 最長休会期間を超えた | システムが警告を表示する。スタッフが退会処理または延長を判断する |
| 再開後に再度休会したい | 通常の休会申請と同様に処理する |
