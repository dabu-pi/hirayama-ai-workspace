# feature-scope

## UI設計方針（2026-04-11 更新）

**Boostcamp準拠**を基本方針とする。

- Boostcamp の実画面に存在する UI 要素は、原則として画面上に置く
- 「MVP だから隠す」という判断は原則しない
- 実装の優先順位（先に作るか後で作るか）と、UI上に置くかどうか（画面設計）は別で判断する
- UI として存在させつつ、実装が間に合わない機能は非活性（押せない状態）または最小動作で出す

---

## 今回入れる機能

- プログラム一覧
- 条件絞り込み
- プログラム詳細
- プログラム開始
- 当日ワークアウト表示
- 実行画面での前回記録表示
- 実行画面での目標表示
- セット入力
- セット / 回数 / 重量 / RPE / メモ記録
- 履歴参照
- 実行画面から履歴への導線
- Add Set（種目内にセットを追加する）
- Swap（種目を置き換える）
- Add Exercise（セッション全体に種目を追加する）
- Rest タイマー（セット間の休憩タイマー）
- Calc（重量計算補助）
- 管理者ログイン
- 管理者によるプログラム登録 / 編集
- 種目マスタ管理
- 作成者情報登録
- 公開 / 非公開管理

## 今回は要件だけ置く機能

- 一般ユーザー登録
- 一般ユーザーログイン
- マイページ
- ユーザー別継続履歴
- 複数ユーザー前提の権限分離

## 後回し機能

- PWA の本格最適化
- 通知
- オフライン対応
- 高度な分析表示
- プログラム進行の自動提案
- お気に入り、保存、比較の拡張
- SNS
- コミュニティ機能
- 課金
- 一般ユーザー投稿

## 今回入れない機能

- ライブ配信やリアルタイム競争機能
- 広告ネットワーク前提の機能

## スコープ上の判断メモ

- 初期価値は「導入しやすいプログラムライブラリ」と「迷わない実行 UI」と「続けやすい記録」
- 特に `前回記録` と `履歴参照` は実行 UI の中心価値として扱う
- SNS、コミュニティ、課金、一般投稿は MVP を重くするため後回しとする
- 将来拡張を意識しつつも、最初から多人数向け運用機能を作り込みすぎない
- Add Set / Swap / Add Exercise / Rest / Calc は Boostcamp 準拠として UI 上に置く。実装の細部は開発フェーズで決める
- Programs / Train / Analytics の 3タブナビゲーションは Boostcamp 準拠として今後の画面設計に反映する
