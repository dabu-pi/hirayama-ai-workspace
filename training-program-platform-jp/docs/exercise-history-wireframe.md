# exercise-history-wireframe

## 位置づけ

この文書は「種目単体履歴」画面の wireframe（画面設計図）である。
「今日のワークアウト」画面で種目名をタップしたときの遷移先として、スマホ縦画面でどこに何を置くかを固定することが目的。

実装方法ではなく、**情報の優先順位・配置・実行画面とのつながり**を明確にする。

前提文書:
- `docs/workout-screen-wireframe.md`（実行画面設計図）
- `docs/workout-screen-interaction-spec.md`（実行画面操作仕様）
- `docs/workout-execution-ui.md`（実行UI要件）
- `docs/screens.md`（画面一覧）
- `docs/requirements.md`（機能要件）

---

## 1. 画面の目的

種目単体履歴は、特定の種目について**過去の記録を一覧で確認し、次の入力判断に使う**画面である。

ユーザーはこの画面で次のことを行う。

- その種目の過去記録を新しい順に振り返る
- 前回値（Previous）の根拠を確認する
- 重量・回数の流れを素早く把握する
- 確認後に実行画面へ戻り、次のセット入力に活かす

この画面は補助画面であり、長時間滞在する画面ではない。
**目的は参照であり、入力はしない。**

---

## 2. ユーザーがこの画面で確認したいこと

| 優先度 | 確認したい内容 |
|---|---|
| 1 | 直近のセット記録（重量・回数） |
| 2 | 前回と前々回の比較 |
| 3 | 過去数回分の流れ（伸びているか、停滞しているか） |
| 4 | どのプログラム・どの日の記録か |
| 5 | メモがあれば内容を確認 |

---

## 3. 画面全体の情報優先順位

1. 戻るボタン（実行画面へ戻る導線）
2. 種目名（今どの種目の履歴を見ているか）
3. 最新記録サマリー（1行で最重要情報を表示）
4. 履歴一覧（新しい順）
5. 補助情報（種目カテゴリ・タグ）

---

## 4. 画面を上から下へどう構成するか

### 1. ヘッダーエリア

役割:
- 今どの種目の履歴を見ているかを示す
- 実行画面への戻り導線を確保する

内容:
- 戻るボタン（`←`）
- 種目名
- 種目タイプ（`T1 / T2 / T3`）または種目カテゴリ（任意）

スマホ縦画面での見せ方:
- 左端に戻るボタン、中央に種目名
- 種目タイプは種目名の右横または下に小さく表示

ワイヤー例:

```text
+--------------------------------------------------+
| ← | Squat                          [T1]          |
+--------------------------------------------------+
```

---

### 2. サマリーエリア

役割:
- 直近の重量・回数を1行で即見せる
- 履歴一覧を開く前に最重要情報を把握させる

内容:
- 最新実施日
- 直近セット記録（重量・回数の代表値、または最終セット値）
- 直近メモ有無フラグ（任意）

スマホ縦画面での見せ方:
- 履歴一覧の上に固定せず、自然に1ブロックとして置く
- 数字を大きめに見せる
- 「前回値」の根拠がここで分かるレベルを維持する

ワイヤー例:

```text
+--------------------------------------------------+
| 最新: 2026-04-01                                  |
| 80kg × 3, 80kg × 3, 80kg × 2                    |
| GZCLP Week 8 Day 1                               |
+--------------------------------------------------+
```

---

### 3. 履歴一覧エリア

役割:
- 過去のセッション記録を新しい順に並べる
- 各セッションでのセット記録を読める粒度で表示する
- 重量・回数の流れを数回分で把握させる

内容（1セッション分の表示ブロック）:
- 実施日
- プログラム名または実施文脈
- セットごとの重量・回数

スマホ縦画面での見せ方:
- 1セッション = 1カードブロック
- セット記録は横並びまたは改行で読みやすく
- カード間は明確に区切る
- スクロールで過去分を遡れる構成

ワイヤー例:

```text
+--------------------------------------------------+
| 2026-04-01  GZCLP Week 8 Day 1                   |
| Set 1: 80kg × 3                                  |
| Set 2: 80kg × 3                                  |
| Set 3: 80kg × 2                                  |
+--------------------------------------------------+
| 2026-03-28  GZCLP Week 7 Day 3                   |
| Set 1: 77.5kg × 3                                |
| Set 2: 77.5kg × 3                                |
| Set 3: 77.5kg × 3                                |
+--------------------------------------------------+
| 2026-03-25  GZCLP Week 7 Day 1                   |
| Set 1: 77.5kg × 5                                |
| Set 2: 77.5kg × 5                                |
| Set 3: 77.5kg × 4                                |
+--------------------------------------------------+
```

---

### 4. 補助表示エリア（任意・最小）

役割:
- Previous として採用される候補がどれかを示す
- 詳細分析や推移グラフは将来の拡張用として場所だけ確保する

内容（暫定）:
- 「前回記録として参照: Set 1-3 / 2026-04-01」のような1行表示（暫定）
- グラフ表示エリアのプレースホルダー（MVP では非表示でよい）

スマホ縦画面での見せ方:
- サマリーエリアの下に小さく表示するか、履歴一覧の上部に置く
- グラフはMVPでは出さない。構造上の場所だけ確保する（暫定）

ワイヤー例:

```text
+--------------------------------------------------+
| 前回記録の根拠: 2026-04-01 のセット記録          |
| （グラフエリア — MVP以降）                        |
+--------------------------------------------------+
```

---

### 5. 下部エリア

役割:
- 実行画面への戻り導線を下部でも確保する
- 長い履歴をスクロールした後でも戻れる

内容:
- 戻るボタン（テキストボタンまたは固定フッター）

スマホ縦画面での見せ方:
- 固定フッターにする場合は最小限（1行）
- ヘッダーの戻るボタンだけで十分と判断した場合は置かない（未確定）

ワイヤー例（固定フッターありパターン）:

```text
+--------------------------------------------------+
| ← 今日のワークアウトに戻る                        |
+--------------------------------------------------+
```

---

## 5. 各UIブロックの役割まとめ

| UIブロック | 役割 |
|---|---|
| ヘッダーエリア | 種目名の表示と実行画面への戻り導線 |
| サマリーエリア | 直近記録の即時確認（参照効率の最大化） |
| 履歴一覧エリア | 過去セッションを新しい順に表示 |
| 補助表示エリア | Previous の根拠表示・グラフ将来枠 |
| 下部エリア | スクロール後の戻り導線 |

---

## 6. モバイル縦画面での見せ方

- 縦スクロール1本で完結する構成
- ヘッダーは固定しない（シンプルな戻るボタンで十分）
- セット記録は横スクロールを使わない
- 1カードに詰め込みすぎず、2〜3行で完結させる
- フォントサイズは実行画面より小さくてよいが、数字（重量・回数）は読みやすさを保つ
- `Previous` の根拠（参照元セッション）はサマリーエリアで把握できればよく、強調しすぎない

---

## 7. 履歴の並び順

- 基本は**新しい順**（最新が一番上）
- 並び替えオプション（古い順・重量順など）はMVP以降の拡張とする

---

## 8. セット記録の表示粒度

| 表示方針 | 内容 |
|---|---|
| 1セッション単位でカード表示 | セットごとの重量・回数をそのまま列挙する |
| 要約は使わない（暫定） | 「80kg × 3set」のような要約は、前回値の根拠として粒度不足になる可能性があるため、MVP ではセット明細をそのまま見せる |
| メモがある場合 | カード内に小さく「メモあり」または本文をそのまま表示（優先度は低い） |

---

## 9. 実行画面からの遷移と戻り

### 遷移元
- 「今日のワークアウト」画面の種目名タップ

### 遷移先
- 種目単体履歴画面（この画面）

### 戻り
- ヘッダーの「←」で「今日のワークアウト」画面へ戻る
- 戻り後、入力中のデータはそのまま維持される（interaction-spec にて確定済み）

---

## 10. 初心者向けの見やすさ配慮

- 種目名を大きく表示し、今どの種目の記録を見ているかが迷わず分かる
- 日付と重量・回数が1行で読める
- 過去記録が多くても、直近2〜3件だけ読めれば目的を果たせる構成にする
- プログラム名は小さく添える程度にとどめ、重量・回数を優先表示する

---

## 11. 中級者向けの比較しやすさ配慮

- サマリーエリアで直近値を即把握できる
- 履歴一覧でセット明細がそのまま見えるため、前回・前々回の比較が1スクロールで可能
- 重量の伸びや停滞がカードを縦に読むだけで分かる
- AMRAP 記録（最大回数挑戦）など回数が変動するセットも、実記録そのままで読める

## 13. 未確定事項

### MVP前に決定が必要

| 項目 | 内容 |
|---|---|
| 下部戻る導線の有無 | 固定フッターに戻るボタンを置くか、ヘッダーの「←」だけで十分か |
| Previous の採用基準の画面表示 | 「前回記録の根拠」行をこの画面で見せるかどうか。実装上の参照ロジックと合わせて決める |

### MVP以降でよい

| 項目 | 内容 |
|---|---|
| 履歴一覧の表示粒度 | セット明細をデフォルト展開にするか、折りたたみ（タップで展開）にするか |
| メモ表示の優先度 | メモを常に表示するか「メモあり」フラグだけ見せるか |
| グラフ（重量・回数の推移表示） | 最初から入れるか、MVP以降に回すか（MVP以降推奨） |
| Previous の参照単位の確定 | セット番号一致優先か、種目内最新完了優先か（interaction-spec から引き継ぎ） |
| 並び替えオプション | 古い順・重量順など |
| 件数制限 | 表示する過去セッションの件数の上限または無限スクロール |
