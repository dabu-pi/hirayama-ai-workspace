# tech-decision

## なぜ初期は Web 寄りか

- 最初の検証速度を上げやすい
- 管理画面と利用画面を同じ基盤で始めやすい
- PC とスマホの両方へ早く届けやすい
- 将来ネイティブ化や PWA 化を考える前に、体験の核を検証しやすい

## なぜスマホ優先か

- 実利用の主戦場はトレーニング中の現場であり、スマホ利用が中心になるため
- 当日ワークアウト表示と記録入力は、片手操作や短時間入力が重要なため
- PC は管理・登録補助には向くが、実行体験の中心ではないため

## なぜ将来 PWA 化しやすい設計にするか

- スマホホーム追加や再開導線を取り込みやすい
- 初期からネイティブアプリを作るより、実装負荷と運用負荷を抑えやすい
- キャッシュ、起動導線、通知などを段階的に追加しやすい
- 「まず Web」で始めつつ、利用体験をアプリに近づける余地を残せる

## なぜ初期から複数ユーザーを本実装しないのか

- 初期運用は管理者 1 名で十分に検証できるため
- MVP の本質は、ライブラリ / 実行 / 記録の体験成立にあるため
- 認証、権限、マイページ、ユーザー分離を同時に進めると初期開発が重くなるため
- まずは単一運用で情報設計と記録導線を磨くほうが価値が高いため

## 将来複数ユーザー対応するときに崩れない前提

- `users` を初期から概念上持つ
- 記録や参加中プログラムは将来 `user_id` で分離できる前提を置く
- 管理者権限と一般ユーザー権限を役割で分けられるようにする
- ライブラリ情報とユーザー固有データを混ぜない
- 管理 UI と利用 UI の責務を整理しやすい構造にする

## 初期技術判断のメモ

- 実装方式は Web アプリ中心
- UI 設計はスマホ優先
- 管理画面は PC 利用を想定
- PWA 化を妨げない画面構成と状態管理を意識する
- 採用技術の製品名レベルの確定は、UI プロトタイプ時点で判断する

## 今は確定しないこと

- フロントエンドの具体フレームワーク
- バックエンドの提供方式
- 認証基盤の具体サービス
- ホスティング先

今回は「崩れない前提」を固定し、製品選定は後続フェーズで決める。
