# workout-screen-wireframe

## 目的

この文書は、「今日のワークアウト」画面の wireframe（画面設計図）を固定するための文書である。
実装方法ではなく、**スマホ縦画面で何をどこに置き、どの順で見せ、どう操作させるか**を明確にする。

この画面はアプリの最重要画面であり、利用者はここで次の 3 つを同時に行う。

- 今日やる内容を確認する
- 前回記録と今回目標を見比べる
- その場で入力して完了させる

## ユーザーがこの画面でやること

- プログラムの当日内容を確認する
- 種目ごとの `Previous` と `Target` を見る
- Kg / Reps を入力する
- セットを完了チェックする
- 必要なら種目単体履歴を見る
- 必要なら `Add Set` / `Swap` / `Add Exercise` を使う
- 最後に `Finish` で記録完了へ進む

## 画面全体の情報優先順位

1. 上部固定エリアの主要操作
2. 種目名と種目タイプ
3. 各セット行の `Previous`
4. 各セット行の `Target`
5. Kg / Reps 入力
6. 完了チェックとロック状態
7. 種目ごとの補助操作
8. 画面下部の `Add Exercise`
9. 補足情報

## 画面を上から下へどう構成するか

### 1. 上部固定エリア

役割:

- 実行中に常に触れる主要操作を固定表示する
- スクロールしてもタイマーと終了導線を失わないようにする

内容:

- `Rest`
- `Calc`
- タイマー
- `Finish`

スマホ縦画面での見せ方:

- 1 行固定
- ボタンは短いラベル
- タイマーは中央寄りで視認性を優先

ワイヤー例:

```text
+--------------------------------------------------+
| Rest | Calc | 02:15 | Finish                     |
+--------------------------------------------------+
```

### 2. プログラム情報エリア

役割:

- 今どのプログラムの何週目 / 何日目かを短く示す
- 進め方の判断に必要な最小説明だけを置く

内容:

- プログラム名
- `Week / Day`
- `Progression Guide`
- `Notes`

スマホ縦画面での見せ方:

- テキスト量は短く保つ
- `Progression Guide` と `Notes` は畳める前提でもよいが、文書上は表示枠を確保する

ワイヤー例:

```text
Program: 5/3/1 Beginner
Week 2 / Day 1
Progression Guide: main setは前回超えを狙う
Notes: last setは無理のない範囲で
```

### 3. 種目ブロック

役割:

- 種目単位で入力のまとまりを作る
- 種目名タップから種目単体履歴へ移動できるようにする

内容:

- 種目名
- 種目タイプ表示
  - `T1 / T2 / T3`
- 種目名タップで履歴へ

スマホ縦画面での見せ方:

- 種目名は大きめ
- 種目タイプは小さなラベル
- 履歴導線は種目名自体を押せる形にする

ワイヤー例:

```text
[T1] Squat >
tap: 種目単体履歴
```

### 4. セット一覧

役割:

- その種目で入力すべき最重要情報を 1 か所にまとめる
- `Previous` と `Target` を見ながら Kg / Reps を埋められるようにする

列:

- `Set`
- `Previous`
- `Target`
- `Kg`
- `Reps`
- `完了`

モバイル縦画面での見せ方:

- 厳密な表より、2 段構成の行カードに近い見せ方でもよい
- ただし情報順は固定する
- `Previous` がない場合は `-`
- Kg 初期値は空欄
- 1 セット目入力後、後続セットへ同重量を自動反映

### 5. 種目ごとの補助操作

役割:

- 入力中の例外対応を種目単位で処理する

内容:

- `Add Set`
- `Swap`
- `...`

配置候補:

- セット一覧の直下
- 種目ブロックの末尾
- 横並び 2 から 3 ボタン

### 6. 画面下部

役割:

- セッション全体に対する追加操作を置く

内容:

- `Add Exercise`

配置候補:

- 画面最下部の大きめボタン
- 種目ブロック列の最後に固定しない

## 各 UI ブロックの役割まとめ

| UI ブロック | 役割 |
|---|---|
| 上部固定エリア | 実行中の主要操作を保持する |
| プログラム情報エリア | 今日の文脈を短く示す |
| 種目ブロック | 入力の単位を作る |
| セット一覧 | 前回比較と入力を同時に行う |
| 種目ごとの補助操作 | 種目単位の例外処理を置く |
| 画面下部 | セッション全体への追加操作を置く |

## モバイル縦画面での見せ方

- 縦スクロール 1 本で進められる構成にする
- 上部固定エリアだけは常に見える
- 横スクロールは避ける
- セット一覧は表そのものより、読みやすい行ブロック優先で考える
- `Previous` と `Target` は入力欄より上か左に近い位置で見せる
- `完了` は誤タップしにくい配置にする

## 1 種目分のセット行 wireframe（画面設計図）例

### 例1: 通常セット

```text
[T1] Squat >

Set 1
Previous: 80kg x 3
Target:   3 reps
Kg:       [     ]
Reps:     [  3  ]
[ ] 完了
```

### 例2: AMRAP 行

```text
[T1] Squat >

Set 5
Previous: 82.5kg x 2
Target:   3+ reps
Kg:       [82.5 ]   ← 空欄でもよい。自動反映後なら表示可
Reps:     [     ]
[ ] 完了
```

## セット一覧の compact（省スペース）例

```text
Set | Previous      | Target   | Kg     | Reps | Done
1   | 80kg x 3      | 3 reps   | [   ]  | [  ] | [ ]
2   | 80kg x 3      | 3 reps   | [80 ]  | [  ] | [ ]
3   | -             | 3+ reps  | [80 ]  | [  ] | [ ]
```

補足:

- 2 行目以降の Kg は、1 セット目入力後の自動反映例
- `Previous=-` は空欄ではなくハイフンで示す

## 履歴導線の置き場所

履歴導線は 2 箇所を前提にする。

1. 種目名タップ
   - 種目単体履歴へ遷移
2. 画面内の履歴導線
   - セッション全体または確認用履歴へ進む補助導線

優先度は次の通り。

- 最優先: 種目名タップによる種目単体履歴
- 補助: 画面内の履歴リンク

## Add Set / Swap / Add Exercise の配置候補

### `Add Set`

- 種目ブロック末尾
- セット一覧の直下

### `Swap`

- 種目ブロック末尾
- `Add Set` の隣

### `Add Exercise`

- 画面最下部
- セッション全体の末尾に独立して置く

## セット行の state（画面状態）

### 1. 未入力

- Kg / Reps 未入力
- `Previous` と `Target` を参照して入力開始する

### 2. 入力済み

- Kg または Reps が入力済み
- まだ編集可能

### 3. チェック済み・ロック

- 完了チェック済み
- 行がロックされ、編集不可

### 4. ロック解除後再編集

- 明示的にロック解除
- 再入力後に再チェック可能

## 初心者でも迷わないための UI 原則

- 1 行ごとに「前回」「今回目標」「入力欄」が完結して見える
- `Previous` は参照用であり、入力欄ではないことを見た目で分ける
- 入力前に必要な情報が揃っている
- 画面を行き来しなくても最低限進められる
- `Finish` は常に見つけやすい

## 中級者がテンポよく入力できるための UI 原則

- 1 セット目入力の重量を後続セットへ自動反映する
- 後続セットは必要ならすぐ上書きできる
- チェック後ロックで誤編集を防ぐ
- 種目名タップで即座に種目単体履歴へ入れる
- `Target` に `3+ reps` などの条件を短く表示できる

```text
+--------------------------------------------------+
```
